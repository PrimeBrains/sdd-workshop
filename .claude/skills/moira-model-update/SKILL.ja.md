# Moira 正典モデル更新（moira-model-update）

Moira の確定設計物（既定の対象は `moira/MODEL.md`）への変更を、**独立した敵対者による
falsifiable なゲート**を通してからのみ確定するスキル。思想の確定物を「書いた本人が満足したら
コミット」する事故を防ぐ。

このスキルの規律は3つの shared-rules に分かれている。手順中で必ず参照すること:
- `model-and-subagent-policy.md` — 推奨モデル/effort（best-effort）、サブエージェント派遣、コスト上限
- `adversarial-vectors.md` — 攻撃角カタログ（モデル攻撃角 V1–V6 / スキル攻撃角 S1–S8）
- `review-gate.md` — 確定ゲート、三者分離、反証の再反論、停止性、§6 既存来歴の扱い

## いつ使うか

- 思考実験の結論を MODEL に正典化するとき
- ユーザーの設計判断を MODEL に反映するとき
- レビューで見つかった欠陥を直すとき
- Moira の命名文書・**本スキル自身・エージェント定義**など確定設計物を改訂するとき

軽微な誤字修正には使わない（高コスト手順のため）。ただし**思想・構造・語彙・既定値に触れる
変更は、どれだけ小さく見えても必ずこのスキルを通す**。判断に迷う「語の言い換え」は、意味（語彙）
を変えるなら通す・字面だけなら通さない、を目安とする。

## 0. 推奨前提の確認（best-effort・機械強制はしない）

`model-and-subagent-policy.md` に従う。Opus 4.8+ / effort max は**強く推奨**だが、モデルの自己
バージョン判定も effort 観測も信頼できないため**自動停止ゲートにはしない**。
- 環境に "exact model ID" があり Opus 4.8 未満と判る場合は、その旨を告げて `/model` 切替を勧める。
- 判らない場合は「Opus 4.8 + effort max を推奨」と正直に伝え、最終責任はユーザーに委ねる。
- 以降のサブエージェントは**役割別のモデル階層**で派遣（実際に制御可能なので強制）: 敵対者
  `moira-adversary`・採点者 `moira-gate-judge` は `model: opus`、事実検証 `moira-fact-checker` は
  `model: sonnet`。敵対攻撃は既定 Claude 1 体＋codex 1 本（`model-and-subagent-policy.md`）。
  プロンプトで「最大徹底度・真に敵対的」を課す。

## 1. 対象種別の判定と変更のフレーミング

1. 対象ファイルを特定し、**対象種別**を判定する。判定は**対象の同一性（役割・パス）**で行い、現在
   章構造を持つか否かでは行わない（ブートストラップ中は構造が未整備でも MODEL 系である）:
   - **MODEL 系**: `moira/MODEL.md`（正典モデル本体）。§6 来歴・§7 確認事項・版ヘッダ・A/I/P/R 体系を
     持つ（持たない素の状態なら、確定時にこれらを整備する）。
   - **設計物その他**: 命名文書・本スキル・エージェント定義など、上記の章構造を本来持たない確定物。
   種別により、後段の**攻撃角**（§2）と**確定手順**（§4）が分岐する。
2. **対象が存在しない場合（ブートストラップ）**: 既定 `moira/MODEL.md` がまだ無いなら、今回の変更は
   「初回正典化」である。取り込み元（例: 既存の v9 文書）の**内容**を対象として §2 以降のフルループを
   回す。**`moira/MODEL.md` を実ファイルとして書き出すのは確定ステップ（§4）でのみ**——ゲート通過前に
   書き出して中断すると、未確定の孤児ファイルが残り次回起動で「既存への変更」と誤判定されるため。
   初回に限り、旧評価語・記述誤りの一度限りの是正と、§6/§7/版ヘッダの新規整備が許される
   （`review-gate.md`）。取り込み元が不明なら、どこから正典化するかを `FORK` としてユーザーに尋ねる。
3. 変更内容（思考実験の結論／ユーザー決定／発見された欠陥）を一段落で明示する。
4. 触れる要素を列挙する: MODEL 系なら **公理(A)・4イベント・状態・不変条件(I)・原理(P)・EARS
   要件(R-U/E/S/T/C/D)**。設計物その他なら該当する手順・規約・主張。
5. 触れる要素が「**構造**（表現可能なもの／手順の骨格）」か「**コミット判断**（人間に委ねるもの）」
   かを区別する（空虚性の回避）。

## 2. 敵対的レビューループ（ラウンドは対象の通し番号で継続）

各ラウンド = 「敵対派遣 → 分岐振り分け → パッチ → 採点」。`review-gate.md` の上限（既定 3、
`--rounds N` は N≥2）まで。1ラウンド内のサブエージェント数は `model-and-subagent-policy.md` の
上限に従う。

1. **敵対派遣**: `moira-adversary` を**並列・独立**に複数起動。派遣プロンプトに、対象ファイル・
   今回の変更・文脈と、**対象種別に応じた攻撃角を読め**と指示する:
   - MODEL 系 → `adversarial-vectors.md` の「モデル攻撃角」V1–V6（必要性の反証／矛盾狩り／過剰主張
     監査／スケール強制／空虚性／完全性）。
   - 設計物その他（本スキル・エージェント等）→ 同ファイルの「スキル攻撃角」S1–S8。
2. **分岐の振り分け**: `FORK`（**事実・論理では決められず人間の意図に属する選択**）が出たら、
   **AskUserQuestion で必ずユーザーに質問**する。勝手に決めない。多立場の設計問題は `moira-adversary`
   を発散立場で並列起動（多者ディベート）→主コンテキストで統合。事実・ドメイン主張（語源・EVM
   理論等）は `moira-fact-checker` を `NO_OBJECTION` まで（最大3巡）反復。
3. **パッチ**: Critical/Important 指摘を、主コンテキストが対象ファイルへ反映する（**パッチは著者＝
   主コンテキストのみ**。レビューワーは書き換えない）。Critical/Important を**反証で退ける**場合は、
   その反証を**同一ラウンド内で別の敵対者に差し戻して再反論**させる（次ラウンドに先送りしない／
   `review-gate.md`）。相反する指摘は著者が独断裁定せず、採点者またはユーザー（FORK）が裁定し、棄却
   側は「裁定により棄却」として決着させる。最小性を保つ——新公理/新イベント/新状態は除去不能の証明が
   ない限り入れない。
4. **採点**: `moira-gate-judge` を1体起動し、対象の現状・全指摘・各指摘の決着（修正済み/再反論を経た
   反証）・相反裁定・分岐のルーティング状況を渡して PASS/FAIL を取る。
5. FAIL なら次ラウンドへ。**最終ラウンドで採点 PASS**になったら §3 へ（収束は「永久に穴が出ない」
   ではなく「その時点の独立攻撃を生き延びた」を意味する／`review-gate.md`）。

## 3. 確定ゲート（`review-gate.md`）

確定してよいのは次が**すべて**満たされたときのみ:
- `moira-gate-judge` が**反論されない Critical = 0、かつ Important は全件 disposition 済み**（修正／健全な反証／追跡付き deferred）と判定（反証は再反論を経た健全なもの）。
- 相反指摘が採点者により裁定済み。
- genuine な分岐がすべてユーザーに回された。
- 上限ラウンド打ち切り時に未決着の Critical または disposition されない Important が残っていない（残るなら**確定しない**）。

満たさなければ確定せず、残課題を §7（または報告）に正直に出して終了する。

## 4. 確定（ゲート通過後のみ・対象種別で分岐）

確定は**全ステップ完遂で一つの不可分な単位**（部分適用禁止／`review-gate.md`）。

**MODEL 系の場合:**
1. 対象ファイルを**その場で編集**（単一の真実源。版番号入りの第二ファイルを作らない）。ブートストラップ
   時は、ここで初めて `moira/MODEL.md` を `Write` する（これがゲート通過後の唯一の実体化点）。
2. 版ヘッダを更新。
3. **§6 来歴に vN→vN+1 の主要転換を追記**。評価語は**「独立採点者の残存 Critical = 0・Important 全件 disposition 済み／
   反証された主要指摘」**で記す。旧来歴行は遡及改変しない（初回正典化時の既知誤り是正のみ例外）。
4. **§7 確認事項**を正直に保つ（解消したら空、残れば明記）。版ヘッダ・§6・§7 が互いに整合した状態で
   のみ確定済みとみなす。
5. `moira/DECISIONS.md` が存在すれば、解消した分岐を追記。

**設計物その他（命名文書・本スキル・エージェント定義）の場合:**
- 当該ファイルをその場で編集（単一の真実源）。§6/§7/版ヘッダは無いので要求しない。
- 確定の来歴は、**同じ編集の中で**当該ファイル/ディレクトリ内の在処（変更ログ、無ければ簡潔な末尾
  注記）に「ゲート通過（残存 Critical = 0・Important 全件 disposition 済み）＋主要変更」として残す（編集と来歴記録を別操作に
  せず一回の書き込みに畳み、部分適用を防ぐ）。コミットメッセージにも記すが、それを在処の代替にしない
  （事後検出できないため）。
- 本スキル自身・エージェント定義を改訂した場合も、同じゲートを通したことを記録する。

## 5. ユーザーへの正直な報告

- 発見した本物の穴と、それをどう塞いだか。
- **自己の過去主張の撤回**を含む正直な訂正（例: 「2箇所→3箇所」型）。
- 意図的に持たせなかったものと、その理由。
- ユーザーに回した分岐とその決着。
- 確定した版（MODEL 系）または改訂内容、と来歴行/コミット記録。

## 出力フォーマット（verdict）

```md
## Moira Update Verdict
- TARGET_KIND: MODEL | 設計物その他
- CHANGE: <一段落>
- TOUCHED: <触れた要素>
- ROUNDS: <各ラウンドの 攻撃→パッチ 要約>
- FORKS_RAISED: <ユーザーに回した分岐と決着>
- CORRECTIONS: <撤回・訂正（自己の過去主張含む）>
- GATE: <moira-gate-judge の最終判定。残存 Critical = 0・Important 全件 disposition 済みの証跡>
- FINAL: <確定 vN+1 + §6 来歴行（MODEL 系）/ 改訂＋来歴記録（その他）/ または「確定せず：残課題」>
```

## 既存スキルとの棲み分け

- `kiro-review`: タスク**実装**が凍結済み spec に従うかのレビュー。
- `kiro-verify-completion`: 完了主張の新鮮な証拠ゲート。
- **本スキル**: **正典設計文書そのもの**を凍結前に敵対的に鍛える。層が異なり重複しない。
