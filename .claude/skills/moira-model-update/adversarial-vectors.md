# 攻撃角カタログ / Adversarial Vectors

`moira-adversary` に派遣時に読ませる攻撃角の一覧。**点数表ではない**——「何を攻撃するか」の観点集。
対象に応じて「モデル攻撃角」または「スキル攻撃角」の節を指定して使う。

旧 v9 までの「6次元10点満点」採点はゲートから廃止された（自己採点インフレで識別力がないため）。
6次元の語彙は、ここでは**攻撃の切り口**として再利用する。点数は付けない。

---

## モデル攻撃角（対象: `moira/MODEL.md` 等の正典設計物）

セッション実績で実際に穴を出した6ベクトル。各角は「破れたら何が壊れるか」を伴って当てる。

### V1. 必要性の反証（最小性）
新規/影響を受けた**公理・イベント・状態・不変条件・原理・要件**を1つずつ「除去」する思考実験を行う。
除去しても全 UC・全決定・思考実験が表現できるなら、その要素は**冗長**＝削るべき。
- 実例: 4イベント（transition/decompose/relate/cost）は各々の除去を試み、relate は feature 横断依存ゆえ単一 decompose で表現不能、cost は遷移なき implementing 中に連続発生するため遷移に載せきれない——として除去不能を証明した。
- 報告には「除去手順」と「それでも成立する/しない根拠」を必ず添える。

### V2. 矛盾狩り（整合性）
公理・原理(P)・不変条件(I)・要件(R) の間の自己矛盾を探す。
- 実例: 「EV は保存しない（導出のみ）」と言いながらキャンセル分を「サンク台帳」に**保存**していた矛盾。台帳を廃止し導出に統一して解消。
- 一方が他方を否定する箇所を引用で対にして示す。

### V3. 過剰主張・隠れ前提の監査（正直さ）
「常に」「すべて」「不可能」等の断定、暗黙の前提、根拠なき歴史的主張を疑う。
**対象が自分（または過去の版）で書いた主張も例外なく疑い**、撤回すべきものを名指す。
- 実例の撤回: 「調整ノブは2箇所」→実は3箇所だった／「A5 は公理」→実は定理だった（P へ降格）／「仕様駆動が見積コスト構造を変えた」→誤った歴史的主張で撤回。

### V4. スケール強制（拡張性）
大規模プロジェクトを想定し、破綻を探す。
- 辺の爆発（decompose 時に流出辺を全子へ継承すると 50子×5辺=250辺）→ 流出辺をノード水準で保持して解消（R-D4）。
- 並列エージェントの単一ログ追記衝突 → feature 単位分割 + (ts,id) マージ。
- 再ベースライン・版管理が履歴として追えるか。

### V5. 空虚性攻撃（関心分離 / 識別力）
「何でも『人間が決める(P0)』に溶かしてモデルが空虚（vacuous）になっていないか」を突く。
- 反証手段: **構造とコミットの境界**を示せるか。モデルは構造（表現可能なもの＝ノード/辺/イベント/導出）を厳密に固定し、コミット判断（粒度・割当・スコープ・見積の深さ）だけを人間に委ねている、と示せれば空虚でない。示せなければ Critical。

### V6. 完全性（未処理ケース）
UC1–UC3・全決定・二つの思考実験（フェーズ移行/エンハンス）・一様な見積モデルが、現行の公理＋イベント＋不変条件で**漏れなく**表現できるか。未処理ケースが1つでも残れば Critical。

## スキル攻撃角（対象: `moira-model-update` スキル一式 + `.claude/agents/` の3定義）

スキルは概念モデルでなく**実行される手続き**。品質軸が違うので、以下の運用的な角で攻撃する。

**まず必要性の反証（V1 のスキル版）を必ず当てる**: 各手順・各ファイル（SKILL.md / SKILL.ja.md /
shared-rules 各 / 各エージェント定義）・各ルールを1つずつ「除去」する思考実験を行い、除去しても
手続きが成立するなら**冗長**として報告する。その上で以下の運用角を当てる:

### S1. 前提ゲートの実効性
- Opus 4.8 未満で**本当に停止するか**。モデル判定の手段は現実的か（環境からモデル ID を確認する手順が書かれているか）。
- effort max を要求する手段は実在するか、あるいは「ユーザーに依頼して確認」に正直に落ちているか。
- 全サブエージェント派遣に `model: opus` 指定と「最大徹底度・真に敵対的」指示が漏れなく書かれているか。

### S2. 攻撃角の網羅性
- レビュー失敗モードに抜けはないか。モデル攻撃角6本＋本スキル攻撃角で、実在する欠陥クラスを取りこぼしていないか。

### S3. ゲートの抜け穴
- **採点者と被採点者が分離**されているか（著者＝主コンテキスト、敵対者＝moira-adversary、判定＝moira-gate-judge が別主体か）。
- 「反論されない Critical/Important = 0」が骨抜きにできないか（甘い反証を PASS にする経路がないか）。
- **停止性**: 収束しない場合の上限ラウンドと、打ち切り時に「未決着が残るなら確定しない」が明記されているか。コスト暴走ガードはあるか。**最終ラウンドの反証・ユーザー決定後の生存指摘など、停止性デッドロック（確定不能に陥る経路）を必ずシミュレートせよ。**

### S4. 分岐処理の一義性
- 「genuine な分岐はユーザーに質問、勝手に決めない」が曖昧でなく実行可能に書けているか。分岐と単なる指摘の区別基準が示されているか。

### S5. 確定ステップの安全性
- 単一の真実源（MODEL.md その場編集、第二ファイルを作らない）・§6 来歴追記・§7 正直さが、手順どおりやれば壊れないか。
- 途中失敗時（パッチ後にゲート FAIL 等）の不整合が起きないか。

### S6. 規約・整合
- frontmatter schema（name/description/allowed-tools/argument-hint/model/metadata.origin/shared-rules）が正しいか。
- `SKILL.md` と `SKILL.ja.md` の frontmatter が一致し、内容が等価か。
- shared-rules の3ファイル名が実在ファイルと一致して解決するか。
- エージェント定義の `tools` に Write/Edit が**入っていない**こと（レビューワー非パッチの独立性）。

### S7. トリガー精度
- description が過剰起動（無関係な話題で発火）も起動漏れ（モデル更新依頼で発火しない）も招かないか。

### S8. 一般的欠陥
- 曖昧さ・矛盾・手順抜け・無限ループ risk・自己言及の破綻。
