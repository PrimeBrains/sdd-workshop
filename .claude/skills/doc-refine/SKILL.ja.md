# doc-refine — 確定文書を独立敵対者ループで磨いて確定する

このリポジトリの確定文書（既定: `.claude/agents/*.md` の専門家エージェント定義。`moira/`・`.kiro/`
配下の確定ナレッジ等にも使える）への変更を、**独立した敵対者による多重レビューループ（falsifiable な
確定ゲート）**を通してからのみ確定する。著者が自分で攻撃し自分で合格を出すことを禁じ、識別力ある
ゲートで「壊れていない」を有限・falsifiable に示す。

規律は3つの補助ファイルに分かれている。**派遣・判定の前に必ず読む**:
- `model-and-subagent-policy.md` — 推奨モデル/effort（best-effort・強制しない正直さ）、サブエージェント派遣、ラウンド内コスト上限。
- `adversarial-vectors.md` — 攻撃角カタログ（「一般文書 攻撃角」G1–G4／「エージェント定義 攻撃角」D1–D6）。
- `review-gate.md` — falsifiable 確定ゲート、三者分離、同一ラウンド内の反証再反論、相反裁定、一次資料確定、停止性。

## 引数

- `<対象>`: ファイルパス（例 `.claude/agents/moira-adversary.md`）またはエージェント名（例 `moira-adversary`）。**複数渡された場合は原則1対象ずつループを回す**（サブエージェントの上限は `model-and-subagent-policy.md` 参照で「1対象×1ラウンド」単位。複数を1ラウンドにまとめると各対象の攻撃が薄まり「最大徹底度」と衝突するため）。
- `--rounds N`: 上限ラウンド（既定 8、**N ≥ 2 を強制**。N<2 は拒否）。

## 手順（規範）

### 0. 起動前提（best-effort）
`model-and-subagent-policy.md` に従い、Opus 4.8+ / effort max を推奨表示。環境の exact model ID が
Opus 4.8 未満と判れば `/model` 切替を勧める。判らなければ正直にそう伝える。機械強制はしない。

### 1. 対象の分類と攻撃角の選定
対象が `.claude/agents/*.md`（エージェント定義）なら `adversarial-vectors.md` の**「エージェント定義 攻撃角」**を、
`moira/`・`.kiro/` 等の解説・設計・評価文書なら**「一般文書 攻撃角」**を使う。両方該当なら両方。エージェント名で渡されたら
`.claude/agents/<名前>.md` に解決する。

### 2. 一次資料の正をユーザーに確定（必須ゲート）
fact-checker の**検証の土台＝一次資料が誤っていれば検証ごと崩れる**。よって fact-checker を回す前に:
- 対象が「正」として引く一次資料の候補を提示する（例: `CLAUDE.md`、`.kiro/steering/` の該当章、`.kiro/specs/<feature>/` の確定文書、`moira/MODEL.md` 等の確定ナレッジ、このリポの確定ブランチ断面）。**特定資産にハードコードせず、対象が実際に依拠する文書を都度同定する。**
- `AskUserQuestion` で**どれを検証の土台にするかユーザーに確定させ、その確定記録を残す**（この記録が後段ゲートの `SOURCE_SET_CONFIRMED` の根拠。doc-fact-checker の `SOURCE_SET: OK` は『リストが渡った』ことしか意味せず、ユーザー確定の代わりにはならない）。
- 確定した一次資料リストを、以降の doc-fact-checker 派遣プロンプトに必ず埋め込む。
- **ユーザーが一次資料を確定できないとき（該当する確定文書が未作成、正となる断面が存在しない等）は永久ブロックにしない**: その主張を「削除する／`⚠️推測`・`未確認` へ hedge 化する／UNVERIFIABLE と明示して事実主張を取り下げる」のいずれにするかを `AskUserQuestion` でユーザーに決めてもらい、決した記録をもって決着扱いとする（review-gate.md 基準4の脱出弁）。

### 3. 敵対者ラウンド（穴出し）
`doc-adversary` を**並列・独立**に起動（`model-and-subagent-policy.md` のコスト上限内。攻撃最大3体）。
各体に「対象・選定した攻撃角節・確定した一次資料・最大徹底度で真に敵対的に・不確実なら破れている側に倒せ」を渡す。
互いの結論は見せない。

### 4. 分岐のユーザールーティング
敵対者が `FORK` として挙げた、または分岐と認めた**人間の意図に属する判断**（事実・論理で決められない選択）は、
著者が勝手に決めず `AskUserQuestion` でユーザーに回す。

### 5. 事実検証
敵対者の `CLAIMS_FOR_FACT_CHECK` と、文書中の事実主張（バージョン・ブランチ・テーブル名・本数・パス等）を
`doc-fact-checker` に渡す（ステップ2で確定した一次資料を明示）。CORRECTED/UNVERIFIABLE は修正対象。
同一主張群の反復は最大3巡（`model-and-subagent-policy.md`）。

### 6. パッチ（著者のみ）
指摘・訂正・ユーザー決定を反映して**主コンテキストだけ**が対象をその場編集する。レビューワーは書き換えない。
反証で退ける Critical/Important は、**同一ラウンド内**で別の `doc-adversary` に差し戻して再反論させる（先送り禁止）。

### 7. ゲート判定（毎ラウンド）
`doc-gate-judge` を1体起動し、`review-gate.md` 基準で PASS/FAIL を取る。FAIL なら次ラウンドへ（3 に戻る）。

### 8. 確定
次が**すべて**満たされたら確定: 生存 Critical/Important = 0／相反指摘は裁定済／**一次資料がユーザー確定済**／
全 FORK ルーティング済／停止性（上限ラウンド打ち切り時に未決着が残るなら確定しない・残課題を正直に報告）。
確定（コミット/push）は CLAUDE.md / リポのブランチ運用に従う（`main` への直コミットは避け、必要ならブランチを
切ってから。コミット/push はユーザーが依頼したときのみ行う）。

## 自己適用時の注意（対象が doc-refine 自身のとき）

本機構で **doc-refine 自身のファイル**（`doc-*.md` や本 skill・補助doc）を磨くと、攻撃に使う規律と
被攻撃対象が同一物になり、ループ途中の編集が「攻撃のルールそのもの」を書き換えて三者分離・ゲート基準が
揺らぐ。これを避けるため、自己適用時は次のように**版（スナップショット）で土台と対象を分離**する（実行可能な手順として）:
- 主コンテキストが**ループ開始時点の規律ファイル内容を `Read` してスナップショットとして固定し、派遣
  プロンプトに本文として埋め込む**（コミット済みなら `git show <commit>:<path>` でも可だが、**未コミットでも
  成立するよう Read による固定を既定とする**——doc-refine 一式は untracked のことがあり `git show` が失敗する）。
  レビューワー（`doc-adversary` / `doc-gate-judge`）が作業ツリーの現行版を Read すると編集途中版を見てしまうため、
  攻撃に使う固定版はプロンプト経由で渡す。
- 攻撃・判定の**土台＝開始時スナップショット（プロンプト埋め込み）**、**攻撃対象＝編集中の作業ツリー版**と
  して版で区別する。対象が規律ファイル自身でも、土台と対象は別版なので「対象側にのみパッチ」が破綻しない。
- 規律そのものを変える変更は、スナップショットを取り直した**次ラウンド**から有効化する。

## 既存スキルとの境界

- `code-review` / `simplify`: **コードの diff** をレビュー/整理する。
- `verify`: 実挙動でコード変更を確認する。
- `moira-model-update`: Moira の**正典モデル（`moira/MODEL.md`）専用**に同型の確定ゲートを回す。
- **本スキル**: **ドキュメント/エージェント定義そのもの**を凍結前に堅くする汎用版（コードではなく文書が対象）。
