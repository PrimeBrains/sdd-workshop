# 攻撃角カタログ / Adversarial Vectors

`doc-adversary` に派遣時に読ませる攻撃角の一覧。**点数表ではない**——「何を攻撃するか」の観点集。
対象に応じて「一般文書 攻撃角」を基本に、エージェント定義なら「エージェント定義 攻撃角」を上乗せして使う。
各角は「破れたら何が壊れるか」を伴って当てる。点数は付けない（自己採点はゲートから廃止済み）。

---

## 一般文書 攻撃角（対象: 確定ナレッジ・設計/評価文書・手順書・カタログ・README 等）

すべての確定文書に当てる基本の角。エージェント定義にもまずこれを当てる。

### G1. 必要性の反証（最小性）
各要素（節・項目・bullet・手順ステップ・ルール・表の行・補助ファイル）を1つずつ「除去」する思考実験を行う。
除去しても文書の目的（読み手が達成すべきこと）が達成できるなら、その要素は**冗長**＝削るべき。
- 報告には「除去手順」と「それでも成立する/しない根拠」を必ず添える。

### G2. 矛盾狩り（整合性）
文書内の自己矛盾、関連文書間（兄弟ファイル・上位カタログ・索引）の矛盾、一次ナレッジとの不整合を探す。
- 一方が他方を否定する箇所を引用で対にして示す。
- 「索引/目次に載っているが本体に無い」「本体にあるが索引に無い」も矛盾として扱う。

### G3. 過剰主張・隠れ前提・正直さ
「必ず」「すべて」「常に」「不可能」「自動で」等の断定、暗黙の前提、根拠なき主張、陳腐化した記述を疑う。
- `⚠️推測`・`[推測]`・`[要検証]`・「未確認」の hedge を**外して断定化**していないか。逆に確証があるのに過剰 hedge していないか。
- **対象が自分（または過去の版）で書いた主張も例外なく疑い**、撤回すべきものを名指す。

### G4. 完全性・出典整合（未処理ケース／裏取り可能性）
- 文書が「扱う」と宣言した範囲（ケース・利用シナリオ・入力・対象）が、本文で**漏れなく**扱われているか。1つでも未処理なら Critical/Important。
- 事実主張（数値・バージョン・名称・本数・パス・日付）が**出典に到達可能**な形で書かれているか。到達不能なものは `doc-fact-checker` 行きの候補として列挙する。
- 一次資料の同定が曖昧（「正」がどれか不明）なら、それ自体を指摘または FORK で上げる。

## エージェント定義 攻撃角（対象: `.claude/agents/*.md`。一般文書 攻撃角に上乗せ）

エージェント定義は「読まれる文書」であると同時に「起動・委譲される実行設定」でもある。運用面の角を足す。

### D1. トリガー精度
- description が**過剰起動**（無関係な話題で発火）も**起動漏れ**（本来発火すべき依頼で沈黙）も招かないか。
- 起動キーワード・用途が具体的で、判別可能か。

### D2. 委譲境界の整合（フリート分割）
- `.claude/agents/` の他エージェントと**境界が重複/欠落**していないか（このリポの実フリート例: `moira-adversary`／`moira-gate-judge`／`moira-fact-checker`／`doc-adversary`／`doc-gate-judge`／`doc-fact-checker`／`sdd-issue-creator`。例: 敵対者系↔採点者系↔事実検証系の役割が混線していないか、moira-* と doc-* の住み分けが宙に浮いていないか）。
- 「これは○○エージェントを使う」という相互参照が双方向で整合し、宙に浮いた領域・二重管轄が無いか。

### D3. 知識ソース整合
- 本文が「必ず Read」と指す一次ナレッジのパスが**実在し**、参照が解決するか。
- 「正」とする資料（一次ナレッジ／確定ブランチ断面）が明示され、記憶依存に逃げていないか。

### D4. 保有ツールと主張する操作の一致
- 本文が促す操作（例: `git show`/`git grep`、Web 検索）に必要なツール（Bash、WebSearch 等）が frontmatter `tools` にあるか。無ければ実行不能な手順＝Critical。
- レビューワー/採点者系なら **Write/Edit が `tools` に入っていない**こと（非パッチの独立性）。

### D5. frontmatter schema
- `name` が**ファイル名と一致**するか。`description`/`tools`/`model` が妥当か。YAML が壊れていないか。

### D6. 回答作法の実行可能性・自己整合
- 「根拠を付す」「推測と事実を区別」等の作法が、保有ツールと矛盾せず実際に実行可能か。
- 「★必ず警告する落とし穴」等が本体の主張と矛盾していないか、陳腐化していないか。
