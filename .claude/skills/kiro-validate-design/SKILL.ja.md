# kiro-validate-design スキル

## 役割
技術設計の品質レビューを対話的に実施し、実装に進める状態かを確認する専門スキル。

## コアミッション
- **ミッション**: 技術設計の品質レビューを対話的に実施し、実装準備が整っていることを確認する
- **成功基準**:
  - 重大な問題の特定（最重要の懸念は最大 3 件）
  - 強みも認めるバランスの取れた評価
  - 根拠を伴う明確な GO/NO-GO 判定
  - 改善が必要な場合の実行可能なフィードバック

## 実行ステップ

### Step 1: コンテキストの収集

ステアリング/スペックのコンテキストが会話から既に得られている場合、重複するファイル読み込みはスキップする。
そうでなければ、必要なコンテキストをすべてロードする:
- `.kiro/specs/{feature}/spec.json` を読み、言語とメタデータを取得
- `.kiro/specs/{feature}/requirements.md` を読み、要件を取得
- `.kiro/specs/{feature}/design.md` を読み、設計ドキュメントを取得
- コアステアリングコンテキスト: `product.md`、`tech.md`、`structure.md`
- 追加のステアリングファイルは、アーキテクチャ境界、統合、ランタイム前提条件、ドメインルール、セキュリティ/パフォーマンス制約、実装準備に影響するチーム規約に直接関連する場合のみ
- ローカルのエージェントスキルやプレイブックは、対象機能の実行環境やユースケースに明確に合致し、レビューに有用なコンテキストを提供する場合のみ

#### 並列リサーチ

以下のリサーチ領域は独立しており、並列実行できる:
1. **コンテキストとルールのロード**: スペックドキュメント、コアステアリング、タスクに関連する追加ステアリング、関連するローカルのエージェントスキル/プレイブック、レビュー基準としてこのスキルのディレクトリにある `rules/design-review.md`
2. **コードベースパターン調査**: レビュー時の参照として、コードベースから既存のアーキテクチャパターン、命名規約、コンポーネント構造を収集する

すべての並列リサーチ完了後、調査結果を統合してレビューを行う。

### Step 2: 設計レビューの実行
- 会話履歴を参照: これまでの要件に関する議論とユーザーが示した設計意図を活用する
- design-review.md のプロセスに従う: 分析 → 重大な問題 → 強み → GO/NO-GO
- 最重要の懸念は 3 件までに絞る
- ユーザーと対話的に進める — 確認の質問を投げかけ、代替案を提案する
- 出力には spec.json で指定された言語を使用する

### Step 3: 判定と次のステップ
- 根拠を伴う明確な GO/NO-GO 判定
- 具体的で実行可能な次のステップを提示する（下記の「次フェーズ」を参照）

### Step 4: バリデーションレポートの永続化

レビュー結果を `.kiro/specs/{feature}/validation-design.md` に書き込む（存在する場合は上書き。過去の実行結果は git 履歴に残る）。このファイルは GUI ツール（SDD Dashboard）が読み取るため、frontmatter のキーは以下のとおり厳密に維持すること:

```markdown
---
type: design
feature: {feature}
date: YYYY-MM-DD
decision: GO | NO-GO
---

# Design Validation: {feature}

## Review Summary
[2-3 sentences]

## Critical Issues
[max 3, with requirement/design references per trace-notation.md; "None" if clean]

## Strengths
[1-2 positive aspects]

## Rationale
[why GO / NO-GO, and agreed follow-ups]
```

本文には spec.json で指定された言語を使用する。NO-GO の場合もファイルを書き込むこと — 未解決の問題こそが価値である。

## 重要な制約
- **完璧の追求ではなく品質保証**: 許容可能なリスクは受け入れる
- **重大事項のみに集中**: 最大 3 件、成功に大きく影響するものに限る
- **会話を意識する**: 要件のコンテキストとユーザーの意図について議論の履歴を活用する
- **対話的アプローチ**: 対話を重ね、確認の質問を投げかけ、代替案を提案する
- **バランスの取れた評価**: 強みと弱みの両方を認識する
- **実行可能なフィードバック**: すべての提案は実装可能でなければならない
- **コンテキスト規律**: コアステアリングから始め、レビューに関連するステアリングやユースケースに合致するローカルのエージェントスキル/プレイブックに限って拡張する

## ツールガイダンス
- **Read を最初に**: レビュー前にスペック、コアステアリング、関連するローカルのプレイブック/エージェントスキル、ルールをロードする
- **必要に応じて Grep**: パターン検証や統合チェックのためにコードベースを検索する
- **対話的に**: レビュープロセス全体を通じてユーザーと対話する

## 出力の説明
spec.json で指定された言語で、以下を含む出力を提供する:

1. **レビューサマリー**: 設計品質と実装準備状況の簡潔な概要（2〜3 文）
2. **重大な問題**: 最大 3 件、design-review.md のフォーマットに従う
3. **設計の強み**: 1〜2 件の良い点
4. **最終評価**: 根拠と次のステップを伴う GO/NO-GO 判定

**フォーマット要件**:
- 明確さのために Markdown 見出しを使用する
- design-review.md の出力フォーマットに従う
- サマリーは簡潔に

## 安全性とフォールバック

### エラーシナリオ
- **設計の欠落**: design.md が存在しない場合、次のメッセージで停止: "Run `/kiro-spec-design {feature}` first to generate design document"
- **設計が未生成**: spec.json で設計フェーズが generated になっていない場合、警告したうえでレビューを続行する
- **ステアリングディレクトリが空**: プロジェクトコンテキストが欠如しておりレビュー品質に影響しうる旨をユーザーに警告する
- **言語が未定義**: spec.json で言語が指定されていない場合は英語（`en`）をデフォルトとする

### 次フェーズ: タスク生成

**設計がバリデーションを通過した場合（GO 判定）**:
- 合意した改善案があれば適用する
- `/kiro-spec-tasks {feature}` を実行して実装タスクを生成する
- または `/kiro-spec-tasks {feature} -y` で自動承認してそのまま進める

**設計に修正が必要な場合（NO-GO 判定）**:
- レビューで特定した重大な問題に対処する
- 改善を反映して `/kiro-spec-design {feature}` を再実行する
- `/kiro-validate-design {feature}` で再バリデーションする

**注**: 設計バリデーションは推奨だが任意。品質レビューは問題の早期発見に役立つ。
