# 抽出フェーズ（atelier-draft）

`atelier/METHODOLOGY.md` の「② 抽出（Extraction）」フェーズを実行する。
対話ログ（`atelier/dialogues/{topic}.md`）を読み込み、
ワークショップ素材として使える構造化ドラフト（`atelier/drafts/{topic}.md`）を生成する。

## 1. トピックの特定

ユーザーがトピックを明示した場合（例: "why-sdd のドラフト"）はそれを使う。
明示されなかった場合:

```bash
ls atelier/dialogues/
```

利用可能なファイルをユーザーに提示し、どのトピックを抽出するか選択させる。

## 2. 対話ログの検証

`atelier/dialogues/{topic}.md` を読み込む。以下を確認する:

1. **Q&A 数**: `**Q**:` の出現回数をカウント。3 件未満は警告を出すが処理は続行する
2. **Insights セクション**: `## Insights（抽出メモ）` の有無を確認
   - **あり**: Insights を主要インプットとして使う（対話セッションが事前整理済み）
   - **なし**: Q&A ペアから直接抽出し、ドラフトにその旨を注記する
3. **継続マーカー**: `*(セッション継続:` があればファイル全体を統合ログとして扱う

**ファイルが存在しない場合**:
`atelier/dialogues/{topic}.md が見つかりません。` と表示し、
`ls atelier/dialogues/` の結果を示す。`/dialogue-session` を先に実行するよう促す。

## 3. 既存ドラフトの確認

```bash
ls atelier/drafts/ 2>/dev/null || echo "(空)"
```

**ファイルが存在する場合**:
ユーザーに確認する:
> 「`atelier/drafts/{topic}.md` が既に存在します。**上書き**しますか、**末尾に追記**しますか？（デフォルト: 追記）」

- 追記: 既存ファイルに `---` セパレーターと日付ヘッダーを付けて追加
- 上書き: 新規として書き込む

**ディレクトリが存在しない場合**:
```bash
mkdir -p atelier/drafts
```

## 4. 抽出チェックリストの適用

対話ログ全体（Q&A + Insights）を読み、METHODOLOGY.md の 5 点チェックを実行する。

### チェック 1: 誤解パターン
- Insights の `**誤解パターン:**` から取得。なければ Q&A の訂正パターンを探す
- 「それは〜ではありません」「正確には」などの言い回しに注目
- 誤解と正しい理解の対を表形式で整理

### チェック 2: 理解の転換点
- Insights の `**理解の転換点:**` から取得。なければ前提が大きく更新された箇所を特定
- 「何が変わったか」「なぜここが転換点か」を一文で補足

### チェック 3: 比喩・アナロジー
- Insights の `**使えそうな比喩・表現:**` から取得
- なければ「〜は〜と同じ」「〜に相当する」「〜のような」パターン、「」強調フレーズを収集
- 各比喩に「使える場面（講義のどの文脈か）」を付記

### チェック 4: 向いている場面・向いていない場面
- 「向いている」「向いていない」「適しない」「コストが見合う」などが含まれる箇所を抽出
- 正と負の両面を整理する

### チェック 5: 演習シナリオ候補
- Insights の `**ドラフト化候補:**` を参照
- 参加者が自分で判断を下せる、または手を動かせる Q&A ペアを特定
- 各シナリオに「設定・問い・期待される気づき・参照 Q&A 番号」を付与

## 4.5 公式ドキュメントとの照合

対話ログに含まれる **主要な主張・定義・事実** を、Kiro 公式ドキュメントと照合する。
ドラフトに「## 7. 公式ドキュメント照合」セクションを生成するための素材を収集する。

### 照合対象ドキュメントの特定

対話ログの末尾に「## 関連 Kiro 公式ドキュメント」セクションがある場合はそこを使う。
ない場合は以下のマスターマップから関連するページを選択して WebFetch で取得する:

| トピックキーワード | Kiro ドキュメント URL |
|---|---|
| SDD の定義・スペック構造 | `https://kiro.dev/docs/specs/` |
| フィーチャースペック・ワークフロー | `https://kiro.dev/docs/specs/feature-specs/` |
| EARS 形式・要件記述 | `https://kiro.dev/docs/specs/analyze-requirements/` |
| 仕様とコードの整合性・Property-based Testing | `https://kiro.dev/docs/specs/correctness/` |
| 向いているケース・ベストプラクティス | `https://kiro.dev/docs/specs/best-practices/` |
| バグ修正 vs 仕様変更 | `https://kiro.dev/docs/specs/bugfix-specs/` |
| 小さな変更への対応 | `https://kiro.dev/docs/specs/quick-plan/` |
| ステアリング・WHY の記録・DDD 接続 | `https://kiro.dev/docs/steering/` |

**取得方法**: WebFetch ツールで各 URL を取得し、該当する記述を抜き出す。
取得できなかった場合（404・アクセス不可）は「取得不可」と記録して処理を続行する。

### 照合の実行

対話ログから「ワークショップで主張として使う重要な文」を 5〜10 件選出し、
各文について以下を判断する:

- **一致**: 公式ドキュメントに同じ内容が明示されている → 原文 URL を引用
- **拡張**: 公式にはない補足・解釈を対話が加えている → 拡張として明示
- **差異**: 公式と対話の記述が食い違う → 差異を明示し、どちらを採用するか注記
- **取得不可**: 対応ドキュメントが存在しないまたは取得できなかった

差異があった場合は対話ログの記述を優先せず、公式の記述を正として扱い、
ドラフトの該当箇所（誤解パターン・転換点等）に補正メモを入れる。

## 5. ドラフトの書き込み

`atelier/drafts/{topic}.md` に以下の構造で書き込む。

**新規または上書きの場合:**

```markdown
# ドラフト: {トピック名}

**抽出元**: `atelier/dialogues/{topic}.md`
**抽出日**: {今日の日付}
**対話Q&A数**: {n}件
**Insightsセクション**: {あり / なし（Q&Aから直接抽出）}

**抽出フェーズ完了チェック**:
- [x or space] 誤解パターン
- [x or space] 理解の転換点
- [x or space] 比喩・アナロジー
- [x or space] 向いている/向いていない場面
- [x or space] 演習シナリオ

---

## 1. 誤解パターン

> **ワークショップでの使い方**: 参加者が同じ誤解を持つ前提で講義を組み立てる。冒頭の「あるある」として提示すると引きが生まれる。

| 誤解 | 正しい理解 |
|------|-----------|
| ... | ... |

---

## 2. 理解の転換点

> **ワークショップでの使い方**: 講義の山場・クライマックスを設計する核。Q&A 形式でそのまま使える。

- **転換点 {n}**: {概要}
  - 引用: 「{核心フレーズ}」
  - なぜ転換点か: {一文説明}

---

## 3. 比喩・アナロジー一覧

> **ワークショップでの使い方**: スライドの見出し、ホワイトボードへの板書、口頭説明のフックとして。

- 「{比喩フレーズ}」
  — 使える場面: {説明}

---

## 4. 向いている場面・向いていない場面

> **ワークショップでの使い方**: 「SDDをいつ使うか」という判断軸を参加者に渡す。グループワークの判断基準としても使える。

**向いている場面:**
- {条件}

**向いていない場面（または注意が必要な場面）:**
- {条件}

---

## 5. 演習シナリオ候補

> **ワークショップでの使い方**: ハンズオンや振り返りディスカッションの問い。

### シナリオ {A/B/...}: {タイトル}
- **設定**: {前提状況}
- **問い**: {参加者に考えさせる問い}
- **期待される気づき**: {腑に落ちてほしいこと}
- **参照Q&A**: Q{n}（対話ログ参照）

---

## 6. スライド素材候補

> **ワークショップでの使い方**: docs/ 昇格時にスライドタイトルや図の素材として。Insights の「ドラフト化候補」も参照。

- **{スライドタイトル案}**: {内容の概要と使える比喩・データ点}

---

## 7. 公式ドキュメント照合

> **ワークショップでの使い方**: 講師が「この内容は公式に確認済み」と参加者に伝える根拠になる。
> 差異がある箇所はコンテンツ修正の優先候補。

| 対話での主張 | 公式ドキュメントの見解 | 照合結果 | 参照 |
|---|---|---|---|
| {対話ログからの主張} | {公式テキストの該当記述または「記載なし」} | 一致 / 拡張 / 差異 / 取得不可 | [ページ名](URL) |

**差異・要注意箇所:**
- {差異があった項目と対処方針}

**対話が公式を超えて補足している点:**
- {拡張として扱う記述の一覧}

---

## 抽出メモ（レビュアー向け）

{薄かったセクション・追加対話セッションで補うべき点・補足観察}
```

**追記の場合:**

既存ファイルの末尾に追加:

```markdown

---

*(追記: {今日の日付} — 対話ログ追加分の抽出)*

## 追記: {セクション名}（{日付}）
...（新たに追加された内容のみ）
```

## 6. 完了報告と次のアクション提案

抽出完了後、以下の構成でまとめを出力する:

```
## 抽出完了: atelier/drafts/{topic}.md

各セクションの抽出結果:
- 誤解パターン: {n}件
- 理解の転換点: {n}件
- 比喩・アナロジー: {n}件
- 向いている/向いていない場面: {あり / 対話ログから抽出できませんでした}
- 演習シナリオ候補: {n}件
- スライド素材候補: {n}件

{薄いセクションがある場合の警告}

---

## 次のアクション

1. **ドラフトのレビュー** — atelier/drafts/{topic}.md を確認し、
   スライド素材候補や演習シナリオを肉付けする

2. **対話セッションの追加** — 薄かったセクションを補強する場合は
   /dialogue-session でトピック「{topic}」を継続

3. **他トピックの抽出** — 対話ログがあるトピック:
   {ls atelier/dialogues/ の結果から未ドラフト分を列挙}

4. **docs/ への昇格判断** — METHODOLOGY.md の昇格基準を確認
   （全トピックのドラフト完了・参加者レビュー済み・わかりにくい箇所修正済み）
```
