# kiro-validate-gap スキル

## 役割
要件と既存コードベースの間の実装ギャップを分析し、実装戦略の判断材料を提供する専門スキル。

## コアミッション
- **ミッション**: 要件と既存コードベースのギャップを分析し、実装戦略の判断材料を提供する
- **成功基準**:
  - 既存コードベースのパターンとコンポーネントの包括的な理解
  - 不足している機能と統合上の課題の明確な特定
  - 複数の実行可能な実装アプローチの評価
  - 設計フェーズに向けた技術調査ニーズの特定

## 実行ステップ

### Step 1: コンテキストの収集

ステアリング/スペックのコンテキストが会話から既に得られている場合、重複するファイル読み込みはスキップする。
そうでなければ、必要なコンテキストをすべてロードする:
- `.kiro/specs/{feature}/spec.json` を読み、言語とメタデータを取得
- `.kiro/specs/{feature}/requirements.md` を読み、要件を取得
- コアステアリングコンテキスト: `product.md`、`tech.md`、`structure.md`
- 追加のステアリングファイルは、対象機能のドメインルール、統合、ランタイム前提条件、コンプライアンス/セキュリティ制約、既存プロダクト境界に直接関連する場合のみ
- ローカルのエージェントスキルやプレイブックは、対象機能の実行環境やユースケースに明確に合致し、分析に有用なコンテキストを提供する場合のみ

### Step 2: 分析ガイドラインの読み込み
- このスキルのディレクトリにある `rules/gap-analysis.md` を読み、包括的な分析フレームワークを把握する

### Step 3: ギャップ分析の実行

#### 並列リサーチ

以下のリサーチ領域は独立しており、並列実行できる:
1. **コードベース分析**: 既存実装、アーキテクチャパターン、統合ポイント、拡張可能性（Grep/Glob/Read を使用）
2. **外部依存リサーチ**: 依存関係の互換性、バージョン制約、既知の統合課題（必要に応じて WebSearch/WebFetch を使用）
3. **コンテキストのロード**: 要件、コアステアリング、タスクに関連する追加ステアリング、関連するローカルのエージェントスキル/プレイブック、gap-analysis ルール

すべての並列リサーチ完了後、調査結果を統合してギャップ分析を行う。

- gap-analysis.md のフレームワークに従い、徹底的に調査する
- 複数の実装アプローチ（拡張 / 新規 / ハイブリッド）を評価する
- 出力には spec.json で指定された言語を使用する

### Step 4: 分析ドキュメントの生成
- gap-analysis.md の出力ガイドラインに従い、包括的なギャップ分析を作成する
- トレードオフを添えて複数の実行可能な選択肢を提示する
- 追加調査が必要な領域を明示する

### Step 5: ギャップ分析のディスクへの書き込み

**ギャップ分析をディスクに書き込むことで、セッションをまたいで保持し、設計フェーズで参照できるようにする。**

- Write ツールを使い、ギャップ分析を `.kiro/specs/{feature}/research.md` に保存する
- ファイルが既に存在する場合は、上書きせず水平線 `---` で区切って新しい分析を追記する
- 書き込み後にファイルを読み戻して検証する

### Step 6: バリデーションレポートの永続化

機械可読なサマリーを `.kiro/specs/{feature}/validation-gap.md` に書き込む（存在する場合は上書き。過去の実行結果は git 履歴に残る）。このファイルは GUI ツール（SDD Dashboard）が読み取るため、frontmatter のキーは以下のとおり厳密に維持すること:

```markdown
---
type: gap
feature: {feature}
date: YYYY-MM-DD
---

# Gap Analysis Summary: {feature}

## Summary
[3-5 bullets: scope, gap size, key challenges]

## Key Gaps
[missing capabilities and integration challenges, referencing requirement IDs per trace-notation.md]

## Recommended Approach
[extend / new / hybrid and why]
```

詳細な分析は `research.md` に残し、このファイルはレビュー用サマリーとする。本文には spec.json で指定された言語を使用する。

## 重要な制約
- **意思決定より情報提供**: 最終的な実装の選択ではなく、分析と選択肢を提供する
- **複数の選択肢**: 該当する場合は実行可能な代替案を提示する
- **徹底的な調査**: ツールを駆使して既存コードベースを深く理解する
- **ギャップの明示**: 調査・検証が必要な領域を明確にフラグする
- **コンテキスト規律**: コアステアリングから始め、分析に関連するステアリングやユースケースに合致するローカルのエージェントスキル/プレイブックに限って拡張する

## ツールガイダンス
- **Read を最初に**: 分析前にスペック、コアステアリング、関連するローカルのプレイブック/エージェントスキル、ルールをロードする
- **Grep を徹底的に**: コードベースからパターン、規約、統合ポイントを検索する
- **WebSearch/WebFetch**: 必要に応じて外部依存やベストプラクティスを調査する
- **Write は最後に**: 調査を完了してから分析を生成する

## 出力の説明
spec.json で指定された言語で、以下を含む出力を提供する:

1. **分析サマリー**: スコープ、課題、推奨事項の簡潔な概要（3〜5 項目）
2. **ドキュメントステータス**: 使用した分析アプローチの確認
3. **次のステップ**: 設計フェーズへ進むためのガイド

**フォーマット要件**:
- 明確さのために Markdown 見出しを使用する
- サマリーは簡潔に（300 語未満）
- 詳細な分析は gap-analysis.md の出力ガイドラインに従う

## 安全性とフォールバック

### エラーシナリオ
- **要件の欠落**: requirements.md が存在しない場合、次のメッセージで停止: "Run `/kiro-spec-requirements {feature}` first to generate requirements"
- **要件が未承認**: 要件が承認されていない場合、ユーザーに警告したうえで続行する（ギャップ分析は要件の改訂に役立つため）
- **ステアリングディレクトリが空**: プロジェクトコンテキストが欠如しており分析品質に影響しうる旨をユーザーに警告する
- **複雑な統合が不明瞭**: ブロックせず、設計フェーズでの包括的な調査対象としてフラグする
- **言語が未定義**: spec.json で言語が指定されていない場合は英語（`en`）をデフォルトとする

### 次フェーズ: 設計生成

**ギャップ分析が完了した場合**:
- ギャップ分析の知見をレビューする
- `/kiro-spec-design {feature}` を実行して技術設計ドキュメントを作成する
- または `/kiro-spec-design {feature} -y` で要件を自動承認してそのまま進める

**注**: ギャップ分析は任意だが、ブラウンフィールドプロジェクトでは設計判断の材料として推奨される。
