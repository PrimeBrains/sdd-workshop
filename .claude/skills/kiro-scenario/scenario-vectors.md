# シナリオ攻撃角カタログ / Scenario Adversarial Vectors

`doc-adversary` に派遣時に読ませる、受け入れシナリオ単位（`.kiro/scenarios/units/{slug}.md`）専用の攻撃角。
**点数表ではない**——「何を攻撃するか」の観点集。各角は「破れたら何が壊れるか」を伴って当てる。点数は付けない。

**まず `doc-refine` の一般文書 攻撃角 G1–G4（`../doc-refine/adversarial-vectors.md`）を当て**、その上で以下の SC1–SC7 を上乗せする。シナリオは「確定文書」であると同時に「人間が所有する受け入れ基準」でもあるため、運用面の角を足す。

---

## シナリオ 攻撃角（対象: `.kiro/scenarios/units/*.md`。一般文書 攻撃角に上乗せ）

### SC1. 完全性（抜けたシナリオ）
- §3 のふるまいが含意する**未記載のシナリオ／エッジケース**を探す（否定系・未承認のまま完了・cancel・supersede・並行・期日超過 等）。`.kiro/scenarios/README.md` が要求する「完全性クリティック」。
- 破れたら何が壊れるか: シナリオ通過＝検証済みという**誤った安心**を生む（サンプリング・オラクルの穴）。1つでも漏れていれば Important、ふるまいの中核を成すケースなら Critical。

### SC2. 現実接地（架空でないか）
- §4 画面が `moira/frontend` の実コンポーネント挙動と一致するか（存在しない画面・誤った遷移を描いていないか）。
- §5 ログの event JSON が `moira/backend/src/types.ts` の実イベント形（`kind`/フィールド）と一致するか。引用パスが実在するか。
- 破れたら何が壊れるか: 「実在に接地」というシナリオの主張が虚偽化し、下流実装者が誤った記録形・画面を正と誤認する。**事実主張は `CLAIMS_FOR_FACT_CHECK` として列挙し `doc-fact-checker` に回す。**

### SC3. MODEL/spec 矛盾（と ESCALATE 判定）
- §3 のふるまい・§6 の受け入れ条件が `moira/MODEL.md` 正典または引用 spec と矛盾しないか。見積の三者乖離のような**表現ギャップ**が無いか。
- 破れたら何が壊れるか: シナリオが正典に反する受け入れ基準を固定し、堅牢度の根が腐る。
- **ESCALATE 判定（重要）:** 矛盾の所在が**シナリオ文書側**なら著者がパッチで直す。矛盾の所在が **MODEL 正典側**（MODEL 自身が曖昧／自己矛盾／未規定）なら、`ESCALATE: moira-model-update` とタグ付けする——シナリオ文書のパッチで正典の穴を**糊塗してはならない**（今セッションの見積三者乖離がこの型）。

### SC4. 人間所有性（ふるまいの 0→1 は人間か）
- §3 のふるまいの**発案（0→1）が人間**か——人間が自分の言葉で語った意図の**転記**なら可、AI が spec/MODEL から**発案**したものなら不可（出所が疑わしければ FORK）。判定基準は「§3 に AI が書いたか」ではなく「**ふるまいを発案したのが誰か**」。人間の語りに無いふるまいが §3 に混入していれば（＝AI 由来の発案、または転記時に足された未確認の boundary）指摘する。
- §6 EARS の受け入れ条件が**非専門家でも falsifiable** か——テスト可能条件に moira 専門用語・skill 実名・MODEL 記号（R-xx/I-x 等）が混入し、人間がチェックできなくなっていないか。
- 破れたら何が壊れるか: シナリオが「AI が書いた仕様を AI が作ったシナリオで通す」自己採点に退化し、外的妥当性の検証という存在意義を失う。

### SC5. シーム被覆（継ぎ目を本当に行使するか）
- 単位が主張する `⟿`（spec/skill を跨ぐ継ぎ目）を**実際に行使**しているか。`⚠`（未実装 skill）を正直に標識しているか。
- 継ぎ目が**主張だけで実質を伴わない**（飾りの ⟿）になっていないか。継ぎ目こそ単体レビューの盲点ゆえ、ここが空なら単位の価値が失われる。
- 破れたら何が壊れるか: 「継ぎ目を行使する E2E」という最大の価値が虚偽化する。

### SC6. トレース全列挙
- frontmatter `touches_specs` / `touches_requirements` が `.kiro/steering/trace-notation.md` に従い**全列挙**されているか（範囲 `1.3-1.5`・ワイルドカード `15.*`・注釈付き・自由文は禁止）。
- 列挙された各 ID が**実在**するか。トレーサビリティ詳細を本文に重複させていないか（frontmatter に集約）。
- 破れたら何が壊れるか: grep ベースのトレーサビリティが中間 ID で破綻し、シナリオ↔要件の対応表が信頼できなくなる。

### SC7. README / 様式 整合
- 必須セクション §1–§7（`.kiro/scenarios/README.md` v0.2）が揃っているか。
- §4 に **HTML モックアップ と データ表の両方**があるか（言葉だけ禁止）。§5 が**3つの出力先**（イベントログ／会話ログ／画面）を名指すか。§6 が `.kiro/steering/requirements-style.md` の EARS トリガー（WHEN/WHILE …）を使うか。
- 本文に**未決の FORK が残っていないか**（決定だけを §7 に書く）。`agreed` 版に比較用の二重表現が残っていないか。
- 破れたら何が壊れるか: レビュー負荷低減・余計なことを書かない、という規約の目的が崩れる。

---

## ESCALATE 規則（節末・必須）

- **シナリオ文書の欠陥**（様式・接地・抜け・トレース）→ 著者（派遣元の主コンテキスト）が**シナリオ文書をパッチ**する。
- **MODEL 正典側の欠陥**（SC3 で MODEL 自身の曖昧/矛盾/未規定を検出）→ `ESCALATE: moira-model-update` とタグ付けし、派遣元は**確認のうえ `moira-model-update` へ委譲**する。**シナリオ単位スキルは `moira/MODEL.md` を編集しない。**
