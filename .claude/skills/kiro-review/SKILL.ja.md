# kiro-review

## 概要

このスキルはタスクローカルな敵対的レビューを実施する。実装が本物であり、完全で、境界内に収まり、承認済みの要件と設計に整合し、機械的な検証証拠に裏付けられていることを検証する。

境界に関する用語の連続性:
- discovery は `Boundary Candidates` を特定する
- design は `Boundary Commitments` を確定する
- tasks は `_Boundary:_` で実行を制約する
- review は具体的な `Boundary Violations` を却下する

## 使用タイミング

- 実装者が `READY_FOR_REVIEW` を報告した後
- 却下されたレビューの修正後
- タスクを `[x]` にマークする前
- タスクを feature レベル検証に受け入れる前

欠落した要件を発明したり、スペックを黙って再解釈したりするためにこのスキルを使わない。

## 入力

以下を提供する:
- タスク ID と `tasks.md` の正確なタスク本文
- 関連する要件セクション番号
- 関連する設計セクション番号
- スペックファイルのパス（`requirements.md`、`design.md`、任意で `tasks.md`）
- 実装者のステータスレポート
- タスクの `_Boundary:_` スコープ制約
- コントローラーが発見した検証コマンド
- 該当する場合、関連するステアリングの抜粋
- 該当する場合、関連する `## Implementation Notes` のエントリ

## 出力

以下のいずれかを返す:
- `APPROVED`
- `REJECTED`

加えて以下も返す:
- 機械的チェックの結果
- 重大度付きの所見
- 必須の修正事項
- 1 文のサマリー

`spec.json` で指定された言語を使用する。

## 最初のアクション

`git diff` を実行して実際のコード変更を確認する。差分が大きい、または曖昧な場合は変更ファイルを直接読む。実装者レポートを真実の源として信用しない。

## コア原則

スペックを自分で読む。差分を自分で読む。可能な限り機械的に検証する。解釈上の楽観ではなく、具体的な不合格に基づいて却下する。
レビューの主たる問いは「動くか？」だけでなく「新たな結合を隠さずに、承認済みの責務境界の内側に留まっているか？」である。

## 機械的チェック

これらのチェックを実行し、結果を第一のシグナルとして用いる。

### 1. リグレッション安全性
- コントローラーが発見した検証コマンドを用いて、プロジェクトの正規テストスイートを実行する。
- テストが失敗したら却下する。

### 2. 残存プレースホルダーマーカーの不在
- 変更ファイルに `TBD`、`TODO`、`FIXME`、`HACK`、`XXX` がないか確認する。
- タスク上の明示的な正当化なしに新しいプレースホルダーマーカーが導入されていたら却下する。

### 3. ハードコードされたシークレットの不在
- 変更ファイルにハードコードされたシークレットや認証情報がないか確認する。
- 具体的なシークレットパターンが導入されていたら却下する。

### 4. 境界の遵守
- 変更ファイルをタスクの `_Boundary:_` スコープと照合する。
- 明示的な正当化なしに変更が承認済み境界の外に漏れ出していたら却下する。
- ローカルであるべきタスクの内側に、隠れた境界横断の調整が導入されていたら却下する。

### 5. RED フェーズの証拠
- 振る舞い変更を伴うタスクについて、実装者のステータスレポートに `RED_PHASE_OUTPUT` が含まれることを検証する。
- RED の証拠が欠落、空、またはタスクの受け入れ基準と無関係なら却下する。

### 6. ランタイムに敏感な静的チェック
- 触れたスタックに対するリントや同等の静的解析がプロジェクトに既にある場合、タスク境界に関連するコマンドを実行する。
- 型チェック／ビルドを通過してもランタイムで失敗しうるパターンに注意する: 値として使われる型専用インポート、修飾名アクセスに対する名前空間値インポートの欠落、未解決のグローバル、対応する起動／ランタイム処理を伴わない新規のランタイム依存。
- プロジェクトにリントコマンドがない場合、変更ファイルに対してそれらのパターンを差分ベースでスポットチェックする。
- 起動時またはモジュールロード時の失敗を現実的に引き起こす具体的な所見があれば却下する。

## 判断チェック

### 7. 実在性チェック
- 実装が本物のプロダクションコードであり、プレースホルダー、スタブ、偽のパス、先送りの殻ではないことを確認する。

### 8. 受け入れ基準のカバレッジ
- タスク説明を読み、主要なハッピーパスだけでなく、すべての側面が実装されていることを確認する。

### 9. 要件との整合
- `requirements.md` の参照セクションを読む。
- 各要件が具体的で観測可能な振る舞いによって満たされていることを確認する。
- 元のセクション番号のみを使用する。

### 10. 設計との整合
- `design.md` の参照セクションを読む。
- 実装が規定された構造、インターフェース、依存方向を使っていることを確認する。
- 設計で義務付けられた選択の無断置き換えは却下する。

### 10.5 境界の監査
- 実装を設計の境界コミットメントおよび境界外宣言と照合する。
- 利便性のために下流固有の振る舞いが上流の境界に押し込まれていたら却下する。
- 実装が隣接境界間に新たな隠れ依存、共有オーナーシップ、未宣言の結合を作っていたら却下する。
- 明示的な統合タスクでないタスクが統合タスクのように振る舞っていたら却下する。

### 11. テスト品質
- テストが足場作りだけでなく、要求された振る舞いを証明していることを確認する。
- 実装を除去または破壊した場合にテストが失敗することを確認する。

### 12. エラーハンドリング
- 関連する失敗パスが処理されており、黙って握り潰されていないことを確認する。

## 重大度モデル

以下を使用する:
- `Critical`: 機能の破壊、無効な検証、データ損失、セキュリティリスク、重大なスコープ違反
- `Important`: 受け入れ前に必須の修正
- `Suggestion`: ブロックしない改善
- `FYI`: 参考情報

## 停止／エスカレーション

以下の場合、問題を糊塗せずエスカレーションする:
- 承認済みスペックが正しさに関わる点で曖昧
- 設計が技術的に可能なことと矛盾
- 必要な証拠を収集できない
- 承認済みスコープから黙って逸脱することでしか実装が動かない
- 要件・設計・タスクスコープから境界のオーナーシップを明確に判定できない

## よくある正当化

| 正当化 | 現実 |
|---|---|
| 「テストがパスしたので承認」 | テストのパスはスペック準拠や境界遵守を証明しない。 |
| 「この追加の振る舞いは有用だ」 | 承認済みスコープ外の追加の振る舞いは依然としてドリフトである。 |
| 「実装者が RED は完了したと言った」 | RED は主張ではなく証拠で示されなければならない。 |
| 「このギャップは見逃せるほど小さい」 | 実在するギャップは却下またはエスカレーションしなければならない。 |

## 出力フォーマット

```md
## Review Verdict
- VERDICT: APPROVED | REJECTED
- TASK: <task-id>
- MECHANICAL_RESULTS:
  - Tests: PASS | FAIL (command and exit code)
  - TBD/TODO grep: CLEAN | <count> matches
  - Secrets grep: CLEAN | <count> matches
  - Static checks: PASS | FAIL | SPOT_CHECKED
  - Boundary: WITHIN | <files outside boundary>
  - Boundary audit: CLEAN | <spillover / hidden dependency findings>
  - RED phase: VERIFIED | MISSING | N/A
- FINDINGS:
  1. <specific finding with exact files/spec refs>
- REMEDIATION: <mandatory if REJECTED>
- SUMMARY: <one sentence>
```
