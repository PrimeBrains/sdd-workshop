# kiro-spec-requirements スキル

## コアミッション
- **成功基準**:
  - ステアリングコンテキストに整合した完全な要件ドキュメントを作成する
  - すべての受け入れ基準にプロジェクトの EARS パターンと制約を適用する
  - 実装詳細を含めず、コア機能にフォーカスする
  - スコープが誤読されうる場合、含む/含まないの境界を明示する
  - 生成ステータスを追跡するためメタデータを更新する

## 実行ステップ

### ステップ 1: コンテキストの収集

ステアリング/スペックのコンテキストが会話から既に得られている場合、重複するファイル読み込みはスキップする。
そうでなければ、必要なコンテキストをすべて読み込む:
- `.kiro/specs/{feature}/spec.json` を読み、言語とメタデータを取得
- `.kiro/specs/{feature}/brief.md` が存在すれば読み込む（ディスカバリーコンテキスト: 問題、アプローチ、スコープ決定、バウンダリ候補）
- `.kiro/specs/{feature}/requirements.md` を読み、プロジェクト記述を取得
- コアステアリングコンテキスト: `product.md`、`tech.md`、`structure.md`
- 追加のステアリングファイルは、フィーチャーのスコープ、ユーザーペルソナ、ビジネス/ドメインルール、コンプライアンス/セキュリティ制約、運用制約、既存プロダクトの境界に直接関連する場合のみ読み込む
- ローカルのエージェントスキルやプレイブックは、フィーチャーのホスト環境やユースケースに明確に合致し、ユーザーが観測可能な要件を形づくるドメイン用語やワークフロールールを含む場合のみ参照する

### ステップ 2: ガイドラインの読み込み
- このスキルのディレクトリにある `rules/ears-format.md` を読み、EARS の構文規則を取得
- このスキルのディレクトリにある `rules/requirements-review-gate.md` を読み、書き込み前レビューの基準を取得
- `.kiro/settings/templates/specs/requirements.md` を読み、ドキュメント構造を取得

#### 並列リサーチ（サブエージェント派遣）

以下のリサーチ領域は独立している。プロジェクトの複雑さに応じて最適な分割を判断し、必要に応じてサブエージェントを分割・統合・追加・省略する。

**Agent ツールでサブエージェントへ委譲**（探索をメインコンテキストの外に保つ）:
- **コードベースの手がかり**（ブラウンフィールドプロジェクト）: 要件スコープに影響する既存実装の探索をサブエージェントに委譲する。プロンプト例: "Explore this codebase for existing features related to [feature area]. Summarize: (1) what already exists, (2) relevant interfaces/APIs, (3) patterns that new requirements should align with. Return a summary under 150 lines."
- **ドメインリサーチ**（外部知識が必要な場合）: WebSearch/WebFetch を使うサブエージェントを派遣し、ドメイン固有の要件・標準・ベストプラクティスを調査する。簡潔な調査結果サマリーを返させる。
- **追加ステアリングとプレイブック**: ステアリングファイルやローカルのエージェントプレイブックが多数存在する場合、サブエージェントに走査させ、このフィーチャーに関連するセクションのみ返させる。

コードベースが最小限のグリーンフィールドプロジェクトでは、サブエージェント派遣をスキップしてコンテキストを直接読み込む。

すべてのリサーチ完了後、要件生成の前にメインコンテキストで調査結果を統合する。

### ステップ 3: 要件ドラフトの生成
- プロジェクト記述に基づいて初期要件ドラフトを作成
- 関連する機能を論理的な要件エリアにグルーピング
- すべての受け入れ基準に EARS フォーマットを適用
- spec.json で指定された言語を使用
- フェーズをまたいだ用語の連続性を維持:
  - discovery = `Boundary Candidates`
  - requirements = 必要に応じて、含む/含まないの明示と隣接する期待事項
  - design = `Boundary Commitments`
  - tasks = `_Boundary:_`
- スコープが誤読されうる場合、実装やアーキテクチャ所有の詳細を持ち込まずに、軽量なバウンダリコンテキストを追加する
- レビューゲートを通過するまではドラフトのままにする。この時点で `requirements.md` を書き込まない

### ステップ 4: 要件ドラフトのレビュー
- `rules/requirements-review-gate.md` の `Requirements Review Gate` を実行
- 確定前に、カバレッジ、EARS 準拠、曖昧さ、隣接する期待事項、スコープ境界をレビュー
- 問題がドラフト内に閉じている場合、要件を修復して再レビュー
- レビューは最大 2 回の修復パスまでに制限する
- ドラフトが本物のスコープの曖昧さや矛盾を露呈した場合、推測で要件を書かずに停止してユーザーに明確化を求める

### ステップ 5: 確定とメタデータ更新
- 要件レビューゲートを通過した後にのみ `.kiro/specs/{feature}/requirements.md` を書き込む
- `phase: "requirements-generated"` を設定
- `approvals.requirements.generated: true` を設定
- `updated_at` タイムスタンプを更新

## 重要な制約

### 要件のスコープ: HOW ではなく WHAT
要件は実装ではなく、ユーザーが観測可能な振る舞いを記述する。ここに属するものと設計に属するものは以下で判断する:

**ユーザーに尋ねるべきこと（要件スコープ）:**
- 機能スコープ — 何が含まれ、何が除外されるか
- ユーザーが観測可能な振る舞い — 「X が起きたとき、ユーザーは何を見る/体験するべきか?」
- ビジネスルールとエッジケース — 上限、エラー条件、特殊ケース
- ユーザーに見える非機能要件 — 応答時間の期待値、可用性、セキュリティレベル
- 隣接する期待事項は、ユーザーに見える振る舞いや運用者の期待を変える場合に限る — このフィーチャーが何に依存し、何を明示的に所有しないか

**尋ねないこと（設計スコープ — 設計フェーズへ先送り）:**
- 技術スタックの選択（データベース、フレームワーク、言語）
- アーキテクチャパターン（マイクロサービス、モノリス、イベント駆動）
- API 設計、データモデル、内部コンポーネント構造
- 非機能要件をどう達成するか（キャッシュ戦略、スケーリング手法）
- 設計に属する内部所有マッピング、コンポーネントの継ぎ目、実装上の境界

**リトマス試験**: EARS の受け入れ基準を技術に言及せずに書けるなら要件に属する。技術選択が必要なら設計に属する。

### その他の制約
- 各要件はテスト可能かつ曖昧さのないものでなければならない。プロジェクト記述がスコープ・振る舞い・境界条件について複数の解釈の余地を残す場合、その要件を生成する前にユーザーに明確化を求める。必要なだけ質問し、自分の推測を含む要件を生成しない
- EARS 文には適切な主語を選択する（ソフトウェアではシステム/サービス名）
- requirements.md の要件見出しは先頭に数値 ID のみを含めなければならない（例: "Requirement 1"、"1."、"2 Feature ..."）。"Requirement A" のような英字 ID は使用しない

## 出力の説明
spec.json で指定された言語で、以下を含めて出力する:

1. **生成した要件のサマリー**: 主要な要件エリアの簡潔な概要（3-5 箇条書き）
2. **ドキュメントステータス**: requirements.md の更新と spec.json メタデータの更新を確認
3. **レビューゲート**: 要件レビューゲートの通過を確認
4. **次のステップ**: 進め方をユーザーに案内（承認して続行、または修正）

**フォーマット要件**:
- わかりやすさのため Markdown 見出しを使用
- ファイルパスはコードブロックで記載
- サマリーは簡潔に保つ（300 語以内）

## 安全策とフォールバック

### エラーシナリオ
- **プロジェクト記述の欠落**: requirements.md にプロジェクト記述がない場合、ユーザーにフィーチャーの詳細を尋ねる
- **テンプレート欠落**: テンプレートファイルが存在しない場合、警告を出してインラインのフォールバック構造を使用する
- **言語未定義**: spec.json に言語指定がない場合、英語（`en`）をデフォルトとする
- **要件の不完全さ**: 生成後、要件が期待される機能をすべてカバーしているかをユーザーに明示的に確認する
- **ステアリングディレクトリが空**: プロジェクトコンテキストが欠落しており要件の品質に影響しうる旨をユーザーに警告する
- **数値でない要件見出し**: 既存の見出しが先頭の数値 ID を含まない場合（例: "Requirement A"）、数値 ID に正規化し、そのマッピングを一貫して維持する（数値ラベルと英字ラベルを混在させない）

### 次のフェーズ: 設計生成

**要件が承認された場合**:
- **任意のギャップ分析**（既存コードベース向け）:
  - `/kiro-validate-gap {feature}` を実行し、実装ギャップを分析
  - ブラウンフィールドプロジェクトに推奨。グリーンフィールドではスキップ
- `/kiro-spec-design {feature}` を実行して設計フェーズへ進む
- または `/kiro-spec-design {feature} -y` で要件を自動承認して直接進む

**修正が必要な場合**:
- フィードバックを提供し、`/kiro-spec-requirements {feature}` を再実行
