# コンテンツ制作方法論

## 概念：ソクラテス式知識抽出

ユーザーが「SDD を何も知らない開発者」を演じて質問し、Claude が SDD 経験者として答える対話を蓄積する。
その対話ログをベースにコンテンツを作ることで、実際のビギナーが詰まるポイントを拾った深みのある素材が生まれる。

---

## ワークフロー

```
① 対話セッション（Dialogue）
   ユーザー = SDD未経験の開発者ペルソナ
   Claude  = SDD経験者
   → atelier/dialogues/{topic}.md に蓄積

② 抽出（Extraction）
   対話ログから「なるほど」ポイント・比喩・誤解パターンを拾い出す
   → atelier/drafts/{topic}.md に整理

③ 結晶化（Crystallization）
   draft を docs/ の形式（スライド・手順書・リファレンス）に仕上げる
   → docs/ へ移動
```

---

## セッションの始め方

### 1. ペルソナを決める

対話を始める前に「どんな開発者として質問するか」をメッセージの冒頭で宣言する。

例:
> 「Java歴3年、普段は Spring Boot で API を作っています。Claude Code は触り始めたばかりで、SDD という言葉は聞いたことがある程度です。このペルソナで質問します。」

ペルソナを固定することで、Claude の回答の粒度が揃い、後でコンテンツに変換しやすくなる。

### 2. トピックを1つに絞る

1セッション = 1トピック。複数のテーマを混ぜると対話ログが整理しにくくなる。

| ファイル名 | 対応モジュール | テーマ |
|-----------|-------------|--------|
| `why-sdd.md` | Module 1 | なぜ仕様駆動開発か |
| `workflow.md` | Module 2 | cc-sdd ワークフロー全体像 |
| `forward-flow.md` | Module 3 | Forward flow（新規機能） |
| `steering.md` | Module 5 | ステアリング強化 |
| `maintenance.md` | Module 6 | 仕様変更と保守 |
| `test-review.md` | Module 7 | テスト・レビュー |
| `team-ops.md` | Module 8 | チーム運用 |

---

## 対話ログのフォーマット

`atelier/dialogues/{topic}.md` に以下の形式で記録する。
対話中にリアルタイムで Claude が書き足していく。

```markdown
# 対話ログ: {トピック名}

**セッション日**: YYYY-MM-DD
**ペルソナ**: {宣言したペルソナの説明}

---

**Q**: {質問}

**A**: {回答}

---

**Q**: ...
```

---

## 良い質問の条件

- **具体的な疑問**: 「なんとなくわかった気がするが、実際どう使うの？」という感覚を言語化する
- **誤解ベース**: 「〜だと思っていたのですが違いますか？」という形式は特に有効
- **「なぜ」を掘る**: 「どうやるか」だけでなく「なぜそうするのか」を聞く
- **自分の文脈に当てはめる**: 「今のプロジェクトで言うと、〇〇に相当しますか？」

---

## 抽出フェーズのチェックリスト

対話ログを読み返して `drafts/` に整理するときに確認する項目:

- [ ] ビギナーが最初に誤解しやすいポイントは何か
- [ ] 「そこか！」という理解の転換点になった質問はどれか
- [ ] 使いやすい比喩・アナロジーが生まれたか
- [ ] 「向いている場面・向いていない場面」の判断基準が出てきたか
- [ ] ワークショップの演習として切り出せるシナリオはあるか

---

## docs/ への昇格基準

`drafts/` のコンテンツを `docs/` に移すタイミング:

- 対象モジュールの全トピックの対話ログが揃っている
- 抽出フェーズのチェックリストを完了している
- 実際の参加者（またはレビュアー）に一読してもらい、わかりにくい箇所を修正済み
