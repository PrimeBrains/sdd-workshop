# プロパティベーステスト（PBT）学習対話ログ

> `article.md`（「AI時代の開発プロジェクトにおいて、ボトルネックは何か？」）の②プロパティベーステストの節を起点に、Moira の実際の PBT 実装（`moira/backend/src/pbt/`）に即して、PBT の核心——「入力とは何か」「生成器の設計」「質的な隅の列挙」「人間のレビュー範囲」「現新比較テストとの関係」「業務制約下の生成器」「AI による SPEC 抽出」——を掘り下げた学習セッションの記録。

---

## Q1. EV が必ず 0〜100% に収まる、という性質をどんな生成器で大量生成しているのか。そもそも「何を入力と捉えるのか」。Moira だとログの組み合わせ？ ログは1億でも1兆でも出せるが、どこまでやるのか。

Moira には実際に PBT 実装がある（`moira/backend/src/pbt/arbitraries.ts` と `green-properties.pbt.test.ts`）。抽象論ではなく `PR-EVPCT-RANGE` の実装そのものに即して、3つの問い（①入力とは何か ②生成器は何か ③無限の入力空間をどこで打ち切るか）に答える。

### ① 「何を入力と捉えるか」— ここが PBT の設計の8割

入力は **「ログ（イベント列）の組み合わせ」**。ただし生成器が直接ランダムなイベント配列を吐くのではなく、**一段抽象化した「構造化スペック」（`ProjectSpec` / `LeafSpec`）を生成し、それをコンパイルして整形式（well-formed）のログに変換する**二段構え。

```
ProjectSpec = { leaves: LeafSpec[] }   // 葉ノード 1〜5 個
LeafSpec = { estimate, agree, lifecycle, cost, scheduled, revertAfterAgree, supersededByNew }
```

`compileLog()` が `decompose / transition / cost / relate` の4種イベント列に展開する。検証対象はルートの `evPercent`。

なぜ生のイベント配列を直接生成しないか：ランダムなイベント配列はほとんどが「親のない子」「存在しないノードを完了」といったゴミログになり、性質検証の前に弾かれる。だから生成器は整形式ログだけを吐き、**そのうえで意地悪な隅（hard corners）に意図的に到達する**ように設計される。これが記事の「テストデータ生成器の設計の難しさ」。

### ② 生成器の正体 — `fast-check` の arbitrary を組み合わせる

`fast-check`（JS/TS の PBT フレームワーク、QuickCheck 相当）の arbitrary コンビネータ（`fc.integer` / `fc.option` / `fc.constantFrom` / `fc.boolean` / `fc.record` / `fc.array`）で「葉の属性 → 葉 → プロジェクト」を再帰的に組み上げる。

Moira 特有の工夫：
- **到達 witness（self-coverage）**: ランダム生成は「意地悪なケースに到達していないのにグリーン」という偽の安心を生む。`cornersOf()` で各ケースがどの隅に当たったかを分類し、**1000サンプル中で全隅が最低1回踏まれていること自体をテスト**する。
- **決定性の強制**: `Math.random`/`Date.now` を禁止し `(ts, id)` を連番で打つ。`PM-ORDER-INV`（記録順を入れ替えても結果同一）を検証するため、物理順だけを `permute()`（LCG で決定的にシャッフル）で変えて比較する。

### ③ 無限の入力空間をどこで打ち切るか

**(A) ケース数は有限で打ち切る（原理的妥協）**: `RUNS = 500`。1つの性質につき500本のランダムなログを試すだけ。PBT は証明ではなく「バグの存在は示せても不在は証明できない」（ダイクストラ）。確率的な歯止め。

**(B) 1本のログのサイズも生成器が小さく上限を切る（意図的設計）**: `fc.array(arbLeaf, { minLength: 1, maxLength: 5 })`。葉は最大5個。**Small Scope Hypothesis（小スコープ仮説）**——「不変条件を破るバグは、たいてい小さな反例でも再現する」。0%〜100% が破れるなら葉2〜3個の意地悪な組み合わせで十分破れる。**shrinking**（反例の自動最小化）もこの設計と噛み合う。

## Q2. ①がまだ分からない。ProjectSpec/LeafSpec はログ履歴丸ごとを定義したもので、これがテストデータの大元だよね。実際のプロダクションコードの `derive` で読んで EV% を導出している？ `derive` を使うと全葉ノードの EV% が取れて、その全部に 0%以上・100%以下をアサートしている？

データの流れの理解は正しいが、最後の一点だけ事実と違う。**`evPercent` は「全葉ノードの EV% の配列」ではなく、プロジェクト全体の達成率を表すたった1つの数値**。

### ステップで追う

```
spec        = arbProjectSpec が生成した「プロジェクト1個の骨組み」   ← ①生成器の出力
  │  ▼ compileLog(spec)
log: Event[] = ログ履歴まるごと                                       ← ②実際のログ
  │  ▼ derive(log, OPTS)    ← プロダクションコード（本物の導出エンジン）
d: DerivedState = 導出結果のかたまり                                  ← ③本番の計算結果
  │  ▼ d.evPercent
0.0〜1.0 の数値 ひとつ                                                ← ④検証対象
```

### 直すべき一点 — EV% は「葉ごと」ではなく「プロジェクト1個に1つ」

`types.ts:165` に `evPercent: number; // ∈ [0,1]`。配列ではない。`computeEvPercent`（`ev.ts`）の中身：

```ts
function computeEvPercent(...): number {
  let denom = 0;
  for (const id of eff.effectiveLeaves) {     // 有効な葉を全部なめて
    if (n.estimateState !== 'agreed') continue; // 「合意済み」の見積りだけ
    denom += n.latestEstimate;                 // 分母 = 合意済み葉の見積り合計
  }
  return denom > 0 ? evAbs / denom : 0;         // 達成率 = 稼いだ価値 / 全体の見積り
}
```

葉が5個あっても畳み込んで最後に出るのは「このプロジェクトは全体として何%できているか」という1個の数。アサーションもその1個に対して `toBeGreaterThanOrEqual(0)` / `toBeLessThanOrEqual(1)`。

**「大量に」とは「葉ごと」ではなく「プロジェクト（＝ログ1本）ごとに大量」**。500個の別々のプロジェクトを生成し、出た500個の達成率スカラーが全部 0〜100% に収まるかを確かめる。

## Q3. なるほどプロジェクト全体の累積 EV か。1プロジェクト5個までしか葉ノードを付けなかったら、気づけない性質のバグもありそう。そういうのが設計の肝なのかな。

直感は正しい。そして「設計の肝はそこか？」への答えは**半分 Yes、でも肝の置き場所が少しズレている**。

### 小スコープは確かにバグを取りこぼす

- **累積・閾値バグ**: `estimate` 最大20・葉最大5 → 分母 `denom` はせいぜい100。浮動小数の精度ずれや桁あふれは規模が大きくならないと出ない。
- **深さ・幅の閾値バグ**: 木は `root → 葉`（or `root → G → 葉`）で深さ最大2。3段4段で壊れるバグは構造上到達不能。
- **多数同時の相互作用**: 5葉なら異なる隅の状態を同時に5種類しか並べられない。

### ただし「肝」は "何個にするか" ではなく "隅に到達できているか"

サイズ（maxLength）は本当に効く指標の雑な代理変数。バグは「葉が多い」から出るのではなく**システムが質的に違う状態（corner）に入った**ときに出る。0%〜100% が破れるとしたら、合意ゼロで `0/0`、全葉キャンセルで分母消滅、supersede で旧ノード消滅——**これらは全部、葉1〜2個で再現する**。

Moira はこれを**到達 witness（self-coverage）で機械化**：1000サンプルで6種の隅が全部踏まれたか自体をテストする。**5という数字が肝ではなく、「5個でも質的な隅を全部踏めることを機械的に保証している」のが肝**。

### 1つの生成器で全部は無理 → 生成器を分ける

Moira は1つの巨大生成器で網羅しない。`PR-DONE-LOCK` 違反（完了ノードを差し戻す）は整形式ログでは到達できないので**専用の別生成器**で狙い撃ち（`arbitraries.ts:18`）。**「広く浅い汎用生成器」＋「狭く深い狙い撃ち生成器」を複数持ち、それぞれが到達証明を伴う**。

理論的裏付け：①小スコープ仮説（Daniel Jackson の Alloy。定理でなく経験則）②shrinking との相性。切り捨てた領域（巨大規模の累積バグ）は PBT で追わず別手段で補う（記事の「E2E と PBT は相補的」と同じ構図）。

## Q4. 質的な隅をそもそもどう列挙するのかが気になる。めっちゃむずそう。

「めっちゃむず」は正しい。でも Moira のコードは、この難しさを**「ひらめき」から「機械的な手続き＋測定」に寄せている**。質的な隅は入力空間をにらんで奇抜な値を想像するのではなく、**すでにある成果物から"読み出す"**。出どころは4つ。

### 出どころ① モデルの規則そのもの（背骨）

`cornersOf()` の6つの隅には全部**規則 ID（R-U8, R-U4, R-E3, R-D7, R-S5, §2.3/I1）が付いている**。隅は思いつきではなく **MODEL に書かれた規則を1個ずつ"裏返したもの"**。Moira の状態は「イベントを畳み込んだ状態機械」で、**規則とはその状態機械のガード（境界）**。規則を列挙すれば隅が機械的に出る。記事の「普遍条件の抽出」の入力側の双子。

### 出どころ② プロダクションコードの分岐

`computeEvPercent` の `denom > 0 ? evAbs / denom : 0` の `: 0` 側（合意済みの葉が1つもない）は規則ではなく**コードの場合分け**。専用の隅テストがある。**本番コードの全分岐を入力側の隅に翻訳する**。カバレッジツールで半ば機械化できる。

### 出どころ③ 規則が"名指ししない"相互作用（本当に難しい）

どの規則も単独では書いていない**規則どうしの衝突**（supersede × cancel の到着順、R-U12 同時刻イベント）。`moira-verification.md:56` はこれを別手段（境界モデル検査、小スコープ「ノード≤4・actor≤3・イベント≤8」）に回す。ここは**敵対的思考**（AI が助けやすい）＋**モデル検査**（小領域を機械的に全探索）の合わせ技。

### 出どころ④ 過去に出たバグ（回帰を隅に昇格）

一度踏んだバグは永続の隅に変える。`PR-DONE-LOCK★` は専用生成器つき RED tripwire。隅の目録は運用しながら増える生き物。

### 「列挙しきれたと確信できない」問題への答え — 測定で falsifiable にする

1. **到達 witness（self-coverage）**: 列挙した隅に実際に到達したかを測る。踏まれない隅は赤く落ちて「未テストの不変条件」として晒される。
2. **mutation testing（非空虚性）**: 本番コードにわざとバグを注入し、その隅が最低1つの注入バグを殺せるか確認する。

「隅の列挙」という主観的作業が、「主張する／到達を証明する／効くことを証明する」という反証可能な3点セットに変わる。完全性は主張せず「宣言スコープ内の高被覆 falsification ＋ 有界 guarantee」と正直に言う。

### おまけ — 書いてはいけない隅・性質（deny-list）

「進捗は単調増加する」は書いてはいけない性質（差し戻しで下がるのが正しい挙動）。"魅力的だが間違った隅"を禁止リスト化するのも列挙の craft の一部。

## Q5. PBT 考えたやつ頭良すぎる。実際 test してバグが見つかるのは質的な隅が多い。到達チェックはフレームワークがやってくれる？ 空虚チェックは AI が自力で検証してるよね。

両方とも「フレームワーク自動」でも「AI が実行時に自力判定」でもなく、**人間/AI が書いた決定的なテストコードが CI で回っている**。AI が関わるのは*実行時の検証*ではなく*オーサリング時*。

### ① 到達チェックはフレームワークがやってくれる？ → No。手書き。

fast-check が提供するのは**サンプリングのプリミティブ（`fc.sample`）だけ**。`cornersOf()`（隅の分類器）も「全隅到達」のアサーションも**テスト著者が手で書いている**。何が"隅"かはドメイン固有なのでフレームワークには原理的に分からない。

ただし**この発想は良すぎて一部のフレームワークには標準搭載**：QuickCheck の `cover`/`checkCoverage`、Hypothesis の `event()`/`target()`。fast-check には無いので Moira は手で組んだ。

### ② 空虚チェックは AI が自力で検証してる？ → ズレ。機械的なコード。

**(a) いま走っている空虚チェックは手書きの決定的コード**：`done-lock.pbt.test.ts:71` に `// Non-vacuity:` 付きのアサーション。`it.fails` トリップワイヤ（既知の不整合を永続 RED として埋め込み、実装が直ったらファイルを赤く落とす）は手で仕込んだミュータント。AI の実行時判定ではなく CI で決定的にグリーン/レッドが出る。

**(b) 全自動の mutation testing（Stryker 等）は理想形でまだ未配線**：`package.json` には vitest / fast-check / coverage-v8 だけ。Moira 自身が「第1器 PBT は最小パイロット稼働」と正直に書いている。

### AI はどこに居るか — 実行時ではなくオーサリング時

隅を列挙する／`cornersOf` を書く／PROPERTIES.md の一文からオラクルを書き起こす——この**設計・執筆**は人間/AI の仕事。しかも「**プロパティ著者 ≠ 実装著者**」で相関誤りを断つ。でもチェックそのものは決定的なコードとして CI が回す。

> 対比：AI が実行時に自力照合する器は別にある（⑥ `decision-conformance`、③ `e2e-scenario-checker`）。PBT の空虚チェックはそっちではなく機械側。

## Q6. 人間はこの PBT コードのどこまでをレビューすればよいのか。

**原理：人間のレビュー量は、機械的な歯止めの強さに反比例させる。** 間違いが*うるさく落ちる*（決定的にレッド）ところは機械に任せ、間違いが*静かにグリーンになる*（空虚に真）ところに人間が立つ。

| 層 | 対象 | 人間の関与 | なぜ | 失敗の出方 |
|---|---|---|---|---|
| 1 | **性質の一文** | 全力＋批准（proposed→agreed） | 意図の本体。機械に検証不能 | 静かにグリーン |
| 2 | **性質の妥当性（deny-list）** | 全力 | ドメイン判断（Yes/No） | 静かにグリーン |
| 3 | **生成器** | 人間＋敵対者（機械が一部補助） | 未列挙の隅・過剰制約は静かに空虚化。`:99`「最大のリスク・一級レビュー対象」 | 静かにグリーン |
| 4 | **オラクルのコード** | スポット（意図と照合・実装とは照合しない） | トリップワイヤ等が一部担保 | 半分うるさい |
| 5 | **足場・本番コード** | 基本見ない | 決定的に落ちる／縮めたい面そのもの | うるさくレッド |

- **第3層の補足**：到達 witness が守れるのは*リストに載っている隅*だけ。「`cornersOf` に載せ忘れた隅は無いか」「生成器が整形規則を厳しくしすぎて面白い領域を削っていないか」という**盲点と過剰制約の判定**が人間の残余。
- **第4層の超重要な How**：オラクルを**実装コードと照らしてはいけない**（相関誤り）。照合先は常に"一文（意図）"。
- **第5層**：本番の `derive` を1行ずつ読まない。PBT を書くこと自体がこの本番コードへのレビューの代替。

人間は一人で全部やらない。敵対者エージェント＋「著者≠実装者」分離が先に攻めて面を削ってから人間に渡す。**静かに嘘をつける場所にだけ人間が立つ。** これが記事の「どこまでを人間が所有し、どこまでを AI に所有させるか」の PBT における具体的な答え。

## Q7. かつて車両登録 API のリプレイス開発で、現行踏襲要件のため8000件のリグレッションデータで現新比較をした。これも「現行で登録可能な車両は新でも登録可能」「固有識別子を除いて車種コードは同じ」という不変条件を確認するプロパティテストといえるよね。コンバート車両という特殊経路だけ NG が出たのが、まさに質的隅かなと思った。認識合ってる？

8割合っていて、2割だけ精密化したい。

| 主張 | 判定 |
|---|---|
| 現新比較は不変条件を確認する**プロパティテスト的**といえる | ⭕️ 正確には「**プロパティ型のオラクル × 例ベースの入力**」のハイブリッド |
| コンバート車両だけ NG = **質的隅** | ⭕️ 教科書通りの質的隅 |

### ① 現新比較は「プロパティテスト」か — 軸を分けると見える

**軸A：オラクル → 完全に PBT。しかも最強の型。** 検証した性質「∀ 車両 v：`register_新(v)` は固有 ID を除いて `register_現(v)` と等価」は**差分テスト（differential / back-to-back testing）**。PBT 最大の難所「普遍条件の抽出」を**丸ごと回避できる**——**現行 API そのものがオラクル**だから。「現行踏襲」リプレイスはこの手法のスイートスポット。しかも「固有の識別子を除いて」と除外したのは **deny-list の規律を直感でやっていた**（Moira が forecast の絶対値を性質に書かないのと同じ）。

**軸B：入力 → 純 PBT と違う。** 8000件の固定リグレッションデータ＝**例ベース**。生成器が乱数で無限に作るのと違い、(1) shrinking がない (2) 本番に存在しなかった有効入力に届かない。正確な名前は「**差分オラクル（PBT 的）を使った、コーパス・リプレイ型の回帰テスト**」。

### ② コンバート車両 = 質的隅 — 完全に正しい

「別の登録経路」という質的に異なるパスであって、量的極端値ではない。経路が違う＝通るコードが違う＝事前条件が違う。質的隅の定義そのもの。

### コーパスと生成器は「隅の発見プロファイル」が相補的

- **生成器ベース PBT**：自分がモデル化した範囲内で未到来の組み合わせまで届く＋shrink。でも**モデルに入れていない隅は永遠に作れない**。
- **コーパス・リプレイ**：人間が誰もモデル化していない隅が**現実のデータに最初から埋め込まれている**。コンバート車両を捕まえられたのは8000件に実物が混ざっていたから。

| 手法 | 強み | 盲点 |
|---|---|---|
| E2E/シナリオ | 現実の具体シナリオが妥当か | 有限の例しか踏めない |
| 生成器 PBT | モデル内の未到来の組み合わせ＋shrink | モデルに無い隅は作れない |
| **コーパス差分** | 現実が未知の隅を持ち込む＋オラクルがタダ | コーパスに無い有効入力に盲目・shrink なし |

当時は**普遍条件の抽出という最難関を「現行踏襲」で回避しつつ、現実データで質的隅を踏み抜く**という、PBT の美味しいところを直感で押さえていた。

## Q8. このとき PBT をやっていたらどうなったか考えたが、生成器を作るのがかなり大変だと思った。車両エンティティは項目が多く（車種・ローン・オプション・画像…）、ブランドがトヨタなら車種もトヨタの車種でないとバリデーションに引っかかる。正常な車両をランダムに N 個生成するには業務ルールを完全に理解して正常な車両を無限に作れないといけない。開発者に「正常系リクエストを適当に作って」と言うだけでも苦労していた。業務ルールがちゃんと spec になってないと無理だよね。

その結論は PBT の最も深い真実のひとつに自力でたどり着いている。

### 一番大事な洞察：生成器とは「正常な入力とは何か」の実行可能な仕様そのもの

「正常な車両を生成する」＝「正常な車両の定義を構成的に書き下す」。生成器を作る作業は**正常系の仕様を articulate する作業と同一**。「業務ルールが spec になってないと無理」＝「生成器を書く＝散らばっていた検証ルールを一箇所に集めて仕様化する」。生成器づくりは spec 発掘の別名。記事の主題（人間の仕事は意図の小さく安定した面に移る）に直結。

### 技術的勘所：「ランダム生成→バリデーションで弾く」は破綻する

正常な車両がランダム空間のごく一部だと99.99%が弾かれてサンプルがスカスカ＆偏る。正解は**「弾く」のではなく「最初から正しく組み立てる（valid-by-construction）」**：ブランドを引く→そのブランドの車種から引く→その車種で有効なオプションから引く…。**業務ルールの依存グラフを依存順に辿る。** これが Moira が生の Event を直接生成せず `ProjectSpec`→`compileLog()` でコンパイルしていた理由（車両 API の経験から独立に再発見）。

### コーパス方式が成立した理由 — validity-spec をタダで借りていた

8000件は全部「現行 API が受理した実績」＝**現実が"正常の証明書"を発行済み**。だから正常性の仕様を一行も書かずに正常な車両を8000個手に入れられた。

| | validity-spec のコスト | 到達できる入力 |
|---|---|---|
| **コーパス** | タダ（現実が正常性を保証） | 過去に来たものだけ（歴史に固定） |
| **生成器** | 高い（全業務ルールを構成的に書く） | 正常な入力すべて（未到来も合成できる） |

「生成器は大変」は、コーパスの天井を破るための代金。

### 罠：生成器を「実装から逆算」してはいけない

「API が通すものを正常とする」生成器は実装のバグを相続し、コンバート車両を一度も生成せずバグに永遠に到達しない。コーパスはこの罠を踏まなかった（新 API のバグより前に作られたから）。生成器の正常性ルールは**実装ではなく仕様（意図）から**引く。

### 「spec が無いと無理」の精密化：生成器づくりは spec を強制的に生み出す装置

「トヨタのブランド→トヨタの車種」を書こうとした瞬間、OEM 供給は？ 生産終了車種は？ コンバート車両はどのブランド扱い？ と問いに突き当たる。**生成器の構築が曖昧だった仕様を尋問して穴を表面化させる。** 全か無かではなく、スコープ宣言＋漸進で進める。

## Q9. 正常な車両の定義ができたうえでランダム生成するには、各項目の型や値域を定義することになる。でも「車種がこれのときは値域がここからここまで、車種が違うと値域が変わる」みたいなのはどう記述する？

答えは1つの仕組みに集約される：**`chain`（モナド的バインド）＝「先に決めた値を読んで、次の生成器を切り替える」**。

### 直接の答え：決定要因を先に引き、それに依存して値域を切り替える

```ts
const SPEC = {
  prius:       { priceMin: 200, priceMax: 400, options: ['solar', 'leather'] },
  landcruiser: { priceMin: 500, priceMax: 900, options: ['offroad', 'tow'] },
} as const;

const arbVehicle = fc
  .constantFrom(...Object.keys(SPEC))        // ① まず車種を引く（決定要因）
  .chain((model) => {                         // ② その車種を読んで…
    const s = SPEC[model];
    return fc.record({                        // ③ 値域が車種依存の生成器を返す
      model:  fc.constant(model),
      price:  fc.integer({ min: s.priceMin, max: s.priceMax }), // ← 車種ごとに違う値域
      option: fc.constantFrom(...s.options),
    });
  });
```

`map` が「生成した値を変換」なのに対し、**`chain` は「生成した値を使って次にどの生成器を使うか自体を決める」**。条件付き値域が `if` 文ではなく生成器の合成として書ける。多段依存（ブランド→車種→グレード→オプション→価格）は `chain` を入れ子にして上流から順に確定。依存グラフのトポロジカル順＝生成順。

### いちばん大事な点：その `SPEC` テーブルはどこから来るのか

本質は `chain` ではなく `SPEC`（車種→値域の対応表）＝ validity-spec の実体。問いは「どう書くか」から「**そのルールの出どころをどう確保するか**」に移る。

| 業務ルールの保持形態 | 生成器の作りやすさ |
|---|---|
| **データ駆動**（車種↔値域がマスタテーブルにある） | 容易。生成器は同じマスタを読むだけ |
| **手続き的に散在**（API 内の if-else に埋没） | 困難。散らばったルールを人間が再抽出 |

`SPEC` をテスト対象 API から逆算してはいけない（実装のバグを相続して自分を盲目にする）。理想は API がバリデーションに使うのと同じ"単一の真実の源"を生成器も読むこと。**これは記事の「土台と計算の分離／二つの真実を作らない」が、なぜテスタビリティにも効くかの具体例。「PBT ができる設計か」は「業務ルールが単一の源にデータとして在るか」とほぼ同義。**

### きれいな階層に収まらない制約には道具を段階的に落とす

1. **`chain`（第一選択）**：階層的依存。棄却ゼロ。
2. **`map`（修復/導出）**：生成してから整合するよう補正。
3. **`filter`（最終手段・葉でだけ）**：棄却率が十分低い末端でだけ。上流で filter すると破綻。

補足：`chain` を深くするほど shrinking は弱くなる。依存は必要な分だけ、独立にできる項目は独立に。

## Q10. AI 以前だと SPEC を車種マスタやそれに紐づく全マスタテーブルから作り込むのが現実的でないレベルの負荷だったが、AI だとそのへん得意そう。

これは**記事の中心的な賭け（脚注 [^pbt-ai]）をそのまま言い当てている**——PBT が20年以上前からあるのに普及しなかった障壁の一つが「生成器設計／普遍条件の抽出の難しさ」で、AI がその障壁を下げられるのではないか。ただし「信頼してよい ≠ 検証を捨ててよい」が効く。

### AI が本当に強い部分：機械的な抽出・転記・足場

スキーマと FK 関係を読んで対応表を列挙する、`chain` ベースの生成器コードに落とす、数十のマスタを手で突き合わせる現実的でなかった負荷を圧縮する——ここは AI の独壇場。**PBT を「理論的にはいいが高すぎて無理」から「払える」に動かす。**

### 得意さが危険に反転する3点

1. **相関誤りの罠が AI で増幅する**：実装コードからSPECを作らせると、実装のバグまで忠実に・網羅的に・自信たっぷりに再現する。AI が優秀なほど、間違った源から吸ったときの被害がもっともらしくて気づきにくい。源は必ず**実装ではなく仕様/マスタ**に向ける。
2. **「検証ルールか、ただの参照データか」の判定は人間に残る**：歴史的残骸・UI ヒント・別所で担保される緩い関係が混じる。AI は候補を提案できるが、どれを不変条件として批准するかは人間（proposed→agreed）。
3. **マスタに無い隅（負の空間）には AI も届かない**：コンバート車両はプロセス上の経路でマスタの一行ではなかった。データモデルに表現されていない隅を生成器に入れられない。だからこそコーパス（現実データ）が強かった。

### 払い：AI が作った生成器を、どう検証するか — 全部がつながる

- **到達 witness**：狙った隅を踏んだか（隅リスト自体は人間/敵対者がレビュー）。
- **正常性の独立チェック**：生成した車両を SUT ではない独立のバリデータ（＝仕様）に通す。

そして伏線回収——**8000件コーパスで AI 製生成器を検証できる**：「生成器の到達空間は8000件を包含しているか？」「8000件の中に生成器が作れないレコードはないか？」（あれば、生成器がモデルし損ねた負の空間の在処を現実が教えている）。**コーパスが生成器の被覆を検証し、生成器がコーパスの天井を超える。** 2手法が互いの検証器になる。

記事自身が「AI で PBT がやりやすくなる、はまだ仮説で実証が要る」と留保している通り。AI は**人間のレビュー面を消すのではなく、マスタ突き合わせという機械作業から、判断という小さく安定した面へ押し上げる。**
