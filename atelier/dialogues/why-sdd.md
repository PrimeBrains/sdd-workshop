# 対話ログ: なぜ仕様駆動開発か

**セッション日**: 2026-05-07
**ペルソナ**: Java歴3年、普段は Spring Boot で API を作っています。Claude Code は触り始めたばかりで、SDD という言葉は聞いたことがある程度です。

---

**Q**: SDDって何が嬉しいんですか？普通に「とりあえず実装してみる」アプローチと何が違うのでしょう？Claude Code を使うなら、AIがコードを書いてくれるので、仕様とか考えなくてもすぐ動くものができるんじゃないかと思っていたのですが。

**A**: その疑問はよくあります。「AIが書いてくれるなら仕様はいらない」という前提が実はズレているんです。

AIはあなたが与えた情報の範囲でしか動きません。指示が曖昧なら、AIは「もっともらしい何か」を返すだけです。100行のコードは一瞬で出てきますが、それが「あなたが本当に作りたいもの」かどうかは別の話です。

仕様駆動開発（SDD）の価値はここにあります。**コードを書く前に「何を作るか」を言語化する**ことで、AIへの指示が明確になり、生成されたコードが意図からズレにくくなります。

「とりあえず実装してみる」アプローチを Spring Boot の開発で想像してください。エンドポイントを作ってみたら、認証フローの考慮が漏れていた、バリデーションが足りなかった、と後から修正が増えた経験はないですか？AIを使うとそのサイクルが速くなるだけで、根本的な問題は解決しません。むしろ「AIが一瞬でそれっぽいコードを出してくれる分、誤った方向に速く進んでしまう」リスクが上がります。

SDD は「設計なしで実装する」ことへのブレーキではなく、「AIと協調して正しいものを速く作る」ための前準備です。

---

**Q**: でも仕様書を書くって時間がかかりますよね？アジャイルで「動くソフトウェアを重視する」という考え方があるのに、重厚な仕様書を書くのはウォーターフォール的な感じがして、相性が悪くないですか？

**A**: その誤解はよくあります。「仕様書 = ウォーターフォールの重厚なドキュメント」というイメージが先行しているんですね。

SDD の「仕様」は Word で50ページ書く要件定義書ではありません。**AIに渡すためのコンテキスト**です。具体的には:

- `requirements.md`（何を作るか、EARS形式で箇条書き）
- `design.md`（どう設計するか、主要な判断とその理由）
- `tasks.md`（実装順序の一覧）

これ3ファイルで、多くの場合1〜2時間で書けます。そしてこの3ファイルがあることで、AIへの指示が「このタスク1を実装して」という一言になります。

アジャイルと相性が悪いのではなく、**アジャイルの「小さく動かす」サイクルを、SDD は1スプリントレベルで適用している**とも言えます。スプリントの最初にこの3ファイルを作り、実装フェーズでAIを使う。「仕様書を書く時間」は無駄になるのではなく、「AIとのやり取りで発生するズレの修正時間」を前倒ししているだけです。

試しに計ってみてください。仕様なしでAIに実装させて後からズレを直す時間と、仕様を書いてから実装する合計時間を比べると、後者のほうが速いことがほとんどです。

---

**Q**: 仕様書って書いたとしても、開発が進むうちにコードと乖離していくのが怖いです。「仕様には〜と書いてあるけど実際の実装は違う」という状態が一番困ると思うんですが、SDD ではどう管理するんですか？

**A**: 鋭い指摘です。「仕様とコードが乖離する」問題は、SDD が解決しようとしている中心課題のひとつです。

まず、SDDの仕様ファイル（`requirements.md`, `design.md`, `tasks.md`）はコードと同じリポジトリに置きます。`.kiro/specs/{feature}/` 以下に入れることで、コードと仕様が常に同じコミット履歴を持ちます。「この実装はいつの仕様に基づいているか」がgit logで追えます。

次に、乖離が生まれるタイミングについて。仕様変更が発生したとき、多くのチームは「コードを直してコミット」で終わらせます。SDD では「仕様ファイルも同時に更新する」をルールにします。AIを使う場合も「この変更を実装して」ではなく「この仕様変更を反映して」とファイルを指定して指示するので、自然と仕様とコードが同期するフローになります。

完璧な同期は難しいですが、Spring Boot のプロジェクトで言えば `application.properties` がコードと同じリポジトリにあるのと同じ感覚です。乖離したら「仕様も直す」のがコーディング規約と同じレベルのルールになる、それが SDD の運用です。

## Insights（抽出メモ）

**誤解パターン:**
- 「AIがコードを書いてくれるなら仕様は不要」→ AIは与えられた情報の範囲でしか動けないため、指示の質が成果の質を決める
- 「仕様書 = ウォーターフォールの重厚ドキュメント」→ SDD の仕様は3ファイル・1〜2時間で書けるAI向けコンテキスト
- 「仕様はコードと乖離していく」→ コードと同じリポジトリに置き、変更時に同時更新するルールで管理できる

**理解の転換点:**
- 「AIが速くコードを出してくれる分、誤った方向に速く進んでしまうリスクが上がる」→ 速さの恩恵をリスクと一緒に語ることで必要性が腑に落ちるポイント
- 「仕様書を書く時間は、AIとのズレを修正する時間を前倒しにしているだけ」→ トータルコストの視点で語ると納得しやすい
- 「仕様変更も『ファイルを指定して指示する』フローに乗せる」→ 乖離防止が運用上自然に組み込まれていると分かる瞬間

**使えそうな比喩・表現:**
- 「AIはもっともらしい何かを返す機械」→ 指示の質が成果を決めることを端的に伝える
- 「アジャイルの小さく動かすサイクルを、1スプリントレベルで適用する」→ アジャイル経験者への橋渡しとして有効
- 「`application.properties` をリポジトリに置くのと同じ感覚」→ Spring Boot 開発者にとって仕様ファイルの位置付けが直感的に分かる
