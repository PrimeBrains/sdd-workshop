# Moira 設計判断目録（Decisions Catalog）

> 人間（プロダクトオーナー＝非専門家）がレビューする一次成果物の3面のうち、**構造的健全性**の面。
> MODEL にもプロパティにもシナリオにも bind できない **0→1 の設計判断**を、非専門家が賛否を持てる形に翻訳して並べる。
> 統治方針は [`.kiro/steering/moira-verification.md`](../.kiro/steering/moira-verification.md)。
>
> **`moira/DECISIONS.md`（意思決定ジャーナル＝来歴・専門家向け）とは別物。** こちらは人間レビューの入口、
> 向こうは確定分岐の archive。ここで `agreed` になった判断は、必要に応じてジャーナルへ来歴記録される。

## 読み方・規律

- **平易な言語規約**: 表面（〜判定する一文）は専門用語・skill 実名を出さない。clause ID・実装名は〔裏面〕にのみ（**安定 A/I/P/R ID で参照・行番号禁止**）。
- **批准**: 起票者（AI）が `proposed` で起こし、人間が `agreed` で批准。批准前のハードニングは `doc-refine`（汎用確定文書ゲート）。
- **仕分け**: **(a)** 責任・帰結・ドメイン意味＝プロダクトオーナーがレビュー可／**(b)** 純工学＝技術敵対ゲート（`doc-refine`）へ分流（表面は観測できる帰結のみ提示）。またがるは (a) 既定。
- **状態**: `proposed`（AI 起票・人間未批准）／`agreed`（人間批准済）。`proposed（既確定の目録化）`＝判断自体は MODEL/spec/journal で確定済みだが、人間レビュー面としては未批准。`proposed（未確定）`＝判断そのものがまだ開いている。
- **計器（テストの落とし先・1判断が複数可）**: **①アーキ適合**（境界・所有権・依存方向＝fitness関数）／**②型・スキーマ**（構造の形）／**③PBT**（帰結が不変条件）／**④E2E/シナリオ**（観測ふるまい）／**⑤境界モデル検査**（順序・状態機械）／**⑥AI整合性チェック**（自動化不能だが実装と照合可・負の照合含む）／**⑦人間のみ**（テスト対象が定義上存在しない）。割付の全体像は末尾の**被覆マップ**、規律は `moira-verification.md`。
- **被覆レポートは持たない**（clause→項目の網羅は不能）が、**Decision→計器 の割付は被覆マップで管理**する（未割付は可視ギャップ）。

# 批准済み（実験レビュー）

## D-1　完了した作業の出来高を、後からの「見積もり直し」から誰が守るか

- **状態**: `agreed`（2026-06-26・人間レビューで批准）　**仕分け**: (a)　**計器**: ③PBT
- **どんな場面の話か**: ある作業を「完了」と記録した後で、誰かがその作業の見積もりを直す（やり直す）ことがある。見積もりを直すと、その作業はいったん「人間が合意する前の見積もり」状態に戻る。一方、達成した出来高は「合意済みの見積もり」だけを数える。すると——**完了した作業の見積もりを直した瞬間に、その作業の出来高が集計から静かに消える**ことが起こりうる。
- **決めたこと（誰の責任か）**: 「いったん完了して確定した出来高は、後から見積もりを直しても減らない」。これを守る責任は **土台の仕組み**が持ち、**完了済みの作業については見積もりのやり直し自体を受け付けない**。
- **他にありえた選び方**: やり直しは許し別の見張り役が後から警告／出来高を数える側が完了済みなら合意前でも数え続ける。
- **外すとどうなるか**: 進捗ダッシュボードの達成出来高が、誰も操作した覚えがないのに突然下がる。原因に気づくのは非常に難しい。
- **あなたが判定する一文**: 「完了して確定した出来高は、後から何をしても減らない。それを守るのは“土台が見積もり直しを断る”が正しい。」
- **未了のフォローアップ**: 決定は批准済みだが**現在の参照実装にガードが無い**。実装是正は別タスク。
- 〔**裏面**〕R-E3／I4。enforce は core（`moira-core` req 7.4）。実装: `fold.ts`（完了ガード無し）・`ev.ts`（agreed 限定）。計器資産: `backend/src/pbt/done-lock.pbt.test.ts`（PR-DONE-LOCK★・現在 RED tripwire）。

## D-2　「もう着手してよい」を、土台が判断するか・別係に渡すか

- **状態**: `agreed`（既確定の目録化。原裁定 2026-06-22）　**仕分け**: (a)　**計器**: ①アーキ適合／②型
- **どんな場面の話か**: 作業の「もう着手してよい」状態には、まぎれやすい二つの意味がある。(1) 担当者が「着手した／していい」と宣言した、という**記録上の状態**。(2) 前提の先行作業が十分終わったから「着手してよいはず」という、**計算で出せる資格**。
- **決めたこと（誰の責任か）**: この二つを別物として分ける。(1) は**土台**が持つ。(2) の計算は土台では行わず**別係**に委ねる。土台は「つながりとしきい値」を保管するだけで評価しない。
- **他にありえた選び方**: 「着手してよい」を一つの概念にまとめ、土台が充足計算まで行う。
- **外すとどうなるか**: 「着手可」が記録なのか計算結果なのか曖昧化し、「着手できる一覧」が現実とずれても気づけない。
- **あなたが判定する一文**: 「“着手したという記録”と“着手してよいはずという資格”は別物として扱うべきだ。」
- 〔**裏面**〕`ready` 二語分離。Boundary Context（2026-06-22 裁定）／R-D1/D2/D4／§2.5。状態保持=core、資格評価=`moira-scope-deps`、`ready` 遷移 emit=`moira-progress`。計器: ①＝core が ready-eligible を評価しない（依存方向）／②＝`ready` は lifecycle 状態として保持。
  - **計器資産（2026-06-27）**: ①＝`backend/.dependency-cruiser.cjs` の `fold-no-downstream`（monolith 段階の弱い代理・CI 稼働）／②＝`backend/src/decisions/ready-lifecycle.test-d.ts`（`ready`∈LifecycleState の正＋`readyEligible` 系算出フィールド不在の負・CI 型ゲートで実走）。

## D-3　すべての記録をどう保存するか（過去を書き換えない積み上げ式）

- **状態**: `agreed`（既に正典。(b) ゆえ批准は技術ゲート相当）　**仕分け**: (b)　**計器**: ②型／③PBT
- **どんな場面の話か**: このシステムの全部の出来事と入力を、どういう形で保存するか。
- **決めたこと（誰の責任か）**: 過去の記録は書き換えず・消さず、変更も「新しい記録の追加」として積み上げる（追記専用）。同じ記録から何度でも同じ結果を再計算できることを保証する。
- **他にありえた選び方**: 最新状態だけを保存して上書きする一般的な方式。
- **外すとどうなるか（人間に見える帰結）**: 「なぜこの数字になったか」を過去にさかのぼって再現・監査できなくなる。
- **あなたが判定する一文（帰結のみ）**: 「過去を書き換えずに保存し、いつでも同じ結果を再現・監査できるべきだ。」
- 〔**裏面**〕A2／R-U14。moira-core req 1・14。計器: ②＝削除 API 不在・イベント4種固定／③＝`PM-ORDER-INV`（順序不変・再導出決定性）。
  - **計器資産（2026-06-27）**: ②＝`backend/src/decisions/events.test-d.ts`（`Event['kind']` ＝4種固定の正＋`'delete'` 等の削除イベント種別を強制しない負・CI 型ゲートで実走）／③＝`backend/src/pbt/green-properties.pbt.test.ts` の PM-ORDER-INV（既存・緑）。

## グループA：CQRS 境界・所有権分配（どの部品が何を持つか）

### D-4　計算の土台と計算そのものを割る（構造＝土台／値・式＝下流）
- **状態** `proposed（既確定の目録化）`　**仕分け** (a)　**計器** ①アーキ適合
- 場面: 「現行有効な作業集合」「達成率%」などの計算を、一箇所に全部置くか、土台と計算を分けるか。
- 決めたこと: 土台は「構造・不変条件・機構」だけを持ち、「評価・値・式」はすべて下流が持つ、を最上位原則とする。
- 他の選び方: 土台が値や ready 判定まで持つ「太い土台」／各画面が土台を自前で作り直す「土台分散」。
- 外すと: 同じ概念が複数箇所で別々に計算され「二つの真実」が出る／土台変更の影響範囲が無限定に広がる。
- 〔裏面〕moira-core Boundary Context「構造境界 core」（doc-refine 2026-06-22 裁定）／core 9.2・9.3・10.1・12.3。

### D-5　異常の「検出」は読み手、「解消」は書き手（自動修復しない）
- **状態** `proposed（既確定の目録化）`　**仕分け** (a)　**計器** ①アーキ適合
- 場面: 「この後続は先行キャンセルで永久に進めない」と気づく仕組みと、それを直す操作を、同じ部品が持つか。
- 決めたこと: 検出（読み）は導出側、解消（辺を消す・繋ぎ直す・後続もキャンセル等の書き込み）は書き手側、と分ける。
- 他の選び方: 検出した側がそのまま自動修復まで行う。
- 外すと: 読み画面が書き込み責務を侵し追記専用の単一書き込み口が崩れる／自動修復が人間の判断を奪う。
- 〔裏面〕moira-scope-deps（R-C3 自動キャンセルしない）・moira-health／roadmap「検出=読/解消=書」。

### D-6　「いま有効な作業の集合」の定義は土台が一手に持つ

### D-10　キャンセルの責務を4つに割る

### D-11　永続化方式を決める責任は土台が持つ

### D-14　土台の契約の「形」の安定性を最優先する

### D-16　容量は「人ごとの日次容量」一本にまとめる（契約率×暦に分けない）
