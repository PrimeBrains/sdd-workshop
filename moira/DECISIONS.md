## 確定済みの分岐(原セッション v1–v9)

- **重みテーブル → 単一式に溶解。** EV はどの階層でも「完了 ÷ 合意済み」。状態に重みを割る発想を廃止。
- **spec 留保率 → ノード化で溶解。** 仕様作業は独自見積を持つただのノード。
- **能力トレース → 廃止。** スキル・習熟度をモデル化しない。割当は人間が外から与える入力(R-U6)。
- **見積 = 合意状態を持つ提案。** `proposed→agreed`、合意は人間のみ(A5/R-U4)。粒度はタスク/feature ノード単位(cc-sdd ゲートと一致、ボトルネック回避)。
- **EV は合意済み見積のみで算出**、未見積は見積カバレッジで可視化(P2/R-S4)。
- **辺ポリシー既定 = implemented**、かつ `implemented ≡ 実装完了 ∧ 仕様FIX`(R-D2)。
- **「見積の見積」→ 規則の底打ちでなく人間のコミット判断(P0 第4項)に還元。**
- **スコープ削除の自動化はしない**(R-T4:アラートのみ、判断は人間)。

## 確定済みの分岐(v9 → v10 ハードニング:独立敵対レビュー)

> 著者が敵対者を兼ねた v1–v9 と異なり、独立した敵対者(moira-adversary)＋独立採点者(moira-gate-judge)が
> 攻撃し、自己レビューが見落とした構造的綻びを是正(確定基準は独立採点者の残存 Critical/Important = 0。
> ラウンド数は規律上、自賛として記録しない)。ユーザーが下した分岐(F1/F2/F3 はレビュー作業上の便宜ラベルで
> MODEL 用語ではない):

- **F1 無限後退の停止 → v9 維持(人間判断)。** ただし「AI が外部所与に接地した入力から提案するので空虚でない」を §2.3 に明示。入力後退は構造的に接地、深さは人間判断で有限化(構造保証とは主張しない)。
- **F2 supersede の EV basis → A:累積保持 + R-S5 明確化。** 旧ノードの EV_abs は**累積稼得 basis**に残す(cancelled とは別)。ただし**カバレッジ(P2)・現行進捗の分母=現行有効集合では二重計上しない**。累積稼得(EV_abs、supersede 込み)と現行進捗(現行有効集合の EV%)を basis ごと分離(§2.7/P2)。
- **F3 金額通貨 → 撤去(最小性)。** A6 をアテンション時間の単貨に、R-U10 調整。消費する UC が無いため。
- **PV/ベースライン → ユーザー判断で PV を扱う**(PV なしに EVM 管理は成り立たないとの判断)。ユーザーの当初直観「PV は見積合意のたびに確定」を起点に EVM ベースライン(PMB)を正規化したが、レビューで**予算は合意時に確定するものの「合意時に全部 FIX」は不正確**と判明し撤回:正しくは**予算=合意時・スロット=初回スケジュール載せ時に、既存 `transition` の凍結属性としてログ記録**(純導出でなく記録)。完了で施錠。SPI は部分集合指標と認め R-S6(スケジュール・カバレッジ)新設。

### v10 で是正した主な構造的綻び(自己レビューの見落とし)
- §6「6次元満点」自己採点・§7「すべて完了」・§5「未処理ケースが残らない」(v9 の文言)・§5 が参照していた未定義の UC1–3 → v10 で撤回/正直化。
- EV の次元破綻 → EV_abs(絶対量)/EV%(達成率)の二形分離、SPI/CPI を予算次元に整合、全要件へ伝播。
- I1 が実装ノード誕生で一時的に破れる → 「合意済みの子のみ Σ、未見積はカバレッジ低下」で整合化。
- §2.5「実装完了∧仕様FIXは必然に同時」→「判定であって保証でない」に(P5 と整合)。
- I7 を最小性で降格(R-D7・I2 へ)。暫定割当の上書き機序・R-D4 述語評価・R-S3 誤発火抑止 ほか。

## 確定済みの分岐(v11 → v12:α_i パラメータ化)

> ユーザーが「人間は 0.2MD/日〜1.0MD/日 とまちまち」と指摘。moira-model-update スキルの
> 敵対的レビューループ(adversary ×3 → 再反論 ×1 → gate-judge)を経て v12 に確定。

- **α_i を公理(A4)レベルで MODEL に導入。** v11 の固定容量 1.0MD/日 を α_i ∈ (0, 1.0] にパラメータ化。α_i なしでは P7 が楽観的スケジュール、R-T3 が偽陰性を生む。P0 で運用吸収可能だが、その場合システムの導出指標が意図的に不正確になり P0 の精神に反する。
- **α_i は外的制約入力であり、廃止済みの裁量パラメータ(重み・留保率・スキル)とは質的に異なる。** L40 を修正し区別を明記。廃止パラメータは構造的代替で還元可能だったが α_i の機能を還元する代替は存在しない。
- **α_i はプロジェクト構成入力として A2 のイベントログとは別次元で管理。** R-U14 で変更追跡を要求。4 イベントへの追加は行わない。
- **α_i の定義域は (0, 1.0]、既定値 1.0MD/日（後方互換）。** α_i = 0 は割当解除で表現。上限 1.0 は標準稼働日容量(A6)に基づく。残業は AC で反映、計画容量には織り込まない。
- **複数プロジェクトの Σα_i 整合はスコープ外。** 本モデルは単一プロジェクト視点。

## 確定済みの分岐(v12 → v13: 暦/容量の単一 c 一般化)

> 「暦(祝日/休暇/稼働日)と α_i の関係」の対話から派生。moira-model-update の敵対ループ
> (adversary ×3 + fact-checker → ユーザー裁定 FORK → 再challenge ×2 → gate-judge)を経て v13 に確定。

- **α_i スカラー → 単一の日次容量 c(i,d) ∈ [0,1.0] MD/日 に一般化。** 祝日・休暇・部分稼働日を正直に time-phased PV へ反映するため。c は理由付き append-only 改定の**区間定数**(暦日ごとの任意関数ではない)。
- **【ユーザー裁定 FORK】容量の表現形 → 単一 c(α は導出 view)を選択。** 敵対者が「二因子 α×暦 はどの導出も積しか読まず冗長、かつ非一様な日次配分で a>1 を要求して破綻」と示し、AskUserQuestion で「単一 c／二因子／v12 維持」を提示。ユーザーは**単一 c** を選択。契約レート α_i は「c の reason=契約 成分」の**名前付き導出 view**(SPI/EV%/CPI と同型)へ降格——v12 で公理化した α_i の一級位置づけを撤回。
- **R-S7 を cause-agnostic 化。** 陳腐化トリガーを「割当変更」から「生きた予測の完了が凍結スロットと乖離(原因＝割当/c のいずれでも)」へ一般化。これにより v11→v12 の「α_i にトリガー条件は存在しない」開示を**撤回**(記録された c 変更が凍結ベースラインを無効化する場合を検知。記録と実態の drift は依然非検知＝P0+R-U14)。
- **c は slot 次元/PV(t) 時間配分に効き、予算総額には効かない。** c=0 は定義域内(祝日/休暇)。恒久離脱は割当解除(§2.4)、一時的不在は c=0——境界は P0(モデルは規定しない)。
- **二層データを正直に開示。** A2「唯一のデータ＝4イベントログ」はノードのワークデータを指し、構成入力 c は別層(R-U14 履歴)。v12 で α_i について既に成立していた区別の一般化。
- **残余開示(§7)**: drift 非検知 / 暦未指定日の楽観既定 / 広範 c 変更のスロット陳腐化洪水 / 過去予測再現の valid-time / 恒久 c=0 の二経路。いずれも P0+実装に委ねる。

独立採点者(moira-gate-judge)の残存 Critical/Important = 0。新公理・新イベント・新状態・新要件番号なし(R-S7/R-T3 は意味論拡張)。
