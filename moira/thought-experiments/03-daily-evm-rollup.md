# 思考実験 03:1日の EV と数日の EVM 推移 —— 抽出・集計・翌朝の把握

> 問い:複数の開発者が1日のうちに、仕様駆動のエンハンス・新規開発・見積のための調査活動・運用タスク・バグ修正を並行してこなしている。この日の EV と、ここ数日の EVM 推移は、どのように **抽出** され、**集計** され、**翌朝 PM** に把握される形になるか。
>
> 01(フェーズ移行)・02(supersede)が**構造**を扱ったのに対し、本実験は**時間軸の集計と読み取り**を扱う。到達点:時間集計は新公理・新イベント・新状態を要さず純導出で閉じ、スケールと運用上の留意も意味論の範囲で確認した。01/02 と異なり、本実験は**純粋に自己完結する確認ではなく**、複数日タスクの日次粒度や運用容量の読みなど、P0 の適用が非自明な箇所を含む（P2 と同じ留保:網羅の証明ではない）。以下は MODEL v10 を前提とする。
>
> **用語上の注記:** 本実験では「as-of prefix の fold」「四つの集計軸」等の略記を用いるが、これらは MODEL v10 の正典語彙ではなく、本実験が導入した便宜的な簡記である。正典の根拠は R-S2（導出状態は同一ログから提供）・I3（大域順序で決定的マージ）・P1–P4 の個別原理にある。

## 素朴な答えと、それが崩れた点

最初は「夜間バッチが日次スナップショットを書き出し、翌朝はそれを読む」と考えた。だがこれは **A2(唯一のデータは追記専用ログ)** に反する。スナップショットは第二の真実源となり、ログとの desync を生み、I3 の決定的マージの恩恵を捨てる。さらに「レポーティング期間(締め)」という EVM 古典の概念を持ち込むが、本モデルにそれは要らない——**任意の ts が status date になりうる**からである。これが素朴案の崩れた点。

## 抽出:メトリクスはログから導出される

導出メトリクス（EV_abs / EV% / AC / SPI / CPI 等）は、ログから再現可能な形で導出される(R-S2)。ある時刻 `t` 時点の状態は、`(ts,id) ≤ t` を満たすイベント列の畳み込みで得る(以下「as-of 導出」と呼ぶ)。なお、PMB のベースライン予算・スロットは既存 `transition` の凍結属性としてログに**記録**されている（§3）——これらは導出メトリクスではなく凍結された事実であり、as-of 導出はこの記録を読み取る。

- **日次バッチも保存スナップショットも意味論上は不要。** 「抽出」とは同一ログからの導出（R-S2）であって、ETL でも夜間ジョブでもない。キャッシュや増分計算は実装上の最適化であり意味論を変えない（スケール上の留意は後述）。
- 複数開発者+エージェントのイベントが同一ログに interleave しても、**I3(大域順序・`(ts,id)` 決定的マージ)** が導出を一意にする。誰の端末で何時に起きた作業も、同一ログの上で順序づく。I3 は**再現性**（同一入力から同一出力）を保証するが、分散環境での**因果的忠実性**（id 順が因果順と一致するか）は保証しない。EVM 導出は因果順序に依存しない集約（完了の総和、コストの総和）が主であり、因果的忠実性の不在は実用上問題にならない。
- **完了寄与は I4 で施錠される**ため、過去の日の「稼得した EV_abs」は後から書き換わらない（ただし cancelled による active basis からの除外は例外;後述）。推移は**安定した台帳**であり、再ベースライン(R-U7/§3)が動かすのは未完了の予算・スロット(=PV と将来 EV)だけ。過去の朝に報告した数字はその朝の as-of 導出で再生でき、説明責任に耐える。

## 集計:一つのログ、四つの集計軸

「集計」は単一のログに対する、異なる軸での導出にすぎない。R-S2 が「同一ログから三ダッシュボードへ提供」と述べるのと同型の発想で、ここでは以下の四軸を識別する。

| 軸 | 何を導出するか | MODEL の根拠 | 出てくるもの |
|---|---|---|---|
| **木(価値)** | 完了サブ単位の **I4 施錠された凍結予算** を親へ集約 | P1(EV_abs/EV%)、I4(施錠) | feature・root への進捗ロールアップ。**現行有効集合(R-S5)のみ**で読む |
| **actor(コスト)** | `cost` を行為者へ帰属 | P3(AC 同型集約) | 人ごと/エージェントごとの AC。1.0MD/日(A4)との対比で**容量**の実消費 |
| **DAG(リスク)** | 後退遷移・解放の伝播 | P5(非単調性)、R-D1/R-D2(辺ポリシー) | at-risk な後続、解放待ち |
| **時間(推移)** | 異なる時刻 `t` の as-of 導出を並べる | R-S2(導出状態の即時提供)、I3(決定的順序) | EVM 推移（時系列）。「日」は**レポート窓のパラメータ**であってモデルのイベントではない |

**軸間の依存に注意:** これら四軸は完全に独立ではない。木(価値)軸は、どのノードが現行有効かを知るために supersede 辺(DAG 軸)の情報を要し、R-S3(thrashing 検出)は EV_abs(木)と AC(actor)の両方を要する。時間軸は他の三軸の導出結果の時点比較である。したがって「四つの独立な fold」ではなく、**共通のログから相互に参照しながら導出される四つの関心**と理解する。

重要なのは、**価値は木に、容量(コスト)は actor に帰属する**点。「開発者 X が今日いくら稼いだか」という per-developer の EV スコアボードは作らない——EV は**仕事の属性**であって**人の属性**ではなく、人の能力をモデル化しない **A4** とも一貫する。PM は feature 別の EV と actor 別の AC を得るが、人を EV で順位づけはしない。

**容量の分割:** 開発者 X が1日のうちに feature-A に 0.3MD、feature-B に 0.5MD、feature-C に 0.2MD を費やした場合、各 feature に `cost` イベントが帰属し、actor 軸での集約は合計 1.0MD/日(A4)との対比で過不足が読める。R-T3(過負荷)は、この合計が 1.0MD を超える場合に当該人物と期間を特定するアラートとして機能する。

## 1日の EV と、推移の読み(二系列は逆方向に動きうる)

- **この日の稼得** `ΔEV_abs(D) = EV_abs(末D) − EV_abs(末D-1)`。通常は非負(完了が積む)。**減少しうるのは**(a) **cancelled**(R-C2 で active basis から除外)と、(b) **後退遷移**(P5(a):完了ノードが un-complete されると「完了サブ単位」の集合から外れ EV_abs が減る)の二経路。supersede は EV_abs を減らさない(§2.7:旧ノードの EV_abs は累積稼得 basis に残る)。大規模プロジェクトで feature の取り下げ(cancelled)が完了済みサブツリーに及ぶと、ΔEV_abs が大きく負に振れうる——これは「スコープ削減で実績が消えた」という正直な信号であり、PM は S11(検証シナリオ)のサンク可視化で読む。
- **この日の達成率変化** `ΔEV%(D)` は **自由に非単調**(P5:後退遷移・上方再見積・新規合意見積の算入)。
- ゆえに同じ1日で **ΔEV_abs > 0(働いた)** と **ΔEV% < 0(達成率は下がった)** が両立しうる。例:見積のための調査(spike)が完了して EV_abs を稼ぐ一方、その成果として合意された見積が basis を広げ EV% を希釈する。**この乖離は PM にとって有用な正直な信号**であり、単線の「進捗%」では消えてしまう(P5 を日次粒度で可視化したもの)。
- 推移は単一の線ではなく、本モデルが随所で要求する**ペア読み**の時系列:EV_abs(累積稼得)/ EV%(現行有効集合の達成率;R-S5)/ 見積カバレッジ(霧は晴れたか;P2)/ SPI とスケジュール・カバレッジ(R-S6)/ CPI。いずれかの指標を単独で読むと、R-S4(低カバレッジ時の de-rate)や R-S6(スケジュール・カバレッジと対の読み)が要求する文脈が失われる。
- **再ベースラインは推移に段差として現れ、隠せない。** 再ベースラインは `decompose` イベント(理由必須;§3)として記録されるので、ログを遡れば「いつ・なぜ計画を引き直したか」が PV の段差として再構成できる。as-of-then(その朝の導出)と as-of-now(後の cancellation 込み)の両方が同じログから読める。

## 複数日タスクと日次粒度(WIP の不可視性)

EV_abs は**完了ベース**(P1:完了サブ単位の凍結予算の総和)であり、仕掛中(WIP)の部分進捗は EV に乗らない。3日かかるタスクでは、初日・二日目は ΔEV_abs = 0 だが AC は毎日積む——日次粒度では R-S3(thrashing)と外見上区別がつかない。

これは MODEL の欠陥ではなく、完了ベース EVM の本質的な性質であり、**分解の粒度で緩和**される。tasks.md のチェック項目を1日以内に完了できる粒度まで分解すれば、日次 ΔEV_abs は非ゼロになる頻度が上がる。分解の粒度をどこまで細かくするかは **P0 のコミット判断**（見積の深さ = 人間が決める）であり、MODEL は「完了ベースの EV + 分解の粒度は人間判断」という構造を提供する。PM は日次で ΔEV_abs = 0 が続くタスクを見たとき、(a) thrashing(空回り)か (b) 粒度不足(大タスクの正常な仕掛中)かを、そのタスクの分解状態を見て判断する——この判断自体が P0 の領分である。

## この日の作業種別の混在(どれが EV を生み、どれが容量だけを食うか)

| 作業 | EV への乗り方 | 集計上の振る舞い |
|---|---|---|
| **新規開発(仕様駆動)** | フェーズ/実装ノードが完了するたび EV_abs(§2.6) | tasks 完了で実装ノードが未見積誕生→カバレッジ低下→est(impl) 合意で回復(§2.3)。当日 EV% が動く主因 |
| **エンハンス(supersede)** | root 直下の新 feature で前進的に稼得(§2.7/02) | 旧ノードは accepted のまま。累積稼得には残り、現行有効集合(R-S5)では新ノードを読む |
| **バグ修正** | 修正タスクを feature の子として `decompose` で追加し、完了で EV_abs | `accepted` 済みの feature にも `decompose` で新たな子タスクを追加できる（I4 は完了サブ単位のベースライン寄与を施錠するが、兄弟の追加を禁じない）。修正タスクの誕生でカバレッジが一時低下し、見積合意で回復する。仕様変更を伴うなら supersede(02 参照)。仕様変更を伴わない実装修正はこちら |
| **見積のための調査** | 重ければノード化し完了で EV_abs、軽ければ畳んで cost のみ(R-E2b) | ノード化した spike の完了は当日 EV を稼ぐが、その成果(合意見積)が EV% を下げうる(上記の乖離) |
| **運用タスク** | 多くは合意見積を持たず **EV を生まない**(価値ベースライン外) | AC(容量消費)だけが立つ。次節の P0 適用例 |
| **待機(外部依存ブロック)** | EV も AC も生まない（開発者は手を動かしていない） | P7 の平準化上は容量が空くが、ブロックされたタスクは `ready` に戻らず DAG 上で依存待ちのまま。**容量の空きは「未割当の可用時間」として別の割当に使える**が、使わなければ可視のギャップとして残る（割当カバレッジ;P0/R-U9） |

## P0 の適用例:運用タスクの容量と CPI

運用(障害対応・デプロイ・問い合わせ)は容量を食うが、合意見積を持たず価値ベースライン(PMB)に載らないことが多い。

- **ノード化しなければ**、その容量消費は P7 から見えず、スケジュールは静かに滑る——**ノード化しない運用は P2 の「霧(未発見作業)」そのもの**。ノード化するか否かは P0 のコミット判断(見積深さと同型)。
- **ノード化すれば**(未見積・cost のみでも)既存機構で扱える:容量は AC と R-T3(過負荷)が捉え、価値を生まないことはカバレッジのギャップとして出る。
- **CPI の読み:** 運用に 0.2MD 食われた日、CPI(=EV_abs/AC)は下がる。これは**正直な信号**である——怠けていないとしても、コスト効率(投入容量に対する価値産出)はその数字が事実。PM は CPI の低下を見て「なぜ効率が落ちたか」を AC の内訳(actor 軸)で分析し、運用負荷が原因と判ればスコープ/要員の判断(R-T4:判断は人間)に進む。CPI を「価値部分木にスコープする」読み方も運用上は有用だが、それは PM の分析行為(P0 のコミット判断)であり、MODEL が構造として新要素を要するものではない。
- **R-S3(thrashing):** 運用が常態化し EV_abs が停滞する一方 AC が**継続的に**増える場合、R-S3 はこれを正しくフラグする。これは R-S3 の誤発火ではなく、**価値産出が停滞している事実の正直な検出**である。PM はフラグの原因を分析し(運用過多か実装の空回りか)対策を決める——この判断が P0 の領分。

## スケールと運用上の留意

本実験の意味論は大規模(20人・500+ノード・エージェント 24 時間稼働)でも成立するが、以下の留意がある。

**増分計算の等価性:** as-of 導出の意味論(ログ全体からの再導出)は正確な定義を与えるが、大規模で毎回全ログを走査するのは実装上非効率である。EVM 導出の大半は**増分で維持可能**:EV_abs は完了イベントで加算(I4 施錠で不変)、AC は cost イベントで加算(P3)、これらは O(1)/event。ただし cancelled による active basis 除外(R-C2)や supersede による現行有効集合の変化(R-S5)は、影響範囲のサブツリー走査を要し O(ノード数) になりうる。増分計算が全ログ再導出と意味論的に等価であることの保証は実装の責務であり、MODEL は意味論(正しい答えの定義)のみを固定する。

**日境界はパラメータ:** 「日」はモデルの概念ではなくレポート窓のパラメータであり、as-of 導出の時刻 `t` は呼び出し側が自由に選べる。チームがタイムゾーンをまたぐ場合(東京 UTC+9、サンフランシスコ UTC-7)、東京の PM と SF の PM は異なる `t`(それぞれのローカル朝)で導出し、異なる「昨日」を見る。これは矛盾ではない——ログは一つ(A2)、順序は一意(I3)であり、**どの窓で切るかが異なるだけ**。ΔEV_abs(D) の「D」はローカル日の定義に依存するが、同じ `t` を与えれば同じ結果が再現される(I3 の再現性保証)。

## 翌朝、PM が見るもの

朝は、**夜間にエージェントが自律で進めた作業を人間が初めて読む touchpoint** である。エージェントは平準化の制約を受けず(R-U11)、かつ眠らないため、運用上は夜通し作業を進められる(これは MODEL の規定ではなく、R-U11 が可能にする運用パターン)。A5 により、エージェントはノードを `implemented` まで押し上げられるが(仕様FIX の**判定**;§2.5)、`implemented→accepted`(人間の品質確認)は人間を待つ。ゆえに朝の導出は以下の関心で構成される:

1. **EVM 推移(多系列・ペア読み):** 日境界の as-of 導出を並べた ΔEV_abs / EV% / カバレッジ / SPI+スケジュール・カバレッジ / CPI。夜間エージェントの稼得もここに乗る。現行有効集合(R-S5)の変化も確認し、supersede で陳腐化した旧機能を現行と誤読しない。
2. **レビューキュー(P4):** 夜の間に `implemented` へ達したノード群=朝の人間レビュー待ち。**キューがそのまま当日のアクション・リスト**になる。
3. **警告:** at-risk(P5:implemented 到達後の後退)、過負荷(R-T3、運用に食われた容量を含む)、thrashing(R-S3)、期日超過(R-T4)、低カバレッジによる de-rate(R-S4/R-S6)。

以下は仮の数値による読みの例:PM の典型的な読みはこうなる——「夜間に 4.2MD 分を稼いだ(ΔEV_abs)。だが EV% は 6pt 下がった。spike が合意見積で basis を広げたからだ(健全な発見)。一方で X は前日に障害対応で 0.5MD 食われ CPI が低下、feature-E のレビューが3件待ち。」乖離と霧と容量が、単一のログから一望に揃う。

## 確認できたこと

時間集計(この日の EV・数日の推移)は、**新公理・新イベント・新状態を要さない**。「日」はレポート窓のパラメータ(タイムゾーン依存)であり、推移は as-of 導出の列、抽出は R-S2 に基づく導出、集計は同一ログからの四つの関心(価値/コスト/リスク/時間;P1–P4 に根拠)に還元される。完了施錠(I4)が過去日の台帳を安定させ、再ベースラインは段差として隠れず残る。日次粒度では ΔEV_abs と ΔEV% の乖離が P5 を可視化し、朝のレビューキュー(P4)が夜間自律作業の人間接点になる。スケールでは増分計算の等価性が実装の責務として残るが、意味論はログ全体の再導出として明確に定義される。

運用タスクの容量会計(EV を生まない容量消費が CPI を下げる)は、**P0 の適用で解消する**:CPI の低下は正直な信号であり、PM がその内訳を分析して対策を判断する(R-T4)。新規則は不要。複数日タスクの日次 ΔEV_abs = 0 は、分解粒度の P0 判断で緩和される。

以上は v10 の機構で一貫して動き、**破綻は見つからなかった**(網羅の証明ではない;P2 と同じ留保。検証シナリオ S8(過負荷リスケ)・S10(手戻り)の日次粒度での詳細な走破は本実験のスコープに含めず、将来の検証の余地として残す)。

## レビュー来歴

独立敵対レビュー(moira-model-update スキル)でゲート通過(残存 Critical/Important = 0)。主な是正:
- ΔEV_abs の減少経路を P5(a) 後退遷移も含む二経路に修正(旧「cancelled のみ」は P5 と矛盾)。
- 四つの集計軸の軸間依存を明示(旧版は独立と読める表現)。バグ修正・待機を作業種別に追加。
- 複数日タスクの WIP 不可視性(日次 ΔEV_abs = 0 が thrashing と区別不能)を新節で扱い、分解粒度を P0 緩和策として提示。
- スケール留意を新節で追加(増分計算の等価性、タイムゾーンの日境界パラメータ化)。
- 運用タスクの容量会計を「圧力点」から「P0 の適用例」に格下げ(ユーザー決定:CPI の低下は正直な信号)。
- R-U11 の誤帰属、R-S2 の誤引用(PMB 凍結属性の未言及)、S13 の非 MODEL 引用をそれぞれ修正。
- 本 TE 導入の略記(as-of 導出・四つの集計軸)を MODEL v10 の正典語彙ではないと明記。
