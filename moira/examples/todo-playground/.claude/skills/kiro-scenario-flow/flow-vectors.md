# フロー攻撃角カタログ / Flow Adversarial Vectors

`doc-adversary` に派遣時に読ませる、E2E フロー文書（`.kiro/scenarios/flows/{slug}.md`）専用の攻撃角。
**点数表ではない**——「何を攻撃するか」の観点集。各角は「破れたら何が壊れるか」を伴って当てる。点数は付けない。

**まず `doc-refine` の一般文書 攻撃角 G1–G4（`../doc-refine/adversarial-vectors.md`）を当て**、その上で以下の FC1–FC7 を上乗せする。フローは「確定文書」であると同時に「複数ユニットを順に合成した E2E の通し」でもあるため、合成・連鎖・E2E 価値の角を足す。

> 単体ユニットの攻撃角（`../kiro-scenario/scenario-vectors.md` の SC1–SC7）は**メンバーユニット自身の確定時**に当て済みである前提（FC7 でメンバー全 agreed を確認する）。本カタログはメンバー間の**継ぎ目と通し**を攻める——個々のユニットでは行使できない盲点が攻撃面である。

---

## フロー 攻撃角（対象: `.kiro/scenarios/flows/*.md`。一般文書 攻撃角に上乗せ）

### FC1. 合成順序の妥当性
- `composes:` のユニット並び順が、現実の遷移（discovery → 見積 → 計画 → 要件 → 設計 → タスク → 実装 等）と cc-sdd の承認順（Requirements → Design → Tasks → Implementation）に整合するか。前後が入れ替わっていないか、飛ばしてはならない段を飛ばしていないか。
- 各メンバーの `precondition` が、**直前までのメンバーの postcondition の累積**で実際に成立しているか（順序が違えば成立しない）。
- 破れたら何が壊れるか: 「再利用可能な背骨」という主張が虚偽化し、この順では現実に通らない E2E を正と誤認させる。順序破綻なら Critical。

### FC2. 境界接続（postcondition → precondition の連続）
- 隣接メンバー間で、**前者の `postcondition` が後者の `precondition` を満たす**か。状態（lifecycle・承認・凍結）と数値（EV%・カバレッジ・人日）が継ぎ目で**連続**しているか——飛び・段差・巻き戻しが無いか。
- フロー §4 が描く通しの弧（例 EV% `0→24→32→80→100(見かけ)→44(正直化)→100(本物)`）の各段が、メンバーの postcondition 数値と**一致**するか。途中の数値がメンバーと食い違っていないか。
- 破れたら何が壊れるか: 継ぎ目で状態/数値が不連続なら、フローは「実際には接続しないユニットの寄せ集め」になる。不連続は Critical。**数値主張は `CLAIMS_FOR_FACT_CHECK` として列挙し `doc-fact-checker` に回す。**

### FC3. E2E 価値・必要性（和を超える主張か）
- フロー §3 の通しの主張が、**単独ユニットの postcondition の単純な和では言えない**ことを述べているか（例: 各継ぎ目で正直な信号——カバレッジ低下・見かけ 100% の正直化・差し戻しの CPI——を保ったまま真の完了まで運べる、という**通しでしか観測できない**性質）。
- フロー自体が**冗長**でないか——`composes:` の和を読めば自明な内容を並べ替えただけなら、フローは存在価値が無い（G1 最小性の逆向き：フロー丸ごとを除去しても通しの主張が残るなら、そのフローは冗長）。
- 破れたら何が壊れるか: 「E2E でしか言えない価値」が空なら、フローは単なるユニットの目次に退化し、合成のコストに見合わない。空虚なら Critical。

### FC4. 人間所有性（スコープ承認＋ドラフト批准＋生成価値の非捏造）
- フローでは通しの価値・観点は **AI が生成してよい**（合成で初めて見える創発性質ゆえ）。人間所有は「発案」ではなく次の2点に宿る——その**いずれかが欠けていたら攻撃する**：
  - **スコープが人間に明示承認**されているか（どのユニットを・どの順で・どこからどこまで・何をスコープ外に＝§1 スコープ承認ゲートの記録があるか）。これは発案でなく選択・合成の承認。
  - AI が生成した**通しのフロー固有の価値・観点（§1/§3）を、人間がドラフトレビューで批准（ratify）**したか——ループ開始前に人間が読み返し「これは本当に確かめたい通しだ」と承認した記録があるか（ループ中に §1/§3 を実質改変したなら再批准されているか）。
- （敵対面・最重要）AI 生成の E2E 価値が**実在の創発性質**を述べているか、それとも**捏造・過大主張**か——members の postcondition から実際には観測できない性質を「通しの価値」と偽っていないか。AI 生成ゆえここを最も疑う。
- フロー §6 EARS の通し受け入れ条件が**非専門家でも falsifiable** か——moira 専門用語・skill 実名・MODEL 記号（R-xx/I-x/P2 等）が混入し人間がチェックできなくなっていないか。
- 破れたら何が壊れるか: スコープ承認もドラフト批准も無いまま AI 生成の通しを固めると、「AI が選んで・AI が価値を作り・AI が通す」自己採点に退化し、外的妥当性が失われる。捏造された創発価値は実装者を誤導する。スコープ未承認／ドラフト未批准／捏造・過大主張の価値は Critical。

### FC5. シーム被覆（unit 単体では行使できない継ぎ目を行使するか）
- フローが主張する**メンバー間の継ぎ目**（`⟿` spec/skill 跨ぎ・前段の出力が次段の入力になる接続点）を**実際に行使**しているか。単体ユニットのレビューでは決して見えない盲点——「前段の postcondition を次段が本当に消費する」——を通しで突いているか。
- `⚠`（未実装 skill）を跨ぐ継ぎ目を正直に標識しているか。継ぎ目が**主張だけで実質を伴わない**（飾りの ⟿）になっていないか。
- 破れたら何が壊れるか: 「継ぎ目を行使する E2E」というフロー最大の価値が虚偽化する。継ぎ目が空なら Critical。

### FC6. トレース合併の健全性
- フロー frontmatter の `touches_specs` / `touches_requirements` が、**全メンバーユニットの当該リストの重複排除した全列挙の和**になっているか（`.kiro/steering/trace-notation.md` 準拠・範囲/ワイルドカード/注釈/自由文 禁止・grep 健全）。
- メンバーにあってフローに**漏れた** ID、フローにあってどのメンバーにも**無い** ID（混入）が無いか。各 ID が実在するか。トレース詳細を本文に重複させていないか（frontmatter に集約）。
- 破れたら何が壊れるか: grep ベースのトレーサビリティがフロー↔要件で破綻し、フローが触れる範囲の対応表が信頼できなくなる。漏れ/混入は Important（中核要件の漏れなら Critical）。

### FC7. README/様式 ＋ composes 整合
- 必須セクション（`.kiro/scenarios/README.md` v0.2＋フロー固有の通し節）が揃っているか。§4 に **HTML モックアップ と データ表の両方**（言葉だけ禁止）があるか。§6 が `.kiro/steering/requirements-style.md` の EARS トリガー（WHEN/WHILE …）を使うか。
- **`composes:` の列挙と本文の語りが一致**するか——`composes:` にあるユニットが本文の合成・連鎖で扱われ、本文が語るユニットが `composes:` に列挙されているか（双方向）。
- **`composes:` の全メンバーが `agreed`** か——draft/in-review のメンバーを合成したフローは agreed にできない（未確定の上に通しを建てない）。
- 本文に**未決の FORK が残っていないか**（決定だけを §決定事項 に書く）。`agreed` 版に比較用の二重表現が残っていないか。
- 破れたら何が壊れるか: composes と本文の乖離・未確定メンバーの混入で、フローの「合成の正」が崩れる。未確定メンバー合成なら Critical。

## ESCALATE 規則（節末・必須）

- **フロー文書の欠陥**（合成順序・境界接続の記述・E2E 価値の不足・シーム記述・トレース合併・様式）→ 著者（派遣元の主コンテキスト）が**フロー文書をパッチ**する。
- **メンバーユニット側の欠陥**（あるメンバーの postcondition/precondition 自体が誤り・メンバーが未 agreed）→ フロー文書のパッチで糊塗せず、`ESCALATE: kiro-scenario`（当該メンバーユニットの再ループ）とタグ付けする。フローはメンバーが正しく agreed になるまで凍結しない。
- **MODEL 正典側の欠陥**（継ぎ目の規定が MODEL 自身に無い・MODEL が曖昧/自己矛盾/未規定）→ `ESCALATE: moira-model-update` とタグ付けし、派遣元は**確認のうえ `moira-model-update` へ委譲**する。**フロー単位スキルは `moira/MODEL.md` を編集しない。**
