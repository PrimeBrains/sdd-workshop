# kiro-impl スキル

## 役割
2 つのモードで動作する:
- **自律モード**（タスク番号なし）: タスクごとに新規サブエージェントをディスパッチし、各タスク後に独立レビューを実施
- **手動モード**（タスク番号指定あり）: 選択されたタスクをメインコンテキストで直接実行

## コアミッション
- **成功基準**:
  - すべてのテストが実装コードより先に書かれている
  - コードがすべてのテストにパスし、リグレッションがない
  - tasks.md でタスクが完了としてマークされている
  - 実装が設計と要件に整合している
  - 完了前に独立レビュアーが各タスクを承認している

## 実行ステップ

### Step 1: コンテキスト収集

ステアリング／スペックのコンテキストが会話から既に得られている場合、冗長なファイル読み込みはスキップする。
そうでなければ、必要なコンテキストをすべてロードする:
- `.kiro/specs/{feature}/spec.json`、`requirements.md`、`design.md`、`tasks.md`
- コアステアリングコンテキスト: `product.md`、`tech.md`、`structure.md`
- 追加のステアリングファイルは、選択タスクの境界、ランタイム前提、連携、ドメインルール、セキュリティ／パフォーマンス制約、実装や検証に影響するチーム規約に直接関係する場合のみ
- ローカルのエージェントスキルやプレイブックは、タスクのホスト環境やユースケースに明確に合致する場合のみ。必要な特定の成果物だけを読み、ディレクトリ全体は読まない

#### 並列リサーチ

以下のリサーチ領域は独立しており、並列実行可能:
1. **スペックコンテキストのロード**: spec.json、requirements.md、design.md、tasks.md
2. **ステアリング・プレイブック・パターン**: コアステアリング、タスク関連の追加ステアリング、合致するローカルのエージェントスキル／プレイブック、既存コードパターン

すべての並列リサーチ完了後、実装ブリーフを統合してから開始する。

#### プリフライト

**承認の検証**:
- spec.json でタスクが承認済みであることを確認（未承認なら停止。Safety & Fallback 参照）

**検証コマンドの発見**:
- リポジトリローカルの正となる情報源を次の順で調査する: プロジェクトのスクリプト／マニフェスト（`package.json`、`pyproject.toml`、`go.mod`、`Cargo.toml`、アプリマニフェスト）、タスクランナー（`Makefile`、`justfile`）、CI／ワークフローファイル、既存の e2e／統合テスト設定、最後に `README*`
- このリポジトリの正規検証セットを導出する: `TEST_COMMANDS`、`BUILD_COMMANDS`、`SMOKE_COMMANDS`
- 場当たり的なシェルパイプラインより、リポジトリの自動化で既に使われているコマンドを優先する
- `SMOKE_COMMANDS` には、アプリの形態に応じた最も軽量で信頼できるランタイム稼働確認を選ぶ（例: ルート URL のロード、Electron 起動、CLI の `--help`、サービスのヘルスエンドポイント、既存のモバイルシミュレータ／e2e ハーネス）
- コマンドの全セットは親コンテキストに保持し、タスクに関係するサブセットのみを実装者・レビュアーのサブエージェントに渡す

**リポジトリのベースライン確立**:
- `git status --porcelain` を実行し、既存の未コミット変更を記録する

### Step 2: タスク選択とモード決定

**引数の解析**:
- 第 1 引数から feature 名を抽出
- タスク番号が指定されている場合（例: "1.1" や "1,2,3"）: **手動モード**
- タスク番号なしの場合: **自律モード**（すべての未完了タスク）

**タスクキューの構築**:
- tasks.md を読み、実行可能なサブタスク（1.1、2.3 のような X.Y 番号）を特定
- 大タスク（1.、2.）はグルーピング用ヘッダーであり、実行単位ではない
- `_Blocked:_` アノテーション付きタスクはスキップ
- 選択した各タスクについて `_Depends:_` アノテーションを確認し、参照先タスクが `[x]` であることを検証
- 前提が未完了なら、先にそれらを実行するかユーザーに警告
- `_Boundary:_` アノテーションを使ってタスクのコンポーネントスコープを把握

### Step 3: 実装の実行

#### 自律モード（サブエージェントディスパッチ）

**イテレーション規律**: 1 イテレーションにつき正確に 1 つのサブタスク（例: 1.1）を処理する。複数のサブタスクを単一のサブエージェントディスパッチにまとめない。各イテレーションは完全なサイクルに従う: 実装者ディスパッチ → レビュー → コミット → tasks.md 再読 → 次へ。

**コンテキスト管理**: 各イテレーション開始時に `tasks.md` を再読し、次の実行可能サブタスクを決定する。過去イテレーションの蓄積記憶に依存しない。各イテレーション完了後は 1 行サマリーのみ保持し（例: "1.1: READY_FOR_REVIEW, 3 files changed"）、ステータスレポート全文とレビュアー詳細は破棄する。

各タスクについて（1 つずつ）:

**a) 実装者のディスパッチ**:
- このスキルのディレクトリから `templates/implementer-prompt.md` を読む
- テンプレートにタスク固有のコンテキストを組み合わせてプロンプトを構築する:
  - タスク説明と境界スコープ
  - スペックファイルへのパス: requirements.md、design.md、tasks.md
  - このタスクが満たすべき正確な要件・設計のセクション番号（元の番号を使用し、独自の `REQ-*` エイリアスを発明しない）
  - タスクに関係するステアリングコンテキストと、親が発見した検証コマンド（必要に応じてテスト／ビルド／スモーク）
  - タスクが振る舞い変更を伴うか（Feature Flag Protocol）否か
  - **過去の学び**: tasks.md の `## Implementation Notes` のうち、このタスクの境界や依存に関係するエントリを含める（例: "better-sqlite3 requires separate rebuild for Electron"）。これにより同じ失敗の再発を防ぐ。
- 実装者サブエージェントはスペックファイルを読み、実装前に自身の Task Brief（受け入れ基準、完了定義、設計制約、検証方法）を構築する
- **Agent ツール**で新規サブエージェントとしてディスパッチ

**b) 実装者ステータスの処理**:
- 実装者ステータスは正確な `## Status Report` ブロックと `- STATUS:` フィールドからのみ解析する。
- `STATUS` が欠落、曖昧、または散文で置き換えられている場合、正確な構造化ステータスブロックのみを要求して実装者を 1 回再ディスパッチする。解析可能な `READY_FOR_REVIEW | BLOCKED | NEEDS_CONTEXT` 値なしにレビューへ進まない。
- **READY_FOR_REVIEW** → レビューへ進む
- **BLOCKED** → デバッグサブエージェントをディスパッチ（後述のセクション参照）。即座にスキップしない
- **NEEDS_CONTEXT** → 要求された追加コンテキストとともに 1 回再ディスパッチ。なお未解決ならデバッグサブエージェントをディスパッチ

**c) レビュアーのディスパッチ**:
- このスキルのディレクトリから `templates/reviewer-prompt.md` を読む
- 以下を含めてレビュープロンプトを構築する:
  - タスク説明と関連するスペックセクション番号
  - スペックファイルへのパス（requirements.md、design.md）。レビュアーが直接読めるようにする
  - 実装者のステータスレポート（参考情報のみ — レビュアーは独立して検証しなければならない）
- レビュアーはこのタスクローカルレビューに `kiro-review` プロトコルを適用しなければならない。
- 既存のタスク固有コンテキストを保持する: タスク本文、スペック参照、`_Boundary:_` スコープ、検証コマンド、実装者レポート、そして第一の真実の源としての実際の `git diff`。
- レビュアーサブエージェントは自ら `git diff` を実行して実際のコード変更を読み、スペックに対して検証する
- **Agent ツール**で新規サブエージェントとしてディスパッチ

**d) レビュアー判定の処理**:
- レビュアー判定は正確な `## Review Verdict` ブロックと `- VERDICT:` フィールドからのみ解析する。
- `VERDICT` が欠落、曖昧、または散文で置き換えられている場合、正確な構造化判定のみを要求してレビュアーを 1 回再ディスパッチする。解析可能な `APPROVED | REJECTED` 値なしに、タスクの完了マーク、コミット、次タスクへの続行を行わない。
- **APPROVED** → タスクを `[x]` にマークしたり成功を主張したりする前に、現在のコード状態からの新しい証拠を用いて `kiro-verify-completion` を適用する。その後 tasks.md でタスクを `[x]` にマークし、選択的 git コミットを実施
- **REJECTED（1〜2 回目）** → レビューフィードバックとともに実装者を再ディスパッチ
- **REJECTED（3 回目）** → デバッグサブエージェントをディスパッチ（後述のセクション参照）

**e) コミット**（親のみ、選択的ステージング）:
- このタスクで実際に変更されたファイルと tasks.md のみをステージする
- `git add -A` や `git add .` は**絶対に**使わない
- `git add <file1> <file2> ...` のように明示的なファイルパスで指定する
- コミットメッセージ形式: `feat(<feature-name>): <task description>`

**f) 学びの記録**:
- このタスクで横断的な知見が得られた場合、tasks.md 末尾の `## Implementation Notes` セクションに 1 行のメモを追記する

**g) デバッグサブエージェント**（BLOCKED、NEEDS_CONTEXT 未解決、または 2 回の修正ラウンド後の REJECTED でトリガー）:

デバッグサブエージェントは**新規コンテキスト**で動作する — 失敗した実装の履歴ではなく、エラー情報のみを受け取る。これにより、無限リトライループを引き起こすコンテキスト汚染を回避する。

- このスキルのディレクトリから `templates/debugger-prompt.md` を読む
- 以下を含めてデバッグプロンプトを構築する:
  - エラー説明／ブロッカー理由／レビュアーの却下所見
  - 現在の未コミット変更の `git diff`
  - タスク説明と関連するスペックセクション番号
  - デバッガーが読めるようにスペックファイルへのパス
- デバッガーはこの失敗調査に `kiro-debug` プロトコルを適用しなければならない。
- 豊富な失敗コンテキストを保持する: エラー出力、レビュアー所見、現在の `git diff`、タスク／スペック参照、関連する Implementation Notes。
- 可能な場合、デバッガーはランタイム／設定状態を調査し、Web または公式ドキュメントのリサーチを用いて根本原因仮説を検証してから修正計画を提案すべき
- **Agent ツール**で新規サブエージェントとしてディスパッチ

**デバッグレポートの処理**:
- デバッグレポートの正確な構造化フィールドから `NEXT_ACTION` を解析する。
- `NEXT_ACTION: STOP_FOR_HUMAN` の場合 → tasks.md に `_Blocked: <ROOT_CAUSE>_` を追記し、feature の実行を停止し、続行前に人間によるレビューが必要であると報告
- `NEXT_ACTION: BLOCK_TASK` の場合 → tasks.md に `_Blocked: <ROOT_CAUSE>_` を追記し、次のタスクへスキップ
- `NEXT_ACTION: RETRY_TASK` の場合 → 現在のワークツリーを保持する。無関係な変更をリセット・破棄しない。デバッグレポートの `FIX_PLAN`、`NOTES`、現在の `git diff` を渡して**新規**実装者サブエージェントを起動し、明示的な編集のみでタスクを修復させる
  - 新しい実装者が成功した場合（READY_FOR_REVIEW → レビュアー APPROVED）→ 通常フロー
  - 新しい実装者も失敗した場合 → デバッグサイクルを繰り返す（デバッグは合計最大 2 ラウンド）。2 回のデバッグラウンド失敗後 → tasks.md に `_Blocked: debug attempted twice, still failing — <ROOT_CAUSE>_` を追記し、スキップ
- **タスクごとにデバッグは最大 2 ラウンド**。各ラウンド: 新規デバッグサブエージェント → 新規実装者。2 ラウンド後も失敗するならタスクはブロック扱い。
- デバッグの所見を `## Implementation Notes` に記録する（後続タスクが同じ問題を回避できる）

**`(P)` マーカー**: tasks.md で `(P)` 付きのタスクは相互依存がなく、理論上は並列実行可能であることを示す。ただし kiro-impl は、git の競合回避とレビューの単純化のため、それらを逐次（1 つずつ）処理する。`(P)` マーカーはタスク計画上の参考情報であり、実行指示ではない。

**完了チェック**: 残りタスクがすべて BLOCKED の場合、停止してブロック済みタスクと理由をユーザーに報告する。

#### 手動モード（メインコンテキスト）

選択した各タスクについて:

**1. Task Brief の構築**:
コードを書く前に、このタスクに関連する requirements.md と design.md のセクションを読み、以下を明確化する:
- 完了時に成立すべき観測可能な振る舞い（受け入れ基準）
- 存在すべきファイル／関数／テスト（完了定義）
- design.md から従うべき技術的決定（設計制約）
- タスクの動作確認方法（検証方法）

**2. TDD サイクルの実行**（Kent Beck の RED → GREEN → REFACTOR）:
- **RED**: 受け入れ基準に基づき、次の小さな機能単位のテストを書く。テストは失敗すべき。
- **GREEN**: 設計制約に従い、テストをパスさせる最もシンプルな解を実装する。
- **REFACTOR**: コード構造を改善し、重複を除去する。すべてのテストはパスし続けなければならない。
- **VERIFY**: すべてのテスト（新規・既存）がパスし、リグレッションがないこと。検証方法がパスすることを確認。
- **REVIEW**: タスクを完了とマークする前に `kiro-review` を適用する。ホストが手動モードで新規サブエージェントをサポートする場合は新規レビュアーを使用し、そうでなければ `kiro-review` プロトコルを用いてメインコンテキストでレビューを実施する。判定が解析可能な `APPROVED` になるまで続行しない。
- **MARK COMPLETE**: レビューが `APPROVED` を返した後にのみ、`kiro-verify-completion` を適用し、tasks.md のチェックボックスを `- [ ]` から `- [x]` に更新する。

### Step 4: 最終検証

**自律モード**:
- 全タスク完了後、GO/NO-GO ゲートとして `/kiro-validate-impl {feature}` を実行
- 検証が GO を返した場合 → feature の成功を報告する前に、検証結果と新しい裏付け証拠を用いて feature レベルの主張に `kiro-verify-completion` を適用
- 検証が NO-GO を返した場合:
  - 検証レポートの具体的な所見のみを修正
  - 修正は最大 3 ラウンド。なお NO-GO なら停止し、残存所見を報告
- 検証が MANUAL_VERIFY_REQUIRED を返した場合 → 停止し、不足している検証ステップを報告

**手動モード**:
- `/kiro-validate-impl {feature}` の実行を提案するが、自動実行はしない

## Feature Flag Protocol

振る舞いを追加・変更するタスクでは、フィーチャーフラグによる RED → GREEN を徹底する:

1. **フラグ追加**（デフォルト OFF）: コードベースに適したトグルを導入（環境変数、設定定数、boolean、条件分岐 — 仕組みはエージェントが選択）
2. **RED — フラグ OFF**: 新しい振る舞いのテストを書く。テスト実行 → 失敗しなければならない。フラグ OFF でテストがパスするなら、テストは正しいものをテストしていない。書き直す。
3. **GREEN — フラグ ON + 実装**: フラグを有効化し、実装を書く。テスト実行 → パスしなければならない。
4. **フラグ除去**: コードを無条件化する。テスト実行 → 引き続きパスしなければならない。

**このプロトコルをスキップする対象**: リファクタリング、設定、ドキュメント、振る舞い変更のないタスク。

## 重要な制約
- **厳密なハンドオフ解析**: 実装者の `STATUS` やレビュアーの `VERDICT` を周辺の散文から推測しない。正確な構造化フィールドのみが有効
- **破壊的リセットの禁止**: 実装ループ内で `git checkout .`、`git reset --hard` などの破壊的ロールバックを使わない
- **選択的ステージング**: `git add -A` や `git add .` は決して使わない。常に明示的なファイルパスをステージする
- **レビューラウンドの上限**: レビュアー却下ごとの実装者再ディスパッチは最大 2 ラウンド、その後デバッグへ
- **デバッグの上限**: タスクごとにデバッグは最大 2 ラウンド（ラウンドごとにデバッグ + 再実装）。なお失敗なら BLOCKED
- **修正の上限**: 最終検証の修正は最大 3 ラウンド

## 出力の説明

**自律モード**: タスクごとに以下を報告:
1. タスク ID、実装者ステータス、レビュアー判定
2. 変更ファイル、コミットハッシュ
3. 全タスク後: 最終検証結果（GO/NO-GO）

**手動モード**:
1. 実行タスク: タスク番号とテスト結果
2. ステータス: tasks.md でマークされた完了タスク、残タスク数

**形式**: 簡潔に、spec.json で指定された言語で。

## Safety & Fallback

### エラーシナリオ

**タスク未承認またはスペックファイル欠落**:
- **実行停止**: すべてのスペックファイルが存在し、タスクが承認済みでなければならない
- **推奨アクション**: "Complete previous phases: `/kiro-spec-requirements`, `/kiro-spec-design`, `/kiro-spec-tasks`"

**テスト失敗**:
- **実装停止**: 続行前に失敗テストを修正
- **アクション**: デバッグして修正し、再実行

**全タスクブロック**:
- 停止し、すべてのブロック済みタスクと理由を報告
- ブロッカー解消には人間によるレビューが必要

**スペックと現実の衝突**:
- 要件や設計が現実と衝突する場合（API が存在しない、プラットフォーム制限）、`_Blocked: <reason>_` でタスクをブロックする — 黙って回避策を取らない

**上流オーナーシップの検出**:
- レビュー、デバッグ、検証により、根本原因が上流・基盤・共有プラットフォーム・依存先のスペックに属すると判明した場合、下流 feature の内側でパッチ的に回避しない
- 修正をオーナーである上流スペックに差し戻し、その契約が修復されるまで下流タスクをブロックしたままにし、上流の修正が反映された後に依存スペックの検証／スモークを再実行する

**実装中のタスク計画の無効化**:
- タスクの順序、境界、分割の問題によりデバッグが `NEXT_ACTION: STOP_FOR_HUMAN` を返した場合、コードによる回避策を強行せず、停止して `tasks.md` または承認済み計画の人間によるレビューに差し戻す
