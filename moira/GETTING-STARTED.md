# Moira を自分のプロジェクトに導入する

このドキュメントは、**Moira をまだ知らない人**が自分のプロジェクトに Moira を導入するためのガイドです。
「何ができるのか」「どうインストールするのか」「どんなユースケースに対応しているのか」
「どういう仕組みで進捗が管理されるのか」を順に説明します。

> Moira そのものの設計思想・正典モデルに興味がある場合は [`README.md`](./README.md)(ワークスペース全体の案内)と
> [`MODEL.md`](./MODEL.md)(単一の真実源)へ。このページは「使う人」向けです。

---

## 1. Moira で何ができるか

**Moira(モイラ)** は、仕様駆動(Spec-Driven)× チケット駆動(Ticket-Driven)× EVM(Earned Value Management)を
一つに統合するプロジェクト管理基盤です。ふつうの管理ツールと違い、**進捗・スケジュール・健全性を人間が
手で入力・更新しません**。開発の節目に「何が起きたか」をイベントとして追記するだけで、あとはすべて計算で導かれます。

導入するとできること:

- **進捗(達成率)の自動導出** — タスクの見積と完了イベントから EV(出来高)/ PV(計画価値)/
  SPI(スケジュール効率)/ CPI(コスト効率)を計算。「今どれだけ進んでいて、ペースは計画どおりか、
  コストは見合っているか」が数字で見える
- **生きた予測スケジュール** — メンバーの日次容量(1日に使える人日)と残作業から、
  「このままいくと いつ終わるか」を基準日を動かしながら再計算できる
- **ターミナルとブラウザの両方で可視化** — `moira show` でターミナルに導出スナップショット、
  `moira ui` でブラウザにダッシュボード(記録の追記はダッシュボードへ自動反映)
- **AI エージェントの作業も同じ土俵で記録** — 人間の作業も Claude Code などの AI の作業も、
  同じイベントログ・同じライフサイクルに載る。ただし**コミットメントを伴う判断(見積の合意など)は
  人間にしかできない**ようゲートされている
- **記録漏れの検知** — cc-sdd プロジェクトでは、spec の進行状況とイベントログを突き合わせて
  「記録が漏れている節目」を報告できる(`moira adapter drift`)

一方で、Moira は**観測と導出に徹する**設計です。タスクの自動割当や外部チケットへの自動書き込みはしません。
コミットメントを伴う五つの判断——**見積の合意・担当の割当・スコープ/期日/目標日の決定・見積の深さ・
容量の宣言**——は、つねに人間の手に残ります。

## 2. インストール

### 前提

- **Node.js 20 以上**(npm 同梱)
- git

### 手順(1度だけ・npm レジストリ公開は不要)

`moira` CLI は npm に公開されていません。この sdd-workshop リポジトリをクローンしてビルドし、
`npm link` でグローバルコマンドにします。

```bash
git clone https://github.com/PrimeBrains/sdd-workshop.git
cd sdd-workshop

cd moira/backend  && npm install && npm run build   # 導出エンジン
cd ../frontend    && npm install && npm run build   # ダッシュボード(moira ui が配信)
cd ../cli         && npm install && npm run build && npm link   # → グローバル `moira`
```

確認: 任意の場所で `moira help` が表示されれば OK。

CLI 本体は sdd-workshop チェックアウトに残り、**導入先の各プロジェクトには `.moira/`(イベントログ等)
だけが作られます**(git や eslint と同じモデル)。更新するときは sdd-workshop を `git pull` して再ビルドします。

## 3. クイックスタート — 最初の10分

自分のプロジェクトに移動して初期化し、タスクを1つ流してみます。

```bash
cd /path/to/my-project
moira init --me me --label "マイプロジェクト"        # .moira/ を作る

moira add add-login --estimate 3 --label "ログイン機能"   # タスクを登録(見積案 3人日)
moira agree add-login                                     # 見積に合意(人間の確約)
moira assign add-login --to me --slot 2026-07-07          # 担当と着手予定を決める
moira start add-login                                     # 着手
moira done  add-login                                     # 実装完了(ここで出来高が立つ)
moira accept add-login                                    # 受入
moira cost  add-login 4                                   # 実コストを記録(実測 4人日)

moira show     # ターミナルに EVM 導出(達成率・効率指標・カバレッジ)
moira ui       # ブラウザでダッシュボード
```

これだけで「見積 3人日のタスクが完了し、実コストは 4人日だった(CPI = 0.75)」という事実が
ログに残り、ダッシュボードに反映されます。コマンドの全一覧・オプションは
[`cli/README.md`](./cli/README.md) を参照してください。

## 4. 対応しているユースケース

### a. 任意のプロジェクトで手動運用(最小構成)

上のクイックスタートそのままです。言語もフレームワークも問いません。開発の節目で
`moira add / agree / assign / start / done / accept / cost` を打てば、それだけで EVM 管理が回ります。
git フック・CI・独自スクリプトから CLI を呼んで自動化することもできます(各コマンドは
「イベント1発ずつ」の独立したプリミティブです)。

### b. cc-sdd プロジェクト — 開発フローを回すだけで自動記録

[cc-sdd](https://github.com/PrimeBrains/cc-sdd)(Kiro スタイルの仕様駆動開発)を使っている
プロジェクトなら、アダプタを1コマンドで設置できます:

```bash
cd /path/to/my-cc-sdd-project
moira adapter install
```

これで `moira-track` スキル・hooks(発火検知と柵)・steering 発火表が設置され、
`/kiro-spec-requirements` → `/kiro-spec-design` → `/kiro-spec-tasks` → `/kiro-impl` と
**いつもの開発フローを回すだけで、節目のイベントが Moira に記録されていきます**。
記録が漏れていないかは `moira adapter drift` で spec と突き合わせて検査できます。

体験用のターンキーなサンプルが [`examples/todo-playground/`](./examples/todo-playground/) にあります
(TODO アプリを cc-sdd で作りながら、その開発が Moira にどう写るかを一通り体験できます)。

> hooks は POSIX シェル前提です(Windows は Git Bash で動作実証済み。ネイティブ shell は対象外)。

### c. チケット駆動 — issue の URL を渡すところから始まるフロー

「GitHub / GitLab / Backlog / Jira に issue を立て、その URL や番号を Claude Code に渡して作業開始」
というフローにも対応しています。アダプタ設置済みのプロジェクトでは、プロンプト中のチケット参照
(issue の URL、または「issue / チケット / バグ」等のキーワードを伴う `#N` / `KEY-N`)を hook が検知し、
チケットをノード化する入口(`/moira-track ticket`)を案内します。進め方は3パターンから人間が選びます:

- **既存パイプラインに合流** — チケットを spec 駆動フローの入力として通常どおり進める
- **plan → 実行** — 単一ノードのライフサイクルで進める(計画を確認してから着手)
- **直接実行** — 確認を1回に束ねた最小の儀式で進める(記録の規律・人間ゲートは同じ)

どのパターンでも同じ4イベント・同じゲートで記録が残り、そのまま可視化されます。
チケット側への書き戻し(状態同期・コメント投稿)はしません — チケットは読み取り専用の写しです。

### d. マルチリポジトリ — ログは1箇所、作業はどこからでも

設計は repo A・実装は repo B のような多リポジトリ構成でも、**1プロジェクトのログは1箇所**
(log home の `.moira/`)に集約できます。各作業リポジトリの直下に1行のポインタファイルを置くだけです:

```
# work-repo/.moira (ディレクトリではなく通常ファイル)
home: ../shared-project-home
```

以後、そのリポジトリで打つ `moira` コマンドはすべて home のログを読み書きします。
一時的な指定は `moira --dir <log-home> <command>` や環境変数 `MOIRA_DIR` でも可能です。
詳細は [`cli/README.md`](./cli/README.md) の「ログ home の解決」を参照してください。

### e. cc-sdd 以外の独自プロセス — 宣言的 provider 設定

cc-sdd を使っていないプロジェクト(独自の設計→実装→レビュー・プロセスなど)でも、
「どのファイルが書かれたらどのフェーズの節目か」を宣言的な設定
(`.claude/moira-provider.json`)で定義してアダプタを汎用に使えます。設定の生成は
インタビュー駆動の authoring スキル `moira-adapter-gen` が面倒を見ます(機械検証とスモーク確認まで)。

### f. AI エージェントの作業を区別して記録

`moira start / done / cost / add` は `--actor agent:claude` のように実行主体を指定でき、
「AI がやった作業」と「人間がやった作業」を同じログの上で区別できます。
ただし `moira agree`(見積合意)は human 限定で、agent が打っても拒否されます —
**確約は人間にしか出せない**ことがデータ層で強制されています。

## 5. 進捗が管理される仕組み

### 状態を書き換えない — イベントを追記するだけ

Moira の唯一の真実源は、**追記専用の4種イベントログ**(`.moira/events.json`)です:

| イベント | 意味 | 発行するコマンドの例 |
|---|---|---|
| `decompose` | タスク(ノード)を親の下に分解する | `moira add` |
| `transition` | 状態を進める(合意・着手・完了・受入・取消) | `moira agree / assign / start / done / accept / cancel` |
| `relate` | ノード間の関係(依存など)を張る/外す | `moira relate` |
| `cost` | 実コスト(人日)を記録する | `moira cost` |

「進捗率を 60% に更新する」のような**状態の直接書き換えは存在しません**。すべての画面・数字は
このログを先頭から畳み込んで(fold)導出されます。だから、進捗とスケジュールと健全性が
別々に管理されて食い違う、ということが構造的に起きません。過去の任意の基準日(asOf)を
指定すれば「あの時点ではどう見えていたか」も再現できます。

### タスクのライフサイクルと出来高

各ノードは2つの直交する状態を持ちます。**見積合意**(`proposed → agreed`)と、
**ライフサイクル**(`pending → ready(割当済み)→ implementing(着手)→ implemented(実装完了)→
accepted(受入)`。取消は `cancelled`)です。クイックスタートのコマンド順
(`add → agree → assign → start → done → accept`)は、この2つを実務の順に進めています。

**出来高(EV)が立つのは、合意済み(agreed)のノードが `done`(implemented)に到達した瞬間**です。
「90% できてます」のような部分進捗の自己申告は入り込めません — 終わったか、終わっていないか、だけです。

### EVM 指標とカバレッジのペア読み

見積(人日)を価値の重みとして、ログから次が導出されます:

- **EV(出来高)** — 合意済みの見積を持つ完了ノードの見積合計。**EV% = 達成率**
- **PV(計画価値)** — 基準日までに完了している「はず」だった見積合計
- **SPI = EV / PV** — スケジュール効率(1.0 未満なら遅れ)
- **CPI = EV / AC** — コスト効率(1.0 未満なら見積超過)

重要なのは、これらの指標が**カバレッジ(どれだけのノードが見積・計画を持っているか)とペアで
提示される**ことです。見積のないタスクだらけの状態で「達成率 100%」と出ても意味がないからです。
Moira は生の指標とカバレッジの両方を返し、数字の割引(デレート)はせず、判断を人間に残します。

### 人間の判断は5つだけ、あとは導出

システムが人間に求める入力は、コミットメントを伴う**五つの判断**だけです:

1. **見積の合意**(`agree`) — この見積で行くと確約する
2. **割当**(`assign`) — 誰が・いつから着手するか
3. **スコープ/期日/目標日の決定**(`deadline` ほか)
4. **見積の深さ** — どこまで分解するか(フェーズ子を省くのも正当な判断)
5. **容量の宣言**(`capacity`) — 各人が1日に使える人日 c(i,d)

それ以外——進捗率・予測完了日・遅延警告・健全性——はすべて計算です。
容量と残作業から「生きた予測スケジュール」が導出され、基準日を動かすと全画面が再導出されます。

### 見る — `moira show` と `moira ui`

```bash
moira show                    # ターミナルに導出スナップショット
moira show --asOf 2026-07-01  # 過去の基準日で再導出
moira log                     # 生のイベント一覧
moira ui                      # ブラウザでダッシュボード(追記は自動反映)
```

ダッシュボードは、仕様・価値/スケジュール・時間/活動履歴/健全性・EVM の4サーフェス+
意思決定インボックス・容量カレンダーで構成され、すべて**同一の導出結果への別視点**です。
管理者/開発者の役割切替も、別アプリではなく同一導出への提示層のプリセット/フィルタとして
設計されています。

### 束ねて見る — `moira ui --portfolio`（複数プロジェクト）

複数案件を担当する場合は、home のリストを書いた portfolio.json を渡すと
案件横断のポートフォリオ表示になります（読み取り専用）:

```bash
moira ui --portfolio portfolio.json
```

```jsonc
// portfolio.json
{
  "schemaVersion": 1,
  "label": "部門ポートフォリオ",
  "homes": [{ "path": "../proj-a" }, { "path": "../proj-b" }]
}
```

**案件並置**（各案件の EV%・SPI/CPI・バッファ残・レビュー待ちを1行1案件で）、
**人横断**（同じ人の掛け持ちの見える化）、行クリックで通常の単一案件ダッシュボードへ
ドリルダウンできます。各案件は home ごとに**独立に導出**され、**同じ基準日なら**単独で
開いたときと完全に同じ数字です（ポートフォリオは比較のため全案件を統一基準日で導出します。
既定は各案件の基準日の最大値 — 画面上部で変更可）。横断の合成会計（EV% のロールアップ等）
は意図的にありません（D-50: 単一プロジェクト視点は不変）。詳細は
[`cli/README.md`](./cli/README.md) の「ポートフォリオ」節へ。
