## 1. 前提: 訂正プリミティブの現在地

Moira は 4 イベント（transition / decompose / relate / cost）の追記専用ログ（第一層）と、c(i,d)・deadline/target の理由付き追記履歴（第二層）で動く。各イベントの「訂正の一次手段」は次の通り。

| イベント型 | 訂正の一次手段 | 根拠 |
|---|---|---|
| transition (lifecycle) | 前進/後退とも fold は無条件受理（状態機械の辺を検証しない） | fold.ts:142-158 |
| transition (estimate) | `agreed→proposed` 再見積（latest-wins、**未完了限定**） | R-E3 (MODEL:308) |
| decompose | 同一子への再 decompose = 所属の latest-wins 置換 | §2.8 (MODEL:161-164) |
| relate | `op=remove` → `op=add` の追記 | §2.8 表 (MODEL:155) |
| cost | **訂正イベントなし**（非負 A6・id dedup のみ） | §7#19(a) (MODEL:598) |
| 第二層 (c / dates) | 新エントリ追記（latest-ts-wins）で常に上書き可能 | R-U14 (MODEL:289), R-T6 (MODEL:357) |

構造的な非対称として、`DecomposeEvent.reason` と `CapacityEntry.reason` は必須なのに、**`CostEvent` / `RelateEvent` には reason フィールド自体が存在しない**（types.ts:68-81）。訂正操作を行っても「なぜ」を構造化して残す場所がない。

## 2. 不可逆パターン網羅カタログ（モデル/イベント面）

### 2.1 cost の誤り — 最も硬い不可逆

| シナリオ | 回復手段 | 根拠 / fold 実挙動 |
|---|---|---|
| 額の誤り（0.5→5） | **なし**。負値相殺不可・訂正イベント無し | A6 (MODEL:53)、§7#19(a) (MODEL:598)。fold.ts:237-241 は負値のみ拒否、正の誤額は無条件加算 |
| ノードの誤り（別タスクに計上） | **なし**。正しいノードへ再計上しても誤ノードの ownCost は減らない=サンク | fold.ts:244 `ensure(ev.node).ownCost += ev.amount` は無条件。幽霊ノードも自動生成 (fold.ts:92-99) |
| 日付(ts)/担当者(actor)の誤り | **なし**。イベント発行後は不変(A2)、reason 欄も無い | types.ts:43-47, 77-81。fold は ts/actor を検証しない |

### 2.2 ライフサイクル遷移の誤り — 構造と規範の二重基準

| シナリオ | 回復手段 | 根拠 / fold 実挙動 |
|---|---|---|
| 誤 cancelled（誤キャンセル） | **部分的**。正典の回復路は「新ノード再作成」のみ（コストは旧ノードにサンク R-C2、系譜は supersede 辺・reason 欄で保ちうるが保証なし） | §7#19(b) (MODEL:598)。**ただし fold は終端性を強制しない**（fold.ts:142-158 は無条件代入）— 技術的には un-cancel が黙って通るが、R-C2/§2.7 の cancelled=終端前提と矛盾する未定義動作になる |
| 誤 implemented（誤完了） | **あり**。後退遷移は P5 が明示的に許容する正直な非単調シグナル | P5(a) (MODEL:199-202)。fold は後退を拒否しない |
| 誤 agreed（誤合意・未完了） | **あり**。R-E3 で proposed へ戻し再見積 | fold.ts:178-193。**ただし frozenBudget は明示クリアされず stale 残留**（実害は導出のゲート次第、要確認） |
| 誤 agreed（完了済みノード） | **なし**。合意した瞬間 I4 施錠が即発動 | I4 (MODEL:170)、fold.ts:184-190 が拒否 |
| スキップ遷移（pending→accepted 等） | fold は**拒否しない**（辺集合の検証コードが存在しない） | I5 (MODEL:171) は machine の明示のみ要求。§7#19(b)「fold 強制の欠如=書き込み層規律」は全遷移に当てはまる |
| machine 誤指定（estimate 値が lifecycle に混入） | **未検証・汚染リスク**。`ev.to as LifecycleState` の無条件キャスト | fold.ts:144-145。ランタイム検証なし |

### 2.3 見積・構造・指名の誤り

| シナリオ | 回復手段 | 根拠 |
|---|---|---|
| 見積額誤り（proposed） | **あり**。latest-wins 再提案 | §2.2 (MODEL:90)、fold.ts:135 |
| 見積額誤り（agreed・未完了） | **あり**。R-E3 経由 | 同上 |
| 見積額誤り（agreed・完了済み） | **なし**。正典の処置は訂正ではなく **supersede**（別会計への切替。旧ノードの誤 frozenBudget と EV 寄与は不動） | I4、§2.7 (MODEL:139-140)、R-E3 本文 |
| 依存辺の誤追加/誤削除 | **あり**。remove/add の追記。**落とし穴**: 辺は重複を許し (fold.ts:212-216)、remove は policy を無視して (from,to) 全件一括削除 (fold.ts:224-226) | §2.8 表、I2/R-D3 |
| supersede 辺の誤追加 | **あり（唯一クリーンな回復路）**。supersede 元を cancelled にすると旧ノードが有効集合に自動復帰 | R-S5 (MODEL:328-329)、§2.7 (MODEL:147-148)。※復帰ロジックは fold 外の導出層の責務 |
| 誤った親への decompose | **あり（所属のみ）**。正しい親へ再 decompose。ただし完了済み葉を子持ちにした場合、補償まで EV_abs/PV 寄与が消える | §2.8 (MODEL:161-164)、§7#19(c) |
| 誤子作成（ID タイプミス→幽霊ノード） | **部分的**。cancelled 化で会計除外はできるがノード自体は消えない | A1/A2、fold.ts:92-99 |
| cancelled/完了済み親への move-in | 構造拒否**されない**（既知の残余） | §7#19(f)、fold.ts:104-138 は親の状態を見ない |
| 所属の矛盾主張（複数人が別親を同時主張） | 警告なし・沈黙 latest-wins | §7#19(d)。R-U12 同型の警告は未拡張 |
| assignee/reviewer の誤指名 | **あり**。latest-wins 置換 | §2.4、fold.ts:148,152 |
| assignee/reviewer の**解除**（未割当に戻す） | **なし**。解除のセンチネルが未定義 | §7#16(c)/#18(d) (MODEL:592,596) |
| 第二層（c(i,d)・deadline/target）の誤登録 | **あり（容易）**。latest-ts-wins 追記で常に訂正可能 | A4、R-U14、R-T6 |

## 3. 操作経路の無警告混入パス（実装面）

前提: **確認プロンプトはリポ全体に一つも存在しない**。ストレージ層は無検証追記のみ。frontend (`moira ui`) は完全読み取り専用で、誤操作経路は CLI と xlsx インポートに集約される。

### 3.1 特に危険な無警告パス（重大度順）

1. **`moira assign --slot` の再訂正がサイレントに無効** — frozenSlot は初回のみ書き込まれ (fold.ts:156-158)、2回目以降は完全無視。**しかし CLI は毎回成功メッセージを出す**ため訂正できたと誤認する。assignee/reviewer は latest-wins なのに frozenSlot だけ非対称で、その旨の表示が一切ない (commands.ts:224-252)。
2. **`moira cost` の NaN 素通り・存在チェック皆無** — `Number("5o")=NaN` は負値ガード（`NaN<0`=false）をすり抜け、ac.ts の合算でプロジェクト全体の AC/CPI を NaN 汚染する (commands.ts:270-281, fold.ts:237-241, ac.ts:16-38)。ノード ID タイプミスは幽霊ノードに無警告計上。
3. **WBS 再インポートによる訂正がブロック** — 既存 ID を「再インポート非対応」としてエラー拒否 (wbs-import.ts:165-167)。記入ミスを直したファイルの再投入という最も自然な訂正フローが構造的に不可能で、回避策は別 ID（重複ノード残留）か events.json 手編集のみ。
4. **ライフサイクル遷移の無検証** — cancelled→implementing の逆行、pending→accepted の飛び級が無警告で成立 (fold.ts:142-158)。

### 3.2 経路別サマリ

| 経路 | 防御 | 復旧性 |
|---|---|---|
| `moira cost` | 負値のみ fold で拒否。NaN/上限/ノード存在は素通り | **裏口（events.json 手編集）のみ** |
| `moira start/done/accept/cancel` | 実質無検証（R-U4 と done-lock のみ） | 上書き遷移は可能だが「打ち消し」専用手段なし |
| `moira assign --slot` | 日付形式検証なし。再実行は無言で無効 | **なし**（成功表示が嘘をつく） |
| `moira add --parent` | 明示 parent 時は既存ノードチェックなし → 無警告で再アタッチ+見積上書き | 再 decompose で戻せるが気付けない |
| `moira capacity` | 日付形式・[0,1] 範囲とも未検証（`member add --capacity` にはあるのに非対称） | latest-ts-wins で**訂正可能** |
| `moira deadline` | isIsoDate あり。target>deadline は警告のみ（意図された仕様） | 追記で**訂正可能** |
| `moira relate` | 循環検出あり・存在確認なし | `--remove` で**取り消し可能**（数少ない組み込み取り消し） |
| `moira import wbs` | 検証は手厚いが、通過後の訂正=再インポートは**拒否** | **裏口のみ** |
| `moira import members` | roster upsert + capacity 追記 | 再インポートで**事実上の訂正可能**（訂正に強い設計） |
| frontend | 書き込み経路ゼロ | — |

## 4. 普遍訂正原則（**ユーザー裁定 2026-07-18: 合意済み**）

当初の「操作別の可/不可リスト」による切り分けを廃し、次の単一原則を採る。

> **すべてのイベントは訂正可能。ただし**
> **(a) 追記のみ** — イベントの書き換え・削除は行わない。訂正それ自体が履歴に残る（A2 維持）
> **(b) reason 必須** — なぜ訂正するかを構造化して残す
> **(c) 元イベント id 名指し** — どの記録への訂正かが一意に辿れる
> **(d) 訂正は計器に見える** — 隠れて直せない。訂正の存在・頻度・対象（特に施錠対象への訂正）は health 計器として常設表示される
> **(e) 対象は記録であって事実ではない** — 「5MD 使ったことを無かったことにする」ではなく「5MD という記録が誤りで真値は 0.5MD」を表明する

### 4.1 原則の下での既存不変量の読み替え

- **A2 追記専用**: そのまま維持（原則 (a) そのもの）。ログの完全性・順序 (I3)・id 冪等性が訂正安全性の土台。
- **A6 非負**: 維持するが射程を明確化 — 負値 cost（マイナス相殺イベント）は導入しない。訂正は「打ち消しの負値」ではなく「元記録 id への訂正表明」(c) で表現するため、A6 と衝突しない。A6 が守るのは**事実の存在論**（負の投下時間は無い）であり、記録の訂正 (e) はその外。
- **I4 完了施錠**: 「不可」から「**訂正すると大きく音が鳴る**」へ格下げ。完了+合意済みへの訂正は禁止しないが、reason 必須+計器常設表示 (d) で統制された公式イベントになる（EVM 実務の統制された再ベースラインと同型）。ゲーミング（都合の悪い過去の「誤記」化）は禁止でなく可視化で防ぐ。
- **cancelled の終端性**: un-cancel 専用機構は作らない。「cancel は誤記だった」という**記録訂正** (c)(e) として表現し、導出層が訂正適用後の状態を再計算する。fold が現状黙って通す裸の逆行遷移（訂正の体裁を持たない書き込み）は、引き続き書き込み層で塞ぐ側。

### 4.2 原則を先取りできる実装群（正典改定前でも着手可 — 書き込み層規律）

原則の (b)(d) 「隠れて直せない・正直に見える」に奉仕する検証・案内。moira-model-update を待たない。

1. `moira cost` の入力検証: `Number.isFinite`・ノード存在確認（幽霊ノード防止）・異常値警告。fold 側にも NaN ガード。
2. `moira assign --slot` の正直化: 2回目以降は「初回凍結済みのため無効」を明示（嘘の成功表示の廃止）。日付形式検証。
3. ライフサイクル遷移の書き込み層検証: 訂正の体裁を持たない裸の逆行・飛び級遷移を CLI で拒否（fold は寛容のまま=過去ログ互換）。
4. `moira capacity` の日付形式・[0,1] 範囲検証（`member add` と対称に）。
5. relate の重複辺防止と remove の policy 指定対応。
6. WBS 再インポートの差分モード（訂正フローの開通）。
7. 破壊的コマンドの確認プロンプト（少なくとも cancel / done）。
8. frozenBudget の stale 残留の実害調査と是正。

### 4.3 残る設計論点（原則は合意済み・詳細が未裁定）

1. **ゲーミング耐性の計器設計** — 「音が鳴る」の具体化: 訂正件数/施錠対象への訂正を health のどこへどう常設するか。閾値や色分けは裁量ノブ（L47/§7#11 の先例）にしない工夫が要る。
2. **歴史の読み直し** — 訂正は現在の射影にのみ効き、保存済みレポート（--out の md）は当時の文書として不変、で足りるか。二時間軸（起きた時刻×知った時刻の bitemporal）まで踏み込むか。前営業日比（moira report の Δ）が訂正を跨ぐときの表示も要検討。
3. **表現形** — 訂正記録を第二層ファイル（R-U14 同型）とするか、5個目のイベント型とするか。§5「イベント4型」の方針とどう折り合うか（第二層案が現時点の推奨）。
4. **訂正の訂正** — 訂正記録自体が誤りだった場合の連鎖規則（latest-wins か、明示 supersede か）。
