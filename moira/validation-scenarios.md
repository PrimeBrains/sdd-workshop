# Moira 検証シナリオ

> 目的:MODEL(v16)が「実際に使えるか」を、状況単位のユースケースで検分する。各シナリオは
> **機能面**(操作が追記イベントで成立するか)と **判断面**(導出された数字が管理者の判断を支えるか)の
> 両面を持ち、対応する EARS 要件に紐づく(トレーサビリティ)。
>
> 完成基準:4群(立ち上げ/健全性/リスケ/変更例外)と主要な判断がカバーされていればよし(形式的網羅は目的でない)。
> 次の一手:**S4(健全性のリアルタイム把握)を一本貫き**、「イベント→導出→表示」の型を作る。

## A. 立ち上げ

### S1 プロジェクト立ち上げ
- **機能面**:計画(roadmap/brief)を入力に root が誕生。`transition`(root: draft→planning)で計画作業が出来高(EV_abs)を獲得、`decompose` で feature 群へ概算配賦(`proposed`)、`relate` で粗い feature 間先行関係。ベースラインは一括凍結でなく、各見積が `agreed` になるたび予算が、初回スケジュール載り時にスロットが**個別に**凍結記録される(§3。立ち上げ時点では多くが proposed=未確定)。
- **判断面**:「WBS=木そのもの・概算見積=配賦イベントの属性」が成立し、独立した「WBSを引く」活動が消えていることを確認(スケジュールは割当が付いてから導出されるので立ち上げ時点では未展開)。
- **EARS**:R-U1, R-U2, R-E1, R-U7, R-D1

### S2 見積の提案とノード単位の人間合意
- **機能面**:AI・過去実績・FP法など出所を問わず `proposed` の見積を提案、人間がタスク/feature ノード単位で `agreed` へ(配下チェック項目は一括)。合意で PV の**予算**が確定(スロットは初回スケジュール載り時;§3)。
- **判断面**:未合意・未見積が**見積カバレッジ**として可視化され、EV%(達成率)が「合意済み領域のみ」を語っていることを確認。
- **EARS**:R-U3, R-U4, R-U5, R-U8, R-U9

### S3 フェーズ進行(requirements 承認→design 移行)
- **機能面**:`req-X` の `implemented→accepted`(actor=human)= requirements 承認。これで辺 req→design(既定 `accepted`;R-D2)が充足し `design-X` が ready(R-D1)。design の見積合意(EV basis)を経て implementing。フェーズ=ノード(§2.6)。
- **判断面**:仕様作業が出来高(EV_abs)に乗り「仕様を書くのも進捗」が実現。低カバレッジゆえ EV%(達成率)を全体完了度として見せない(R-S4)。→ 詳細は `thought-experiments/01`
- **EARS**:R-E1, R-E2, R-D2, R-S4

## B. 健全性(判断面が主)

### S4 健全性のリアルタイム把握 ★最初の一本
- **機能面**:任意のイベント追記時、同一ログから導出状態(各ノード状態・EV%・EV_abs・**見積カバレッジ・実行カバレッジ(R-S8)・スケジュール・カバレッジ**・PV(ベースライン)・AC・SPI・CPI・キュー・予測スケジュール・未割当バックログ・**スケジュール・バッファ残量/消費率(R-T6)**)を提供。
- **判断面**:管理者が「EV% 80% / 見積カバレッジ 30% / **実行カバレッジ 60%** / SPI(スケジュール・カバレッジ付き) / CPI / **バッファ残量**」を**対で**読み、今の健全性を即座に把握できるか。EV% と見積カバレッジ、SPI とスケジュール・カバレッジのペア読みに加え、**EV% と実行カバレッジの三者対読み**(R-S8;EV% が低いが実行カバレッジが高い＝仕掛中は多いがまだ完了していない、EV% が高く実行カバレッジが低い＝大部分完了で残りが少ない)、および**バッファ残量と SPI の組み合わせ**(R-T6;SPI 良好でもバッファが縮小していれば先行消費)が機能するか。
- **EARS**:R-S2, R-S4, R-S6, R-S8, R-T6, R-U9
- **備考**:判断面の最重要シナリオ。「イベント→導出→表示」のリアルタイム追随(R-S2)はここで型を確立できる。ただし各シナリオは固有の判断ロジック(S7 の thrashing 二項条件・S10 の at-risk 前後差・S6 のスケジュール・カバレッジ de-rate 等)を持つので、S4 が他を**代替するわけではなく**、共通基盤を S4 で固める位置づけ。

### S5 期日検算
- **機能面**:生きた予測スケジュール(P7/P8、**人間のみ** c(i,d) 平準化;エージェントはスラックなしで間隙を埋める)を外部期日・目標日と照合。クリティカルパスは人間タスク連鎖。期日・目標日はプロジェクト構成入力(R-T6)であり、**スケジュール・バッファ残量 = max(0, 期日 − D_pred)** と **バッファ消費率 = clamp₀¹((D_pred − 目標日)/(期日 − 目標日))** を導出。
- **判断面**:工数ベース進捗と独立に「期日に間に合うか」を検算し、超過時はアラートのみ(自動で要員追加・スコープ削除しない;判断は人間)。バッファ残量による**超過前の監視**が R-T4 アラート(**超過時の警告**)の前段として機能し、スケジュール・カバレッジとの de-rate 対読みが偽の安心を防止するか(R-S6 同型)。境界条件(目標日なし/期日なし/目標日=期日/目標日>期日)はそれぞれ §3/R-T6 のとおり扱われるか。
- **EARS**:R-T1, R-T4, R-T6

### S6 見積精度(カバレッジ)の信頼性
- **機能面**:見積カバレッジ・スケジュール・カバレッジを導出。
- **判断面**:低カバレッジ時に EV%/SPI が割り引かれ、「霧の中の既知部分の達成度」と正しく解釈されるか(偽りの安心の防止)。
- **EARS**:R-S4, R-S6, R-U9

### S7 エージェント空回り(thrashing)検出
- **機能面**:EV_abs が停滞する一方 AC が継続的に増える状況をフラグ。畳んだ見積活動の一度きり cost は誤発火させない。
- **判断面**:「エージェントが空回りしている」を管理者がリアルタイムに検出できるか(この仕組み固有の最重要シグナル)。
- **EARS**:R-S3

## C. リスケ

### S8 過負荷起点リスケ
- **機能面**:ある人物・期間で c(i,d) 平準化に収まらない割当 → 過負荷アラート。割当を別人/エージェントへ付け替え(被割当者を名指す `transition` の追記、append-only)。
- **判断面**:付け替えの瞬間にスケジュールと EV がリアルタイムに追随し「これで収まった/まだ無理」を即読めるか。
- **EARS**:R-T3, R-T5

### S9 見積膨張起点リスケ
- **機能面**:上方再見積(`proposed` へ戻し人間合意)→ EV% 低下を「発見/スコープ増大」の正直な信号として顕在化。凍結値+理由を保持(説明責任)。
- **判断面**:EV の非単調性(P5)を異常でなく正直な信号として読み、期日への影響を検算できるか。
- **EARS**:R-E3, R-E4, R-U7, R-S2

### S10 手戻り起点リスケ
- **機能面**:タスク却下で後退遷移。`implemented`(=仕様FIX)到達の前後で意味が異なる(到達後の差し戻し=異常)。下流の解放済み後続を at-risk として警告。
- **判断面**:玉突きの手戻りを DAG から辿り、どの後続が危険かを特定できるか。→ P5 の at-risk が真の対象を指すか。
- **EARS**:R-D1, R-D2, R-E4(後退遷移で EV 低下を許容)/ 原理:P5(implemented 到達前後で後退の意味が異なる。EARS 要件でなく原理)

## D. 変更・例外

### S11 スコープ削除とサンクの可視化
- **機能面**:feature 取り下げ=終端 `cancelled` への `transition`(イベント削除しない)。取り消しノードを active basis から除外、サンク EV_abs を導出(既に費やした出来高は append-only で消えず**回収不能**)。
- **判断面**:サンク(無駄になった労働)が可視化され、現行 basis と区別されるか。supersede(一度役立った)との違いが読めるか。
- **EARS**:R-C1, R-C2

### S12 エンハンス(supersede)
- **機能面**:エンハンスは root 直下の新 feature。既存変更は後退でなく新ノード+supersede 辺(R-D7)。旧ノード不変。
- **判断面**:累積EV(EV_abs、supersede 込み。NAMING §7 正式語、旧称「累積稼得」)と現行有効集合(R-S5)が区別表示され、陳腐化した旧機能を現行有効と誤読しないか。累積EV(EV_abs)は coverage と独立に読む(§2.7)。S11 の cancelled が basis から除外されるのと対照的に supersede は EV_abs に残る——だが両者ともカバレッジ分母には二重計上しない(P2)。→ 詳細は `thought-experiments/02`
- **EARS**:R-D7, R-S5

### S13 再ベースライン(説明責任)
- **機能面**:凍結属性の改訂(理由必須)で未完了ノードの PMB を動かす——**予算**は凍結見積値の改訂(R-U7)、**スロット**は初回スケジュール載せの凍結属性の改訂(§3。予算とは確定契機が別)。完了ノードは施錠(I4)。
- **判断面**:「どのタスクが・どの理由で・どれだけ積み増したか」を時系列で再構成でき、顧客への説明責任を果たせるか(S9 が動かす latest=予測/EV% と異なり、S13 は凍結ベースライン=PV/PMB を動かす)。
- **EARS**:R-U7, R-E3(/ スロット側は §3 のベースライン改訂)

## E. v15→v16 固有判断（S4 の共通基盤では代替できない）

### S14 執行中の可視化（実行カバレッジ R-S8）★健全性群
- **機能面**:合意済み有効葉のうち `implementing` にあるノード数比率を導出（分母0→0・honest empty）。EV%・見積カバレッジと三者併置。
- **判断面**:完了主義の EV% が動かない執行中区間を集約レベルで読めるか。**固有ロジック**：(a) `implementing` に居座るだけの葉が実行カバレッジを高止まりさせ「仕掛中は多いが進んでいない」偽信号を出す場合、R-S7 乖離・P6 リードタイムと**対読み**して実際の前進と区別できるか。(b) EV% と算術和して全体進捗としない（仕掛中の量≠出来高；R-S4/R-S6 同型の de-rate）規律が守られるか。(c) `ready→implemented` スキップ経路では doing が立たず忠実度が落ちる（凍結スロットが SPI の前提であるのと同型）ことを認識した上で読めるか。
- **EARS**:R-S8（補助:R-S7, P6 と対読み）
- **備考**:S4 備考が「S4 は固有判断を代替しない」と明示するとおり、R-S8 の居座り検出・スキップ忠実度は S4 の三者対読みでは尽きず本シナリオで個別検分する。

### S15 スケジュール・バッファの境界条件（R-T6）★リスケ群
- **機能面**:期日・目標日（構成入力 R-T6）と D_pred から、バッファ残量 = max(0, 期日 − D_pred)・消費率 = clamp₀¹((D_pred − 目標日)/(期日 − 目標日)) を導出。スケジュール・カバレッジと対で de-rate。
- **判断面**:**固有ロジック（境界条件）**を各個に検分する——目標日なし→残量のみ・消費率 N/A、期日なし→バッファなし、目標日 = 期日→消費率 N/A、目標日 > 期日→構成エラー警告（自動拒否せず；§2.1）、D_pred なし→算出不能として可視ギャップ（P0）。低スケジュール・カバレッジ時にバッファを de-rate しないと未割当バックログが偽の安心を生む（R-S6 同型）ことを示せるか。R-T4（期日超過警告）と R-T6（超過前監視）が同一導出完了の表裏として整合するか。
- **EARS**:R-T6（補助:R-T4 と表裏、R-S6 と de-rate 対読み）
- **備考**:S5（期日検算）はバッファの正常系を扱い、S15 は境界条件・de-rate・R-T4 表裏の固有判断を担う。

## カバレッジ確認
- **5群を覆う**:A 立ち上げ(S1–S3)・B 健全性(S4–S7)・C リスケ(S8–S10)・D 変更例外(S11–S13)・E v15→v16 固有判断(S14–S15)の15シナリオで、観測・導出・警告とコミット判断(合意・割当・スコープ/期日/目標日)を網羅する。MODEL は「思想の確定」であり UC の正式な列挙は持たないため、被覆は本文書の群構成で閉じて主張する。
- **共通基盤と固有判断**:多くのシナリオの判断面は「導出がイベント後リアルタイムに追随し読めるか」(R-S2)を共有し、S4 でこの共通基盤の型を固められる。ただし各シナリオは固有の判断ロジックを持ち S4 では代替できない:S6 のスケジュール・カバレッジ de-rate(R-S6)、S7 の thrashing 二項条件と誤発火抑止(R-S3)、S10 の at-risk 前後差(P5)、S11/S12 のサンク/累積の区別(R-C2/R-S5)、**S14 の実行カバレッジ居座り検出・スキップ忠実度(R-S8)、S15 のバッファ境界条件・de-rate・R-T4 表裏(R-T6)**など。これらは個別に検分する（S4 はこれらを代替せず共通基盤を固める位置づけ、と S4 備考が明示）。
- **v9→v16 の反映**:EV を EV_abs/EV% に二形分離、SPI はスケジュール・カバレッジ(R-S6)と対読み、単一通貨(A6)、EVM ベースライン(予算=合意時・スロット=初回スケジュール時に凍結記録)を反映。ベースライン二次元は主に S2(予算)・S5(スロット/期日)・S13(再ベースライン)で扱う。
- **v12→v13 反映**:固定容量 1.0MD/日 を日次容量 c(i,d) へ一般化。S5/S8 の平準化条件を c(i,d) に更新。
- **v14→v15 反映**:S5 にスケジュール・バッファ(R-T6)の期日・目標日構成入力とバッファ残量/消費率を追加。境界条件の固有判断は S15 で個別検分。
- **v15→v16 反映**:S4 に実行カバレッジ(R-S8)を共通基盤として追加し、その固有判断（居座り検出・スキップ忠実度）は S14 で個別検分。
- **0e（MODEL v16 Phase 0）整合**:S1 EARS に R-D1（relate による粗い feature 間先行）を含む、S8 EARS に R-T5（単一被割当者）を含むことを確認済み。本文書は MODEL v16・moira-model-update Phase 0 独立敵対ゲート通過（残存 Critical/Important = 0）に整合。
