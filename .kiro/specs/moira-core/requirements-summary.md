# moira-core 要件 早わかり

> requirements.md(EARS 形式の要件本文)を読む前に全体像を掴むための要約。詳細は [requirements.md](./requirements.md) へ。

## 目的

`moira-core` は、Moira 正典モデル `moira/MODEL.md`(SSOT) を本番アーキテクチャへ落とす **CQRS 分解の Wave0（基盤契約）**。Moira は read spec 全10本＋write skill 全14本で構成され、core はその Wave0 で、**core 以外の全 spec/skill が依存する書き込み契約（emit）・読み出し契約（derive）とデータモデル**を最初に凍結する。

設計の最上位原則は **「core は構造・不変条件・機構だけを所有し、計算式・しきい値の評価・値の算出はすべて下流 spec が持つ」**（"構造境界 core"）。MODEL の文言は変えず・新概念は足さず、その実装への落とし込みに徹する。

## 主要な要件の要点

- **二層データと単一実体（要件1）**：実体は spec とその分解であるノードのみ。ノードのワークデータは 4 イベント（`transition`/`decompose`/`relate`/`cost`）の追記専用ログ、構成入力（capacity c・期日/目標日）は別の第二層に固定する。ログは削除せず不変（A1/A2）。
- **唯一の書き込み口（要件2）**：書き込みは emit／第二層追記のみ。保存された導出状態への直接 mutation を拒否し、読み出しは状態を変更しない。第二層の同一キー矛盾は `(ts,id)` latest-wins で解決する（I3 同型）。
- **4 イベントのスキーマと意味（要件3）**：4 種それぞれの意味を一意に固定し、誰が emit しても同じ畳み込み結果になるようにする。実コストは人間アテンション時間（MD）の単一通貨で記録し EV に混ぜない（A6）。
- **見積合意は人間限定（要件4）**：あらゆる見積を `proposed`/`agreed` 状態を持つ提案として扱い、非人間（エージェント）の `agreed` を構造的に拒否する（I6）。再見積は人間が再合意するまで `proposed` に戻す。承認手続き自体は `moira-estimate-agree` が所有。
- **ライフサイクル状態機械（要件5）**：`pending→ready→implementing→implemented→accepted`（終端 `cancelled` は任意の非終端状態から到達可。`accepted→cancelled`＝feature 取り下げを含む）を全ノードへ一様適用する。各 `transition` は対象機械（lifecycle／見積合意）を明示する。`implemented` は「実装完了 ∧ 仕様FIX」の**判定**（保証ではない）。`cancelled` は受理して畳み込む。後退遷移は合法性を理由に拒否せず記録し、拒否は構造違反（循環 I2・非人間 agreed I6）に限る（意味的異常の警告は下流 P5／`moira-health`）。
- **単一被割当者（要件6）**：被割当者を `transition` の属性として単一・置換（latest-wins）で記録する。人間のスキル/習熟はモデル化しない。
- **凍結ベースラインの記録（要件7）**：見積を「EVM 用の最新値」と「説明責任用の凍結値」で持つ。予算＝合意時、スロット＝初回スケジュール時に凍結記録し、完了時に確定済み次元を施錠（I4）。完了時の凍結見積寄与値の**算出**（R-S1）は `moira-evm`、core は凍結値の記録と施錠を担う。
- **決定的マージ（要件8）**：全 event id を大域ソート可能にし `(ts,id)` で決定的にマージ。同一ログ・同一実装で再導出が再現する（I3/R-D5）。
- **辺の構造保持と循環拒否（要件9）**：辺の型・充足ポリシー・既定値を構造として保持し、分解しても辺をノード水準で保持して子に増殖させない。循環 `relate` は拒否（I2）。述語の評価・既定の適用は `moira-scope-deps`。
- **supersede と現行有効集合（要件10）**：現行有効集合を「supersede されていない葉」として累積 EV とは区別して導出。エンハンスは新ノード＋supersede 辺で表し旧ノードは不変。新ノード cancel 時は旧ノードが復帰する（R-S5）。
- **I1 見積整合（要件11）**：親の最新見積を合意済み子の総和に保つ。未見積で生まれた子と、完了済みだが未合意のノードは総和から除外する（カバレッジ低下・未合意完了警告の顕在化は下流 `moira-evm`/`moira-health`）。
- **オーケストレータ seam と再導出契機（要件12）**：R-S2 の導出群（ノード状態・EV・カバレッジ・PV/AC/SPI/CPI・各キュー・予測スケジュール・バッファ 等）を同一ログから読み出す単一 seam を core が開ける（値は下流供給・消費者ごとの再計算は禁止）。再導出契機はイベント追記または構成入力変更に限る。未コミット領域は可視ギャップとして公開する。
- **チケットは射影（要件13）**：spec とノード分解を唯一の真実とし、チケットはノードの読み出し専用射影。チケット→ノードの逆書き込み経路は持たない。
- **永続化方式の決定（要件14）**：`(ts,id)` 順序が再ロードで保たれ、ログから再導出が再現でき、第二層を独立 tier として扱える永続化方式を本 spec の設計で決める。

## スコープ

**やること（core が所有 — 構造・不変条件・機構）**
- **API 契約**：emit／derive／capacity-write／config-write（型・前提・後条件）。
- **データモデル**：4 イベント型と第二層スキーマ、ノード ライフサイクル機械・見積合意機械（人間限定）、辺の型・既定 policy の構造保持、チケット＝読み出し専用射影の不変条件。
- **fold が機械的に担保する不変条件**：`(ts,id)` 決定的マージ（I3）、循環拒否（I2）、非人間 agreed 拒否（I6）、effective-set と supersede×cancel 復帰（R-S5）、I1 総和除外、辺の非物理増殖、旧ノード append-only、ベースライン凍結記録と完了施錠（I4）。
- **永続化方式の決定**（永続層の境界・再導出再現性）。

**やらないこと（下流が所有）**
- ready-eligible 評価・辺述語の評価＝`moira-scope-deps`（tree+DAG 表示・orphan・restoration の読みも）。
- ライフサイクル状態遷移の emit＝`moira-progress`。
- 指標式本体（EV_abs/EV%/見積カバレッジ/実行カバレッジ/PV/AC/SPI/CPI/sunk）＝`moira-evm`。
- 平準化（P7/P8）・予測・slot 充填・スケジュールバッファ＝`moira-schedule`。
- 9 警告の確定・集約・clearance＝`moira-health`。
- spec-unit 正規化＝`moira-ingestion-adapter`（roadmap 上は条件付き存続。確定次第 `spec-ingest` に吸収されうる）。
- 全 read サーフェス（UI）と全 write skill（イベント発行・人間承認オーケストレーション）。
