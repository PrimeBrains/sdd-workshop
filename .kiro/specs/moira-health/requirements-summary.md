# moira-health 要件 早わかり

> requirements.md(EARS 形式の要件本文)を読む前に全体像を掴むための要約。詳細は [requirements.md](./requirements.md) へ。

## 目的

`moira-health` は、Moira 正典モデル `moira/MODEL.md`(SSOT) を本番アーキテクチャへ落とす **CQRS 分解の Wave2（警告の確定・集約・clearance）**。MODEL の中核思想「システムは観測・導出・警告に徹し、コミットメントを伴う判断は人間に残す」(§0/§2.1) を受け、**警告(warning)の確定・集約・clearance(消滅)**を所有する read=導出 spec。あわせて各警告の**行為列挙を導出層で一度だけ単一定義**する。**上流** derivation(`moira-evm`/`moira-schedule`/`moira-scope-deps`)が出す検出データを受けて警告を確定し、`moira-core` の derive 契約を消費する。自前の真実源・可変状態は持たない。

## 主要な要件の要点

- **（要件1）** 9 警告すべて(R-U12・R-U13・R-T3・R-T4・R-S3・R-S6・R-S7・R-C3 および P5 at-risk)を**現在の導出状態に対する述語**として導出する。歴史的事実の蓄積ではない。4 イベント追記時および構成入力(c・期日・目標日)変更時に全警告述語を再評価する。上流 derivation の検出データと `moira-core` の derive 出力を消費し、自前の真実源も隠れた可変キャッシュも持たない。
- **（要件2）** 各警告の clearance(消滅)は**条件を偽化する入力(4 イベント追記 または 構成入力変更)でのみ**発生する。条件が真の間は、人間が見て受容しイベントを起こさない(正当な非コミット)場合でも警告は持続する。acknowledge・dismiss・既読の可変状態を一切保持しない。各警告について clearance 条件を導出データとして提示する。
- **（要件3）** 行為を伴う各警告の具体的な行為列挙を**導出層に一度だけ**読み出し専用の導出データとして定義する(R-U13={即時事後合意・再見積(R-E3)→合意・cancel}、R-T4={スコープ削減・要員追加・期日変更}、R-S7={理由付き再ベースライン・確認}、R-C3={辺の除去(`relate`)・代替先行への付け替え(`relate`)・後続のキャンセル(`transition`)})。各警告要件(要件6/7/8)は具体リストを再掲せずこの列挙を参照する。下流サーフェスや skill は行為列挙を再定義しない。
- **（要件4）** 判断・行為を要する 8 件の警告(R-U12・R-U13・R-T3・R-T4・R-S3・R-S7・R-C3 および P5 at-risk)を decision インボックスへ集約する。de-rate 型の R-S6(SPI de-rate)は inbox 項目ではなく**常駐メトリクス修飾という集約形**に分類し、9 件すべてに host を確保する(inbox 8 件+R-S6 常駐=計 9)。R-S6 の SPI de-rate 消費は `moira-evm`、提示は `moira-surface-health` が所有し、health は分類のみ。
- **（要件5）** 条件が真であるすべての警告を**可視ギャップの会計**(カウントとリスト)に残す。提示が淡色化・畳み込み・グループ化・並び替えをしても会計から除去しない(P0)。据え置き(再導出をまたいで持続)警告を新規に真になった警告とともに含む可視ギャップ件数を提示し、会計算入が観測可能であるようにする。
- **（要件6）** 合意/状態起因の 3 警告を `moira-core` の projected 状態から確定する。R-U12=二人以上の人間が同一見積ノードに異なる現行凍結 agreed 値を持つとき矛盾合意警告を発し、値を揃える `agreed` 再発行で消す(自動解決はしない。latest-wins 値(I3)が現行値として残り、人間集合内部の権威裁定はスコープ外)。R-U13=見積 `proposed`(未合意)のまま完了状態(`implemented` または `accepted`)に到達したとき未合意完了警告を発す。P5 at-risk=`implemented` 到達後に後退し未再到達の間に at-risk 警告を発し、当該ノードが `implemented` へ再到達したときにのみ消す(後続自身の完了では消さない)。
- **（要件7）** `moira-schedule` の検出データからスケジュール起因の 3 警告を確定する。R-T3=schedule の検出データが過負荷(c=0 の日の AC を含む)を報告する間に過負荷警告を発し、報告がなくなれば消す(偽化の機序や発火窓は schedule 検出側が所有し health は定義しない)。R-T4=導出スケジュールが外部期日を超える間に超過量と行為列挙を伴う期日超過警告を発し、コミットで期日内に収まったときにのみ消す(超過の受容では消さない)。R-S7=生きた予測完了が凍結ベースライン・スロットと乖離する間にスロット陳腐化警告を発し、陳腐化スロット数を原因別(割当変更 vs c 変更)に提示し、再ベースラインか生きた予測の再収束でのみ消す(確認=意図的な非コミットでは消さない)。R-S7 は R-S6 とは独立した可視ギャップとして表出する。
- **（要件8）** `moira-evm`/`moira-scope-deps` の検出データから thrashing とキャンセル孤児を確定する。R-S3=EV_abs 非増かつ AC が継続的な期間にわたり増え続ける間に thrashing 警告を発す(畳んだ見積活動(R-E2b)の一度きりの cost には発しない)。R-C3=先行ノードが終端 `cancelled` で依存辺の充足閾値が永久に充足不能のとき、被ブロック後続と未充足辺を特定し行為列挙を伴うキャンセル孤児警告を発す(後続の自動キャンセルはしない。辺種別の限定や木 vs DAG の評価規則は `moira-scope-deps` から消費し health は再定義しない)。

## スコープ

**やること（health が所有 — 警告の確定・集約・clearance・行為列挙）**
- 9 警告(R-U12/R-U13/R-T3/R-T4/R-S3/R-S6/R-S7/R-C3 + P5 at-risk)の**確定・集約**(検出データの受け口)。各警告を**現在の導出状態に対する述語**として導出。
- 各警告の**clearance(消滅トリガー)の単一定義**: 条件偽化入力(4 イベント追記 または 構成入力 c/期日/目標日 変更)でのみ消滅し、acknowledge では消えない(§2.1 警告持続)。acknowledge/dismiss/seen 可変状態の不在(最小性)。
- 各警告の**行為列挙(取りうる行為)の単一定義**: 導出層に一度だけ定義し、下流は読むだけ。
- 判断型(8 件)の decision インボックス集約と、de-rate 型(R-S6)の常駐メトリクス修飾への分類。
- **可視ギャップ会計からの除去禁止**(P0 falsifiable な線)。

**やらないこと（上流/下流が所有）**
- 各警告の**検出述語/ロジック本体**: R-T3/R-T4/R-S6/R-S7 → `moira-schedule`、R-S3 → `moira-evm`、R-C3 依存辺評価 → `moira-scope-deps`、R-U12/R-U13/P5 の基礎状態 → `moira-core`(状態機械/projected)。
- emit/derive/二層データ/effective-set/latest-wins/状態機械/凍結記録 の**契約定義** → `moira-core`(本 spec は消費)。
- 指標式本体 → `moira-evm`、leveler/予測/buffer 導出 → `moira-schedule`。
- 警告の**提示**(レイアウト・畳み込み・淡色化・並び替え・deep-link) → surface spec。本 spec は提示の下限(何を surface すべきか・会計に残すこと)のみ。
- 警告→サーフェスの **host/ルーティング**(どのダッシュボードへ載せるか) → `moira-surface-decision`/各ダッシュボード。health は集約形(判断型 or de-rate 型)のみ定義。
- **seen 状態の区別・判定** → 提示層の責務(§2.1 顕著さの制御)。health は seen の真実源を持たず、条件が真の警告を会計に残す件数のみを語る。
- 警告解消の**書き(commit)** → write skill(reschedule/cancel-scope/estimate-agree 等)。検出=読 / 解消=書 の分離。
