# Requirements Document

## Introduction

`moira-surface-decision` は Moira 正典モデル `moira/MODEL.md`(v20) を本番アーキテクチャへ落とす **CQRS 分解の Wave3**（surface 群）であり、MODEL の中核思想「システムは観測・導出・警告に徹し、コミットメントを伴う判断は人間に残す」(§0/§2.1) のうち、**横断的に散らばる警告と人間のコミット判断を一望し、各々を解消する文脈ビュー（write）へ動線を引く**「横断・行為（decision）」面を所有する read=導出サーフェス spec である。具体的には次を所有する:

1. **判断・行為を要する警告の集約一望** — 上流 `moira-health` が確定・集約した判断型 8 警告（R-U12/R-U13/R-T3/R-T4/R-S3/R-S7/R-C3 および P5 at-risk）を、decision インボックスへ集約して一望する read 提示。de-rate 型の警告 R-S6（SPI de-rate）は集約せず health の常駐メトリクス修飾に委ねる（MODEL §2.1 の警告 9 件中、唯一の de-rate 型；UI-ARCH §4.2 注記＝inbox 集約 8 件＋R-S6 常駐＝計 9 件すべてに host）。なお R-S4（低カバレッジでの EV de-rate）は §2.1 の「警告」ではなく State 型の区別表示規則（UI-ARCH §4.3）であり、その host は spec-value（＋ health の EV%）＝本 surface の責務外であって、本 surface は集約も表出も行わない。
2. **コミット判断の集約（同一クエリの actor フィルタ）** — 5 コミット判断のうち 4 つ（見積合意・割当・スコープ/期日・見積の深さ）を、判断待ち＝同一 DAG×ログへの actor フィルタ違い（P4）として集約する。第 5（c 宣言）は capacity·calendar config 面の責務ゆえ集約対象外。
2b. **受入（検収待ち）区画と判断種別4区画の提示（2026-07-04 裁定・Req 7）** — インボックスは判断種別の 4 区画（見積合意・割当・受入・警告）で提示し、受入区画は上流 `humanReviewQueue` 導出（implemented の有効葉・actor 非依存）の**同一導出への別 read**＋受入 write への入口とする。これは 6 つ目のコミット判断を新設するものではない（五判断不変・承認は品質確認＝MODEL §2.1）。
3. **文脈ビュー（write）への deep-link ルーティング** — 各行から、当該警告/判断を解消する文脈ビューの write へ deep-link する（参照であって再計算でない・横断判断は複数 deep-link を許す）。各警告の詳細・次の行為・判断基準（取りうる行為列挙）は導出層（health）に一本化された単一定義を**読むだけ**で表示する。
4. **自前状態の不在と可視ギャップ会計の保持** — acknowledge/dismiss/seen の可変状態を持たず（§2.1 警告持続・最小性）、「新規」と「据え置き・既知」の区別表示はしてよい——ただしこの区別は**人間が見たか（seen）に依らず**、導出層が供給する**警告条件の真化年齢（直近 as-of 導出での初出か否か＝truth-onset）**という客観・ログ決定的な量で判定する（MODEL §2.1 が presentation の自由とする「据え置き・既知/新規」の区別を、seen 状態を持ち込まずに導出へ接地する）。いかなる提示も警告を可視ギャップの会計（件数・リスト）から除いてはならない（P0 falsifiable な線）。
5. **capacity heatmap は host せず deep-link のみ** — c(i,d) heatmap は decision 面で描かず（host は capacity·calendar config 面）、c 起因の判断（R-T3 過負荷／R-S7 の c 起因分）は capacity config 面（`moira-capacity` skill）への deep-link へ送る（UI-ARCH §5・roadmap.md:111 改訂）。c 宣言・heatmap の write/READ は capacity config 面が所有する。

本 spec は MODEL を唯一の真実源(SSOT)とし、MODEL の文言を変えず・新概念を足さず、その実装落とし込み（横断・行為面の集約提示・deep-link ルーティング・新規/据え置き区別・可視ギャップ会計の保持）に徹する。警告の**検出・確定・集約・clearance・行為列挙の単一定義**は上流 `moira-health` が所有し（roadmap Boundary Strategy「行為列挙の単一定義は moira-health に一度だけ・surface-decision は読む」）、本 spec はそれらと `moira-core` の derive 契約（R-S2 導出群・三キュー P4）を**消費**し、自前の真実源・可変状態を持たない。検出=読/解消=書の分離に従い、本 spec は警告を解消（write）しない。

## Boundary Context

- **In scope（surface-decision が所有）**: 判断型 8 警告の集約一望（health 確定結果の read 提示）、5 コミット判断のうち 4 つ（見積合意・割当・スコープ/期日・見積の深さ）の判断待ち集約（同一クエリの actor フィルタ＝P4）、各行から文脈ビュー write への deep-link ルーティング（複数 deep-link 可・参照のみ）、各警告の詳細・次の行為・判断基準（行為列挙の単一定義 by health）を読むだけで表示、「新規」と「据え置き・既知」の区別表示（seen 状態に依らず導出層供給の truth-onset 年齢で判定）と可視ギャップ会計の保持（P0・件数サマリ・空状態 N=0 を含む）、c 起因判断（R-T3／R-S7 の c 起因分）の capacity config 面への deep-link ルーティング（capacity heatmap は decision 面で host せず・描かない）、自前状態（dismiss/seen/snooze）の不在と条件偽化での自動消滅。
- **Out of scope（上流/下流が所有）**: 9 警告（MODEL §2.1 ＝判断型 8 ＋ de-rate 型 R-S6 の計 9；UI-ARCH §4.2 注記）の**検出述語/ロジック本体**= 各 derivation（`moira-evm`/`moira-schedule`/`moira-scope-deps`）、警告の**確定・集約・clearance（消滅トリガー）の単一定義・行為列挙（取りうる行為）の単一定義・判断型/de-rate 型の区別** = `moira-health`（本 spec は読む側）。de-rate 型警告 R-S6 の表出（常駐メトリクス修飾）= health。R-S4（§2.1 の警告ではなく区別表示規則 UI-ARCH §4.3：低カバレッジでの EV de-rate）の表出 = spec-value（＋ health の EV%）。全 write（見積合意 proposed→agreed・再見積・割当・スコープ/期日変更・再ベースライン・cancel・config）= moira-* write skill 群（reschedule/cancel-scope/estimate-agree/assign-schedule/project-config 等）。c 宣言の write・c(i,d) per-date 入力・理由付き改定・**capacity heatmap の host/READ 表示** = capacity config 面 / capacity write skill（本 spec は heatmap を host も READ 表示もせず、c 起因判断を当該面へ deep-link するのみ；UI-ARCH §5）。emit/derive・二層データ・R-S2 導出オーケストレータ・三キュー（P4）の導出・latest-wins・状態機械 = `moira-core`（本 spec は消費）。他軸の常駐ダッシュボード（spec-value/schedule-time/health）の提示 = 各 surface spec。
- **Adjacent expectations**: 本 spec は上流 `moira-health` が**定義**する警告の確定結果・集約形・行為列挙・clearance 条件と、`moira-core` の derive（R-S2・三キュー P4）を**消費**する。health の集約結果・行為列挙の形、core の derive 契約・キュー射影が変われば本 spec が再検証を要する。本 spec が引く deep-link の**着地先**（文脈ビューの write）は各 surface spec / write skill が所有し、本 spec はルーティングのみを担い着地先の責務を侵さない（横断・行為面はルーター、write は各文脈ビュー）。
  - **新規/据え置きの導出基底（health/core が供給する客観量を消費）**: 「新規 vs 据え置き・既知」を seen 状態に依らず接地するため、本 surface は各警告について導出層が供給する **truth-onset（その警告条件が現行 as-of 導出で真化した時点／初出か否か）** を読むことを期待する。これは health/core が R-S2 再評価（イベント追記・構成入力変更で導出を読み直す）から決定的に導く量であり、本 surface が seen-state を保持して算出するものではない。この量を health/core が供給しない場合、本 surface は M（据え置き件数）を seen-state で代替算出してはならず——その場合は Req 5 AC4 に従い件数サマリを「全 N 件（会計に全算入）」に留め、新旧の顕著さ区別は truth-onset 順ソート等の seen 不要 presentation に限る。
  - **ready の二語分離（roadmap 構造境界に準拠）**: 本 surface が inbox 行の表示・deep-link ラベルで触れうる lifecycle 状態（`ready` 等; progress emit）と ready-eligible 導出（scope-deps の R-D1/D2/D4 述語評価）を**二語として区別**し、両者を混同しない（roadmap doc-refine 確定「ready=lifecycle 状態＋ready-eligible 導出の二語分離」）。行タイトル・ラベルで `ready` 語を両義に使い「二つの真実」類似の語彙乖離を招かない。
  - **FORK 確定（ユーザー裁定 2026-06-22）— capacity heatmap は decision 面で host せず deep-link のみ**: 旧版は roadmap 初版の「capacity heatmap=READ」(host) と UI-ARCH/参照実装（heatmap は `CapacitySurface.tsx` 専属・`DecisionInboxSurface.tsx` に不在）の矛盾を未解決の FORK として開示していた。本矛盾はユーザー裁定で **選択肢 (B)** に確定した: decision 面は capacity heatmap を **host せず**、c 起因の判断（R-T3 過負荷／R-S7 の c 起因分）は **capacity·calendar config 面（`moira-capacity` skill）への deep-link** へ送る（roadmap.md:111 改訂「capacity heatmap は deep-link で capacity config 面へ・decision 面は host しない」・UI-ARCH §5 decision インボックス「capacity heatmap は host しない」・§4.1 注「heatmap 非 host」・参照実装の現状に一致）。capacity heatmap の host は capacity·calendar config 面（UI-ARCH §5）である。本確定に伴い Req 6 を「heatmap の READ host」から「c 起因判断の deep-link ルーティング（host しない・heatmap は描かない）」へ改訂する（FORK は確定済みであり、再 fork にはしない）。
  - **UI-ARCH §3 との差分の開示（commit 行の縮約）**: UI-ARCH §3 層B は inbox が「5 コミット判断のうち 4 つ（見積合意・割当・スコープ/期日・見積の深さ）」を集約すると記すが、本 spec はこの 4 つを文字どおり 4 つの commit 行とはせず、二重計上回避（本 spec 起源の整合規律・Req 2 AC1 注記）のため **スコープ/期日 を R-T4 警告レーンへ一本化**し、**見積の深さ を導出層が判断待ち集合を供給する場合に限る条件付き集約**へ縮約する（結果、commit レーンに常時立つ独立行は最大 2＝見積合意・割当）。これは UI-ARCH §3「4 つを集約」の文言を上書きするのではなく、その 4 つをレーン分け（commit 行 2＋警告行 1＋条件付き 1）として**解釈的に具体化**したものであり、両者が同一の表現でない事実をここに正直に開示する（一次資料 UI-ARCH を正としつつ、本 spec はそこからレーン割付で逸脱する）。
  - **受入区画の追加（2026-07-04 実画面裁定・issue #12・DECISIONS-CATALOG D-69）**: 旧版の本 spec と関連文書（moira-surface-schedule Req13 の旧 note・agreed 済み scenario unit 群）は「成果物の承認はインボックスに出ない」としていたが、issue #12（ユーザー起票）が検収待ちの集約を意図してインボックスを判断種別 4 区画（見積合意・割当・**受入**・警告）へ再構成し、E2E 同期（2026-07-04・実画面提示）でユーザーが**実装側を正**と裁定した。本 spec はこれを Req 7 として要件化する。整合の要点: (i) 受入区画は既存 `humanReviewQueue` 導出（actor 非依存・単一）の**同一導出への別 read**であって第二の計算ではない（P4・「二つの真実」禁止は不変）、(ii) **五判断の数は不変**（承認は品質確認＝MODEL §2.1・§7#18(e)）で、受入区画はコミット判断の行を追加しない（Req 2 AC1「ちょうど 2 つの独立 commit 行」は据え置き）、(iii) 差し戻しリスク（P5）は従来どおり警告レーン（Req 1）で扱い、受入区画と二重計上しない。
  - **commit レーンの判断待ち集合の導出ソース**: 見積の合意＝`moira-core` の現行有効葉（effectiveLeaves）のうち `proposed` の集合、割当＝`moira-core` の合意済み未割当 backlog（unassignedBacklog）を読む（いずれも P4 同一クエリの read 派生＝参照実装 `warnings.ts#computeInbox` の 2 commit 種に一致）。
    - **UI-ARCH §4.1 副 host の消化**: §4.1 被覆表は decision インボックスを「各キュー（P4：作業/レビュー/エージェント）」の副 host（判断待ち＝同一クエリの actor フィルタ・再計算なし）および「未割当バックログ」の副 host（参照のみ）として割り付けている（主 host はともに schedule-time）。本 spec ではこの両副 host は Req 2 の commit レーンに subsume される: 「見積の合意」「割当」の判断待ち集合は三キュー（作業/レビュー/エージェント）に対する actor フィルタ射影そのもの（Req 2 AC3 = 同一 DAG×ログへの actor フィルタ違いの read 派生）であり、未割当 backlog は割当判断待ち行の導出ソース（Req 2 AC1(b)）として参照のみで読む。したがって本 surface は §4.1 の二副 host を独立 host 列としてではなく commit レーンの導出ソースとして消化し（host は schedule-time、本 surface は read 派生の副提示）、空セルを残さない。見積の深さ（R-E2b）は警告も標準的な pending 集合も持たず、host は spec-value（UI-ARCH §5）であるため、本 surface は導出層が R-E2b の判断待ち集合を明示的に供給しない限り独立 inbox 行を立てない（roadmap の `moira-health` 責務は 9 警告＋行為列挙の単一定義であり「見積深さ pending 集合」を列挙していない）。
  - **参照実装 `warnings.ts` との差分（凍結時に固定）**: 参照実装 `moira/frontend/src/moira/warnings.ts#computeInbox` はすり合わせ済みの構造（read 派生フィルタ・自前状態なし・条件偽化で自動消滅）を体現するフォワード本番だが、自己完結プロトであるため警告述語を surface 側でローカル再検出（R-U12=`vals.size>1`、R-U13=`completed && proposed`、P5、R-C3 のエッジ走査）し、行為列挙（R-U13=['事後合意','再見積→合意','cancel']、R-C3=['辺除去','代替へ付替','後続 cancel']、P5=['後退 transition','再見積','依存付替']）と `clearWhen` 文字列を surface 側にハードコードしている。本 spec はこのローカル保持を採らず——Req 1 AC1（再検出・再確定の禁止）および Req 4 AC1/AC2（行為列挙の単一定義は `moira-health` を読むだけ・再定義/再実装の禁止）に従い——**述語の検出結果・行為列挙・clearance 条件の出所を `moira-health` の単一定義へ置換**する（参照実装の構造＝read 派生・自前状態なしは流用するが、述語/列挙/clearance の出所だけは health に移す）。
  - **per-node EVM 射影の許容（UI-ARCH §6・ユーザー裁定 2026-06-22）**: 本 surface が集約行に per-task の EVM 詳細を補助提示する場合、それは **per-node 属性（`frozenBudget`/`ac`）からの read-only な射影（projection）として許容**される——per-task PV/EV を属性から合成し、SV/CV をその差分の提示恒等式（`sv = EV − PV`／`cv = EV − AC`）とする二段（UI-ARCH §6(b)）。これは集約 derive() が一度だけ所有する指標（EV%/EV_abs/PV/AC/SPI/CPI/各カバレッジ）の**二系統計算ではない**——属性の表示用写像であり、UI-ARCH §6(a) が禁ずる「集約導出値の surface 再計算」に当たらない。線引きは参照実装 `surfaces/schedule/Inspector.tsx:44-56`（per-node 属性からの PV/EV 射影＋「SV/CV are presentation identities … not canon indices」）に接地し、Req 3 AC5 に landing する。集約指標を per-node から再合成すること（例: project SPI/CPI のローカル算出）は禁止であり、それらは health/spec-value の derive() 所有 host で読む。
    - **集約件数の差分（参照実装は 8 件中 4 件のみ集約・実装範囲外を明示）**: 参照実装 `computeInbox` が実際に集約する警告は 8 件中 4 件（R-U12・R-U13・P5・R-C3）のみであり、**R-T3・R-T4・R-S3・R-S7 は `computeInbox` に存在しない**（schedule/health 由来の述語が自己完結プロトに未移植のため）。さらに `DecisionInboxSurface.tsx:1` の冒頭コメントは「aggregates the 8 action-warnings」と自称しており、参照実装内部で件数（コメントの 8 vs 実体の 4）が乖離している。本 spec はこの乖離を採らず、**R-T3・R-T4・R-S3・R-S7 を含む 8 件すべて**を `moira-health` 集約の read として host する（Req 1 AC1・提示下限 R-S2／UI-ARCH §4.2 の 8 件集約）。これら 4 件は**参照実装がカバーしない範囲**であり、参照実装の述語構造をコピーするだけでは提示下限を満たさない——`moira-health` の集約結果を読む経路を本 spec が新設する。

## Requirements

### Requirement 1: 判断・行為を要する警告の集約一望（警告集約の出所＝UI-ARCHITECTURE §3/§4.2＋roadmap `moira-health` 集約；AC3 再評価契機＝R-S2；AC4 同一クエリ参照＝P4）

**Objective:** decision インボックス作者として、上流 health が確定した判断・行為を要する警告を一望する単一の集約面がほしい。それにより横断的に散らばる警告を、解消動線とともに一箇所で読める。

#### Acceptance Criteria

1. The system shall aggregate the eight judgement/action warnings (R-U12, R-U13, R-T3, R-T4, R-S3, R-S7, R-C3, and P5 at-risk) into the decision-inbox collection by reading the upstream `moira-health` aggregation, without re-detecting or re-determining them.
   - 和訳: システムは、判断・行為を要する 8 件の警告（R-U12・R-U13・R-T3・R-T4・R-S3・R-S7・R-C3 および P5 at-risk）を、上流 `moira-health` の集約を読み取って decision インボックス集合へ集約し、再検出・再確定してはならない。
2. The system shall not aggregate the de-rate-type warning (R-S6 SPI de-rate — the sole de-rate-type member of the MODEL §2.1 warning set) into the decision inbox, leaving it to be surfaced as a standing metric modifier on its host dashboard (health). R-S4 (low-coverage EV de-rate) is NOT a §2.1 warning but a State-type distinct-presentation rule (UI-ARCH §4.3) hosted by spec-value (and health's EV%); it is outside this surface's responsibility and is neither aggregated nor surfaced here.
   - 和訳: システムは、de-rate 型の警告（R-S6 SPI de-rate ＝ MODEL §2.1 警告集合中で唯一の de-rate 型）を decision インボックスへ集約してはならず、それは host ダッシュボード（health）の常駐メトリクス修飾として表出されるに委ねなければならない。R-S4（低カバレッジでの EV de-rate）は §2.1 の警告ではなく State 型の区別表示規則（UI-ARCH §4.3）であり、その host は spec-value（＋ health の EV%）＝本 surface の責務外であって、ここでは集約も表出もしない。
3. When any of the four events is appended, or when a configuration input (c, deadline, or target date) changes, the system shall re-read the upstream warning aggregation so the inbox reflects the re-evaluated derived state.
   - 和訳: 4 イベントのいずれかが追記されたとき、または構成入力（c・期日・目標日）が変化したとき、システムは上流の警告集約を読み直し、インボックスが再評価後の導出状態を反映するようにしなければならない。
4. The system shall derive the inbox collection by consuming the `moira-health` and `moira-core` derive output, holding no source of truth, no hidden mutable cache, and no precomputed derived projection of its own — it references the same query (P4) as the three dashboards and never runs a second computation pipeline.
   - 和訳: システムはインボックス集合を、`moira-health` と `moira-core` の derive 出力を消費して導出し、自前の真実源も隠れた可変キャッシュも事前計算した派生も持ってはならない——表示するいかなる値も 3 ダッシュボードと同一のクエリ（P4）を参照し、第二の計算パイプラインを走らせてはならない。

### Requirement 2: コミット判断の集約は同一クエリの actor フィルタ（P4・§2.1・UI-ARCHITECTURE §3）

**Objective:** 横断・行為面の読み手として、人間に残された 5 コミット判断のうち 4 つを判断待ちとして集約したい。それにより役割で物理分割せず、同一導出への actor フィルタとして判断を一望できる。

#### Acceptance Criteria

1. Of the five commitment-bearing decisions, the system shall aggregate **exactly two as independent commit rows** (estimate agreement, assignment), route one through the **warning lane** (scope/deadline as the R-T4 warning row), gate one on a derivation-supplied pending-set (estimation-depth), and exclude one (capacity-c, AC2) — and it shall not double-count any single decision across the warning lane (Req 1) and the commit lane: (a) **estimate agreement** is aggregated as a pending-commit row derived from effective leaves still `proposed`; (b) **assignment** is aggregated as a pending-commit row derived from the agreed-but-unassigned backlog; (c) the **scope/deadline-target call** surfaces through the **R-T4 warning row of Req 1**, NOT as a separate commit row — the same decision point shall be presented in exactly one lane (the warning lane), never both [double-count ban — see note]; (d) the **estimation-depth (R-E2b) call** is hosted by spec-value (UI-ARCH §5 "見積の深さ判断 R-E2b") and is reached via deep-link (Req 3), so this surface aggregates it only if the derivation layer (`moira-health`/`moira-core`) supplies a derivable pending-set for it — absent such a source it is not aggregated as a standalone inbox row.
   - 和訳: システムは、5 つのコミット判断のうち、**ちょうど 2 つを独立した commit 行として集約**し（見積の合意・割当）、1 つを警告レーンへ振り替え（スコープ/期日は R-T4 警告行）、1 つを導出供給される判断待ち集合の有無で条件付き集約（見積の深さ）、1 つを除外（容量 c、AC2）する——そして単一の判断を警告レーン（Req 1）と commit レーンで二重計上してはならない: (a) **見積の合意**は、現行有効葉のうち `proposed` のものから導出する判断待ち commit 行として集約する; (b) **割当**は、合意済み未割当 backlog から導出する判断待ち commit 行として集約する; (c) **スコープ/期日・目標日の決定**は **Req 1 の R-T4 警告行**として表出し、別個の commit 行にはしない——同一の意思決定点はちょうど一つのレーン（警告レーン）のみで提示し、両レーンに重複させない［二重計上禁止——注記参照］; (d) **見積の深さ（R-E2b）判断**の host は spec-value（UI-ARCH §5「見積の深さ判断 R-E2b」）であり deep-link（Req 3）で到達する。本 surface は導出層（`moira-health`/`moira-core`）が判断待ち集合を導出供給する場合に限り集約し、その導出ソースが無い場合は独立した inbox 行として集約しない。
   - 注記（二重計上禁止条項の出所＝本 spec 起源の整合規律）: 「同一の意思決定点はちょうど一つのレーンのみで提示し両レーンに重複させない」という制約は MODEL・UI-ARCH に直接の規定が無い（両文書に "exactly one lane"／"二重計上" に当たる規律は存在しない）。これは、§2.1#3 のスコープ/期日判断が Req 1 の R-T4 警告と同一の意思決定点である事実から導く**本 spec 起源の整合規律**であり、一次資料が支持する以上の規範性を持たせない。
2. The system shall not aggregate the fifth commitment-bearing decision — capacity-c declaration — in the decision inbox, since c declaration is hosted by the capacity·calendar config surface.
   - 和訳: システムは、5 つ目のコミット判断（容量 c の宣言）を decision インボックスへ集約してはならない（c 宣言は capacity·calendar config 面が host するため）。
3. The system shall derive the pending-decision queues as actor-filtered views of the same DAG and log query (P4), never as a separately recomputed projection; the actor filter is a projection-filter / initial preset over a single derivation, NOT a physical split by role (manager/developer) into separate screens or separate data sources — physically splitting by role and computing separately would produce "two truths" (UI-ARCH §2), which is forbidden.
   - 和訳: システムは、判断待ちキューを同一 DAG×ログのクエリに対する actor フィルタ違いのビュー（P4）として導出し、別系統で再計算した射影として導出してはならない。actor フィルタは同一導出への射影フィルタ／初期プリセットであって、役割（管理者/開発者）による物理分割・別系統データ源ではない——役割で物理分割して別系統計算すると同一導出が乖離する「二つの真実」（UI-ARCH §2）を招き、これを禁止する。
4. The system shall present commitment-bearing decisions as belonging to humans and shall not auto-resolve any of them, consistent with the meta-principle that the system observes, derives, and warns while the human commits.
   - 和訳: システムは、コミット判断を人間に属するものとして提示し、いずれも自動解決してはならない（システムは観測・導出・警告に徹し、人間がコミットするというメタ原理に整合する）。

### Requirement 3: 文脈ビュー（write）への deep-link ルーティングと共有部品・per-node 射影（UI-ARCHITECTURE §4.2/§6・検出=読/解消=書）

**Objective:** 判断・行為の起点に立つ人間として、各警告/判断の行から、それを解消する文脈ビューの write へ最短で着地したい。それにより横断面で発見し、各文脈で行為できる。

#### Acceptance Criteria

1. The system shall provide, on each aggregated warning and pending decision, a deep-link to the context view (write) that resolves it, the deep-link being a reference to the same query, not a recomputation.
   - 和訳: システムは、集約された各警告および判断待ちに、それを解消する文脈ビュー（write）への deep-link を提供し、deep-link は同一クエリへの参照であって再計算であってはならない。
2. Where a decision requires more than one context (e.g. R-T4 scope/deadline spanning health and schedule-time), the system shall allow multiple deep-links rather than collapsing it into a single jump.
   - 和訳: 判断が複数の文脈を要する場合（例: R-T4 のスコープ/期日が health と schedule-time にまたがる）、システムは単一ジャンプへ畳まず複数の deep-link を許可しなければならない。
3. The system shall not itself perform any write that resolves a warning or commitment decision, leaving every resolution to the write skill reached via the deep-link (detection-as-read / resolution-as-write separation).
   - 和訳: システムは、警告やコミット判断を解消する write を自ら一切行ってはならず、すべての解消を deep-link 先の write skill に委ねなければならない（検出=読/解消=書 の分離）。
4. The system shall reuse the shared theme components (`theme/atoms.tsx` の Card/Pill/SectionTitle・`theme/tokens.ts` の EVM トークン、必要に応じ `surfaces/schedule/gantt-geometry.ts`) for its presentation, and shall not re-implement presentation parts or tokens surface-locally in a way that diverges visually or in implementation from the other surfaces.
   - 和訳: システムは、提示にあたり共有テーマ部品（`theme/atoms.tsx` の Card/Pill/SectionTitle・`theme/tokens.ts` の EVM トークン、必要に応じ `surfaces/schedule/gantt-geometry.ts`）を再利用しなければならず、提示部品・トークンを surface 独自に再実装して他サーフェスと視覚・実装上乖離させてはならない（UI-ARCH §6 の共有・read 派生規律／参照実装＝フォワード本番への接地）。
5. Where the system presents per-task EVM detail on an aggregated row, it shall obtain it only as a read-only projection of per-node attributes — composing per-task PV/EV from per-node attributes (`frozenBudget`/`ac`) and SV/CV as presentation identities (`sv = EV − PV`/`cv = EV − AC`) — and shall NOT recompute any aggregate derive()-owned index (EV%/EV_abs/PV/AC/SPI/CPI/coverage); per-node attribute projection is allowed as read-only, but re-synthesizing an aggregate derive-owned metric surface-locally is forbidden (UI-ARCH §6 の二系統区別／参照実装 `surfaces/schedule/Inspector.tsx:55-56`「SV/CV are presentation identities … not canon indices」に接地)。
   - 和訳: システムは、集約行に per-task の EVM 詳細を提示する場合、それを per-node 属性の **read-only な射影（projection）としてのみ**得なければならない——per-node 属性（`frozenBudget`/`ac`）から per-task の PV/EV を合成し、SV/CV はその表示値の差分による提示恒等式（`sv = EV − PV`／`cv = EV − AC`）として——、いかなる集約 derive() 所有指標（EV%/EV_abs/PV/AC/SPI/CPI/各カバレッジ）も再計算してはならない。per-node 属性の射影は read-only として許容するが、集約 derive 所有指標を surface 独自に再合成することは禁止する（UI-ARCH §6 の「集約 derive 所有指標 vs per-node 属性射影」の線／参照実装 `surfaces/schedule/Inspector.tsx:55-56`「SV/CV are presentation identities … not canon indices」に接地）。

### Requirement 4: 詳細・次の行為・判断基準は導出層の単一定義を読むだけ（UI-ARCHITECTURE §6・行為列挙の単一定義）

**Objective:** 横断・行為面の作者として、各警告の取りうる行為・判断基準を導出層（health）から読むだけにしたい。それにより行為候補が文脈ビューと二重実装されず「二つの真実」を招かない。

#### Acceptance Criteria

1. The system shall display each warning's detail, next actions, and decision basis by reading the single action enumeration defined once in the derivation layer (`moira-health`), without re-defining it.
   - 和訳: システムは、各警告の詳細・次の行為・判断基準を、導出層（`moira-health`）に一度だけ定義された単一の行為列挙を読み取って表示し、これを再定義してはならない。
2. The system shall not re-implement any warning's action enumeration (e.g. R-T4's three options, R-U13's three options, R-C3's three actions); it consumes the enumeration as read-only derived data.
   - 和訳: システムは、いかなる警告の行為列挙（例: R-T4 の 3 択・R-U13 の 3 択・R-C3 の 3 行動）も再実装してはならず、列挙を読み出し専用の導出データとして消費しなければならない。
3. The system shall display, for each warning, the clearance condition that falsifies it, as read-only derived data sourced from the derivation layer.
   - 和訳: システムは各警告について、それを偽化する clearance 条件を、導出層を出所とする読み出し専用の導出データとして表示しなければならない。
4. When aggregating R-S7 (stale baseline slots), the system shall preserve and present the cause-attribution (assignment change vs. capacity-c-driven) that the derivation layer supplies, so the inbox row honors the cause-attributed presentation discipline of MODEL R-S7 and UI-ARCH §4.2/§4.3; it shall not collapse R-S7 into a cause-agnostic count. As a co-host of R-S7 (UI-ARCH §4.3 host = schedule-time＋inbox), this surface reads the cause attribute rather than recomputing it. The deep-link target shall match the cause-attributed context view of UI-ARCH §4.2/§5 (R-S7 文脈ビュー = schedule-time〔再ベースライン／据え置き受容〕＋config〔c 起因分〕): assignment-change rows deep-link to schedule-time (re-baseline), and **capacity-c-driven rows additionally deep-link to the capacity·calendar config surface (c 改定)** — so the cause attribution connects to its resolution动線 rather than presenting a cause-agnostic single jump (Req 3 AC2 multiple deep-links).
   - 和訳: システムは、R-S7（陳腐化スロット）を集約する際、導出層が供給する原因別属性（割当変更 vs. c 起因）を保って提示し、inbox 行が MODEL R-S7・UI-ARCH §4.2/§4.3 の原因別提示規律を引き継ぐようにしなければならず、原因を捨象した単一カウントへ畳んではならない。本 surface は R-S7 の co-host（UI-ARCH §4.3 host=schedule-time＋inbox）として、原因属性を再計算せず読み取る。deep-link 先は UI-ARCH §4.2/§5 の原因別文脈ビュー（R-S7 文脈ビュー＝schedule-time〔再ベースライン／据え置き受容〕＋config〔c 起因分〕）に一致させる: 割当変更起因の行は schedule-time（再ベースライン）へ、**c 起因の行は加えて capacity·calendar config 面（c 改定）へも deep-link** し（Req 3 AC2 の複数 deep-link）、原因別属性が原因を捨象した単一ジャンプに畳まれず解消動線へ接続するようにしなければならない。

### Requirement 5: acknowledge を持たず、可視ギャップ会計から警告を除かない（§2.1・P0・R-U9）

**Objective:** モデル設計者として、横断・行為面が acknowledge 可変状態を持たず、いかなる提示も警告を会計から落とさないことを保証したい。それにより true positive を隠す dismiss/snooze の反パターン（§0/P0）を構造的に排除できる。

#### Acceptance Criteria

1. The system shall not maintain any acknowledge, dismiss, snooze, or seen mutable state, presenting a warning iff its current condition holds.
   - 和訳: システムは、acknowledge・dismiss・snooze・既読のいかなる可変状態も保持してはならず、現在条件が真である場合に限り警告を提示しなければならない。
2. While a warning's condition remains true, the system shall keep it present even when a human has seen and accepted it without emitting any event (a legitimate non-commitment), and shall remove it only when an event or configuration-input change falsifies the condition.
   - 和訳: 警告の条件が真である間、システムは、人間がそれを見て受容しイベントを起こさない（正当な非コミット）場合でも警告を残し、イベントまたは構成入力の変更が条件を偽化したときに**のみ**除去しなければならない。
3. While presentation de-emphasizes, collapses, groups, or re-sorts warnings (e.g. distinguishing "standing/known" from "new"), the system shall keep every still-true warning in the visible-gap accounting (the count and list).
   - 和訳: 提示が警告を淡色化・畳み込み・グループ化・並び替えしている間（例: 「据え置き・既知」と「新規」の区別）も、システムは依然真であるすべての警告を可視ギャップの会計（件数・リスト）に残さなければならない。
4. The system shall present a visible-gap count summary that shows the breakdown "全N件・うち据え置きM件" (total N, of which standing M), where the new-vs-standing split is derived **without any seen-state**: a warning is "new" iff the derivation layer's truth-onset marks it as first-true in the current as-of derivation, and "standing" otherwise (M = N − newly-true) — the system shall NOT introduce any seen/acknowledge state to compute M (consistent with AC1). When the derivation layer does not supply a truth-onset, the system shall present "全N件" only and shall not fabricate M from seen-state. The summary shall be presented even when N=0 (showing "全0件"), and its non-display shall never be used to drop the observability of accounting inclusion — so that "no gap" is distinguishable from "presentation dropped the gap" (UI-ARCH §5 decision インボックス「件数サマリ＝全N件・うち据え置きM件 で会計算入を可視化」, P0).
   - 和訳: システムは、「全N件・うち据え置きM件」の内訳を示す可視ギャップ件数サマリを提示し、**新規/据え置きの区別はいかなる seen 状態にも依らず導出**しなければならない: ある警告は、導出層の truth-onset が「現行 as-of 導出で初めて真化した」と標すとき「新規」、そうでなければ「据え置き」とし（M = N − 新真化件）、M の算出のために seen/acknowledge 状態を導入してはならない（AC1 と整合）。導出層が truth-onset を供給しない場合、システムは「全N件」のみを提示し、M を seen 状態から捏造してはならない。件数サマリは N=0 のときも提示し（「全0件」を示す）、その非表示によって会計算入の観測可能性を損なってはならない——「ギャップが無い」ことと「提示がギャップを落とした」ことが区別可能であるように（UI-ARCH §5 decision インボックス「件数サマリ＝全N件・うち据え置きM件 で会計算入を可視化」・P0）。

### Requirement 6: capacity heatmap は host せず c 起因判断を deep-link でルーティングする（A4 二層データ・UI-ARCH §5・roadmap `moira-surface-decision` 責務・FORK 確定 2026-06-22）

**Objective:** 横断・行為面の読み手として、c 起因の判断（R-T3 過負荷・R-S7 c 起因スロット陳腐化）を、capacity heatmap を decision 面で自前に描かず、それを host する capacity·calendar config 面へ deep-link で送りたい。それにより heatmap の host・c 宣言の write をともに config 面に残し、decision 面を read-only の集約・ルーターに保てる。

#### Acceptance Criteria

1. The system shall NOT host or render the capacity c(i,d) heatmap on the decision surface — the heatmap's host is the capacity·calendar config surface (UI-ARCH §5) — and shall not write or revise c; the c-declaration/per-date input/reasoned-revision write and the heatmap host/READ display both remain with the capacity·calendar config surface, so the decision surface duplicates neither that surface's write nor its heatmap host responsibility.
   - 和訳: システムは、容量 c(i,d) の heatmap を decision 面で host も描画もしてはならず——heatmap の host は capacity·calendar config 面（UI-ARCH §5）である——c の write・改定も行ってはならない。c 宣言・per-date 入力・理由付き改定の write、および heatmap の host/READ 表示は、いずれも capacity·calendar config 面に残り、decision 面は当該面の write 責務も heatmap host 責務も二重化しない。
2. The system shall route any capacity-c-driven judgement (R-T3 over-allocation, and the capacity-c-driven portion of R-S7 stale slots) via deep-link to the capacity·calendar config surface (`moira-capacity` skill), which hosts both the c-declaration write and the heatmap; the deep-link is a reference (not a recomputation, consistent with Req 3 AC1), and any c declaration or revision action is reached only through this deep-link.
   - 和訳: システムは、c 起因の判断（R-T3 過負荷、および R-S7 陳腐化スロットの c 起因分）を、c 宣言の write と heatmap をともに host する capacity·calendar config 面（`moira-capacity` skill）へ deep-link でルーティングしなければならない。deep-link は参照であって（再計算でない・Req 3 AC1 と整合）、c の宣言・改定の行為はこの deep-link を介してのみ到達する。

### Requirement 7: 受入（検収待ち）区画と判断種別4区画の提示（2026-07-04 実画面裁定・issue #12・DECISIONS-CATALOG D-69；humanReviewQueue の同一導出 read・五判断不変）

**Objective:** 横断・行為面の読み手として、検収待ち（人間レビュー待ち）を判断種別の区画として一望し、受入操作の入口へ最短で到達したい。それにより人間ゲート待ちのボトルネックが見え、かつコミット判断の数は増えない。

#### Acceptance Criteria

1. The system shall present the inbox grouped into exactly four decision-type sections, ordered top-down as humans clear them: estimate agreement (見積に合意する), assignment (担当を割り当てる), acceptance (受入判断する), warnings (警告に対処する).
   - 和訳: システムは、インボックスを判断種別のちょうど 4 つの区画——「見積に合意する」「担当を割り当てる」「受入判断する」「警告に対処する」——に分け、人が上から片付ける順に提示しなければならない。
2. The system shall populate the acceptance section by reading the upstream human-review-queue derivation (implemented effective leaves, actor-independent) as **another read of the same single derivation** — never a second computation pipeline (P4, consistent with Req 1 AC4) — and each acceptance item shall deep-link to the write entry that resolves it (accept / return), consistent with Req 3.
   - 和訳: システムは、受入区画を上流の人間レビュー待ち導出（implemented の有効葉・actor 非依存）の**同一導出への別 read** として構成しなければならず、第二の計算パイプラインを走らせてはならない（P4・Req 1 AC4 と整合）。各受入項目は、それを解消する write 入口（受入／差し戻し）へ deep-link する（Req 3 と整合）。
3. The acceptance section shall NOT constitute a sixth commitment-bearing decision: the five-decision count is unchanged and accepting remains a quality check (MODEL §2.1, §7#18(e)); the section presents the standing awaiting-acceptance condition — an item is present iff its node is implemented (awaiting acceptance) and disappears only when acceptance or return falsifies the condition (consistent with Req 5 AC1/AC2). Req 2 AC1's "exactly two independent commit rows" is unaffected — acceptance items are queue reads, not commit rows.
   - 和訳: 受入区画は 6 つ目のコミット判断を構成してはならない: 五判断の数は不変であり、承認（受入）は品質確認のまま（MODEL §2.1・§7#18(e)）。この区画は「検収待ちである」という継続条件を提示する——項目はノードが implemented（検収待ち）である間に限り存在し、受入または差し戻しが条件を偽化したときにのみ消える（Req 5 AC1/AC2 と整合）。Req 2 AC1 の「ちょうど 2 つの独立 commit 行」は影響を受けない——受入項目はキューの read であって commit 行ではない。

The system shall provide an assignee-based display filter (all / mine / unassigned / by-assignee) over the inbox as a display refinement only: the standing total count shall keep accounting for ALL items regardless of the active filter (P0 accounting; consistent with Req 5 AC3), item matching shall use the node's assignee OR reviewer (acceptance is the reviewer's job), and the "mine" preset shall be enabled only when a viewpoint actor is supplied at launch — the system shall not hold or require an authenticated identity (MODEL §7#18(f) と整合・視点は提示層入力).
   - 和訳: システムは、インボックスに担当者ベースの表示フィルタ（全員／自分／未割当／担当者指定）を**表示上の絞り込みとしてのみ**提供しなければならない: 総数の会計は有効なフィルタに依らず全項目を算入し続け（P0 会計・Req 5 AC3 と整合）、項目の一致はノードの担当（assignee）または レビュー担当（reviewer）で判定し（受入はレビュー担当の仕事）、「自分」プリセットは視点 actor が起動時に供給された場合に限り有効とする——システムは認証済みの本人性を保持・要求してはならない（MODEL §7#18(f) と整合・視点は提示層入力）。
5. When a return-risk signal (P5 at-risk) holds for a node, the system shall surface it in the warnings section (Req 1) and shall not duplicate it as an acceptance item — a returned (regressed) node is not awaiting acceptance and shall not appear in the acceptance section while regressed.
   - 和訳: あるノードに差し戻しリスク信号（P5 at-risk）が立っている間、システムはそれを警告区画（Req 1）で表出し、受入項目として二重計上してはならない——差し戻された（後退した）ノードは検収待ちではなく、後退中は受入区画に現れてはならない。
