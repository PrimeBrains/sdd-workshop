# ADR-0005: 複数プロジェクトは読み取り専用の並置で束ねる（ポートフォリオビュー・会計の合成なし）

> **status: proposed** — doc-refine 独立敵対ゲートは **PASS**（2026-07-05・敵対者3体〔Opus〕＋
> fact-checker NO_OBJECTION＋独立採点者、生存 Critical/Important = 0・FORK 4件ユーザー裁定済）。
> **人間批准は保留中**（ゲート通過は批准の代わりにならない）。批准で accepted へ。経緯は issue #23。

## Context

issue #23（issue #15 適用ギャップ項目52）: 部門で複数案件を束ねる PM/PMO が、案件横断の
状況把握（各案件の EV/SPI/CPI・バッファ残・レビュー待ち）や、同じ人が複数案件に出ている
ときの掛け持ちを見たい。

正典側の制約: moira は「1プロジェクト＝1つの記録の家（home）」であり、
- **D-50**: 複数プロジェクト掛け持ちの容量合算はスコープ外（単一プロジェクト視点）
- **MODEL §5「意図的に持たないもの」**（現行 v20 では L433 付近）: 複数プロジェクトにわたる
  **Σc（容量）** の整合はモデルのスコープ外（組織レベル管理の責務 = A4）。
  ※この条項の射程は容量整合であり、EV 会計の横断合成の可否はここからは出ない
  （非合成は Decision 3 の本 ADR 判断）
- **ADR-0003**: multi-repo ≠ multi-project — home 解決はどの経路でも 1 つしか返さない

つまり「束ねて見たい」という要求と「単一プロジェクト視点」という正典の境界を、
**境界を壊さずに**両立させる必要があった。エンジン（4イベント・fold・derive）と MODEL は変更不要。

## Decision

ポートフォリオを**提示層の読み取り並置**として実装する（ADR-0004 と同じ
「コアに触れない拡張」パターン）:

1. **宣言的な portfolio.json**（schemaVersion 1）が N 個の home を名指しする。
   各エントリの `path` は**ディレクトリ**を指す — その直下の `.moira` がディレクトリなら
   home root そのもの、通常ファイル（ポインタ）なら 1 ホップ辿る（ポインタファイルの
   パス自体を書くのではない）。`moira ui --portfolio <file>` が配信の入口。
2. **各 home は 1 つずつ独立に解決・fold・derive** する。エントリごとに home.ts の
   `resolveExplicit` を 1 回呼ぶだけで、ログのマージ・イベント合流・複数 home を返す
   解決 API は存在しない（D-50/ADR-0003 の「1解決=1home」不変）。重複 home の検出は
   **realpath 正規化キー**で行う（symlink 別名・大文字小文字違い（win32 等の
   case-insensitive FS）でも同一 home の二重計上を拒否する）。
3. **横断で合算するのは件数のみ**（レビュー待ち件数・構造エラー数。読めた案件のみの和で、
   その旨を表示に明記）。**EV%/SPI/CPI の横断合成値は出さない** — これは正典が禁じる
   計算だからではない: 単位はアテンション時間（MD）で全案件共通（MODEL A6 単一通貨）
   であり、Σ EV/Σ PV は次元的には計算可能。それでも出さないのは本 ADR の**新規の設計
   判断**である — (i) 集約は案件間の分散（危機案件）を平均の中に隠す、(ii) D-50 の
   単一プロジェクト会計境界を提示層でも保守的に保つ、(iii) 合成の「見せ方」（対象選択・
   ゼロ PV 期の扱い等）が調整点として増殖し、「導出を動かす裁量ノブを持たない」原則
   （MODEL L47・§7#11 の工数軸バッファ不採用と同型の判断）の提示層版を破りかねない。
   将来 Σ 提示を採る場合は本 ADR の supersede/改訂として明示的に行う（再訪可）。
   なお件数の合算も**ノード分解の粒度（案件ごとの人間裁量・D-38）に依存**し案件間比較
   には向かない — この caveat を表示に常設する。
4. **人横断ビューは Actor.id の文字列一致のみ**で束ねる（グローバル人物台帳は持たない）。
   同日複数案件マーカーは「見える化」であって整合違反の判定ではない — 実体は
   **完了予定日・凍結スロットの同日一致**という近接プロキシであり（作業区間の重なり
   ではない。開始日は導出であって記録されないため区間は測れない）、**容量 c を一切
   参照しない**（c=0.3+0.3 でも点灯する）。よって視覚コードは**中立色**とし、判定色
   （赤）を使わない。この意味論は UI に開示する。容量整合の強制・平準化はせず、開示
   文言を常設する。宣言容量の合計は明示宣言分のみ（未宣言日を 1.0 と見なして合算しない）。
5. **home 単位の読込失敗はエラー行として可視**（ゼロの捏造禁止）。起動失敗となるのは
   (a) portfolio.json 自体の不備（読取不能・JSON 不正・スキーマ検証エラー・home 重複 —
   これらは設定ファイルのバグなので fail fast）と (b) **読める home がゼロ**のとき。
   個々の home の読込失敗では起動は継続しエラー行になる。なお CLI 読込は成功したが
   frontend の導出が例外を出した案件も同様にエラー行となる（ゼロ表示に化けない）。
6. **統一 asOf**（`--asOf` > 読めた home の config.asOf の最大 > 今日）で全案件を導出する
   （比較可能性）。この既定の帰結を正直に開示する: (a) max 以外の**全案件**が自分の
   config.asOf と異なる基準日で導出される（外れ値 1 件が全体を引きずる）。(b)
   config.asOf を持たない home は単独では today 基準だが、ポートフォリオでは他案件の
   max に引きずられる（max < today なら**過去方向**にも丸まる）。(c) max は**読めた
   home の宣言分のみ**から取るため、最遅 asOf の home が一時的に読めないと統一 asOf
   が変わり全案件が別の基準日で再導出される（読込成否で揺れる）。UI の基準日入力で
   いつでも変更できる。
7. **ドリルダウンは既存の単一案件ダッシュボードの再マウント**（読み取り専用のまま・
   ポートフォリオの統一 asOf で開く）。並置・ドリルの数値は「**同じ asOf で**その案件を
   単独導出した値」と完全一致する（INV-2 golden で機械固定 — `portfolio-derive.ts` 経由
   と直接 `derive()` の deep-equal）。**単独ダッシュボードの既定（各 home 自身の
   config.asOf）とは基準日が異なり得るため、その意味での無条件一致ではない**（#6 の
   開示と対）。この限定は UI・利用者向け文書にも明記する。

## Consequences

- エンジン（`moira/backend`）と MODEL.md は無改修（実装コミットの diff ゼロ）。
- **ADR-0003 との関係（射程の明示・0003 は不改変）**: ADR-0003 の「複数 home のマージ・
  集計 API は存在しない」は**解決系（resolver）とログ層**についての言明であり、それは
  本 ADR 後も真（resolveExplicit は 1 home を返し、イベントログは決してマージされない）。
  本 ADR が加えた**提示層の件数合算**（読めた案件のレビュー待ち・構造エラーの件数の和）
  はこの言明の射程外で、これに反しない。字義の衝突を避けるための言明の narrow-scope は
  0003 側でなく本 ADR のこの節が担う（accepted 文書の事後改変を避けるユーザー裁定）。
- 機械ガードの変更は**二方向**を正直に区別する:
  - **締め**: depcruise `surfaces-read-via-hooks-not-store` の禁止対象に portfolio-store を追加
    （surface からの直 import を新たに禁止）。
  - **白リスト拡張（緩和側）**: fidelity gate の derive import 許可リストが 2→3 モジュール
    （`portfolio-derive.ts` 追加）。監視点が 1 つ増えた=許可点が 1 つ開いた、であり
    「ガード強化」ではない。exact-match のため 4 件目の import は依然トリップする。
- 提示層の窓パラメータ: 人横断ビューの対象期間は**14 日固定**（`WINDOW_DAYS`）。導出には
  一切影響しない提示の選択（P0）だが、ノブであることに変わりはないため UI と本 ADR の
  両方で開示する（値の変更・可変化は提示層の自由）。
- 正直さの負債として開示するもの:
  - レビュー待ちの**最古経過日数**は未実装（「人間ゲート待ち」計器の導入時に追加予定・画面に注記済み）
  - 統一 asOf の帰結（Decision 6 の (a)(b)(c) — max 以外の全案件が自前 config.asOf と別基準日・
    no-asOf home の巻き込み・読込成否での揺れ）
  - portfolio.json の home 追加・削除の fs.watch 反映は再起動が必要（ブラウザ再読込では反映される）。
    **起動時に読めなかった home はその後修復されても watch されない**（fs.watch は起動時に
    読めた home のみ・ブラウザ再読込では反映される）
  - 「INV-2」は正典/PROPERTIES に台帳を持たないテストローカルの呼称（定義は
    `portfolio-derive.ts` ヘッダと `portfolio.golden.test.ts`。内容は「同一 asOf での導出
    パリティ」— Decision 7 の限定と同じ）
- シナリオ E2E（計器③/④）は人間の When/Then 種待ち（kiro-scenario → kiro-scenario-e2e）。
