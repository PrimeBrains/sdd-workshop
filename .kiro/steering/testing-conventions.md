# Testing Conventions

テスト戦略の learning 集約ファイル。新しいテスト関連の Try が出たら、このファイルに H2 セクションで append する（1 トピック 1 ファイルではなく、テスト戦略カテゴリで束ねる）。
（旧プロトタイプ由来の具体事例は postmortem リセットに伴い除去済み。汎用原則のみを残す。）

---

## 「自明」と判断したら疑う (assumption-error 対策)

「自明な変換」「ROI が低いから省略」と判断した箇所こそ暗黙仮定が積まれている。

### Rule

1. **単位を持つ数値の変換・整形関数**（MD ／時間／比率／バイト等）は最低 1 ケースの boundary 値テスト必須（例: `70 → "70.0 MD"`）。Moira では MD（man-day = アテンション時間）が単一通貨であり、単位の取り違えは全表示を壊す。
2. **設計書・PR の「自明 / 不要 / 変わらない」記述を見たら反例を 1 つ挙げる。**
3. **要件文書に数式・magic number を埋め込まない**（実装詳細は design 以下に置く）。

---

## データフロー結合テストを書く (integration-test-coverage)

「複数プロセス × 複数ファイル × データフロー」を跨ぐ結合は、プロセス内 fixture（インメモリ・同一プロセス seed）で隠蔽されやすい。検証層を**「ファイル単位」ではなく「データフロー単位」**で考える。

### Rule

1. **プロセス内 fixture の罠を意識する**: seed と runtime を同一プロセスに閉じ込めると、プロセスを跨ぐ不整合が見えなくなる。
2. **入力 → 導出 → read 口を跨ぐデータフロー結合テストを書く**: Moira なら「イベントログ → fold → derive → snapshot CLI／read API」のフロー全体を 1 つの自動テストで担保する（参照実装 `moira/backend` の golden／rederive テストが該当）。
3. **e2e を整合性検証の代替にしない**: integration 層で先に潰せる問題は integration で潰す。

---

## 偽 pass を防ぐ (preventing-false-pass)

「テキストが存在する」「要素が見える」レベルの緩いアサートは、データが描画・導出されていない世界でも pass する。

### Rule

1. **厳密値アサート**: 具体値を突き合わせる（`toBe(expected)` 等）。「存在する／見える」だけのアサートは「データ空でも pass する」と疑う。
2. **動的期待値**: 期待値を API レスポンス／導出結果から取得して突き合わせる（手書き定数に固定しない）。
3. **偽 pass 検出**: 新規テストを書いたら「データなし／異常データ」状態で 1 回流して fail することを確認してから本シナリオに進む。
4. **前提条件の境界アサート**: 本体テスト前に「データが正常に得られている」ことを 1 行で確認する。
