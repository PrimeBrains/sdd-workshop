# Moira 正典モデルの制約チェックリスト

本ファイルは `moira/MODEL.md`（唯一の真実源）の**順守チェックリストと章ポインタ**であり、モデルの再導出（証明・本文）ではない。各 spec／design は下記を満たし、詳細は MODEL の該当章を引く。矛盾時は MODEL が勝つ。指標の**定義**は [moira-naming.md](moira-naming.md) が所有する。

## 公理（A1–A6, `MODEL.md §1`）

- **A1** ノードのみが実体。チケットは射影（R-U1）。
- **A2** 唯一のデータ = 4イベント追記専用ログ。保存状態への直接 mutate は禁止（R-U2）。
- **A3** ノードは見積を持ち状態機械上にある。木（価値集約）と DAG（時間制約・辺ごとポリシー）が重なる。
- **A4** 単一希少資源は人間。各人間は日次容量 c(i,d) を持つ（c・α_i の定義は [moira-naming.md](moira-naming.md)）。
- **A5** 人間とエージェント: 表現は対称、資源は非対称（平準化は人間のみ）、権限は非対称（合意は人間のみ）。
- **A6** 単一通貨 = 人間アテンション時間（MD）。金銭は持たない。

> 裁量で導出を調整するパラメータは持たない（c は外的事実入力で例外）。`MODEL.md §1` 末尾注。

## 4イベントのみ（`MODEL.md §2.8`）

`transition`（ライフサイクル／見積合意／被割当者の名指し）・`decompose`（子と見積の設定・改訂、理由必須）・`relate`（DAG 辺の追加削除、種別=依存／置換 supersede + 充足ポリシー）・`cost`（実コスト加算、id で dedup）。**第5イベントを足さない**——割当も c 変更も既存の仕組みで表す（§5）。

## 不変条件（I1–I6, `MODEL.md §2.9`。設計が保つ）

- **I1** 親の最新見積 = Σ(合意済み子の最新見積)。未見積はカバレッジ低下として現れる（破れではない）。
- **I2** `relate` は循環を作れない（依存・置換の全種別; R-D3）。
- **I3** event id は大域ソート可能、`(ts,id)` で決定的マージ（R-D5）。
- **I4** 完了サブ単位のベースラインは施錠（確定済み次元のみ: 予算は合意済み、スロットはスケジュール済み）。
- **I5** すべての `transition` は対象状態機械を明示（R-D6）。
- **I6** `proposed→agreed` は人間のみ。矛盾は latest-wins(I3) + 警告(R-U12)、人間内部権威はスコープ外。

## 導出指標の規律（`MODEL.md §3`, P1/P2。定義は [moira-naming.md](moira-naming.md)）

- **EV_abs と EV% を区別表示する**（混同しない）。EV% に重みテーブルは使わない（R-U8）。
- **EV% は見積カバレッジ（P2）と、SPI はスケジュールカバレッジ（R-S6）と必ず対で読む。**
- ベースラインは**予算=合意時・スロット=初回スケジュール時**の二契機で凍結し、完了で施錠する（I4）。合意済み未スケジュール／未合意のノードは PV に入れず、可視ギャップとして示す（§3）。
- **SPI はスケジュール済み領域限定**であり全体進捗ではない。累積 EV_abs と現行進捗 EV%（現行有効集合）を区別する（R-S5）。

## 9 警告と持続／消滅意味論（`MODEL.md §2.1`, EARS は `§4`）

警告は**現在の導出状態に対する述語**。条件が真の間だけ成立し、**条件を偽化する入力（4イベントの追記 または c の変更）でのみ消える**。「見て受容するだけ（イベント無し）」では消えない。**acknowledge／dismiss／既読は持たない**（最小性）。提示は顕著さを操作してよいが、警告を**可視ギャップの会計から落としてはならない**（falsifiable な線）。

9 警告（一行カタログ。各 EARS は `§4` が正典）: **R-U12** 矛盾合意 / **R-U13** 未合意完了 / **R-T3** 過負荷 / **R-T4** 期日超過 / **R-S3** スラッシュ（thrashing）/ **R-S6** スケジュールカバレッジ・デレート / **R-S7** スロット陳腐化 / **R-C3** キャンセル孤児 / **P5** at-risk（implemented 到達後に後退し未再到達）。

## 五つの人間専任コミット判断

**5 つのコミット判断を自動化しない**（`MODEL.md §2.1`）。判断の一覧と詳細・各サブ引用は [product.md](product.md) が所有するので、ここでは再掲せずそれを指す。

## 変更時のゲート対応

- モデルの制約・語彙・既定を変える → `moira-model-update` スキル（独立敵対ゲートで `moira/MODEL.md` を確定）。
- 確定文書（steering・エージェント定義等）を磨く → `doc-refine` スキル。
