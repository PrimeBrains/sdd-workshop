# 技術方針: Moira

Moira は「思想（MODEL）の確定」が先にあり、実装技術はそれを侵さない範囲で選ぶ。本ファイルは**スタックに依存しないアーキ不変条件**を確定し、具体スタックの確定状況を記す。正典は `moira/MODEL.md`（迷ったら章 ID で参照。バージョン番号は steering に書かない）。モデル制約の一覧は [moira-model.md](moira-model.md)。

## アーキ不変条件（スタック非依存・厳守）

どの実装も次を満たさなければならない:

- **単一データ = 4イベント追記専用ログ。** ノードのワークデータ（進捗・状態・スコープ・コスト）の唯一の真実源は `transition`／`decompose`／`relate`／`cost` の追記専用ログ。保存状態への直接 mutate は禁止（A2 / R-U2）。
- **二層データ。** 人間の日次容量 c(i,d) は 4 イベントとは別の第二層で、自前の理由付き・追記専用・タイムスタンプ付き履歴を持つ（A4 / R-U14 / §5「二層データ」）。c はノード属性ではないため第5イベントを足さない。
- **観測・導出・警告に徹する。** 進捗・スケジュール・健全性は導出であり、保存された派生状態を真実源にしない。コミット判断（[product.md](product.md) の 5 判断）を自動化しない（§0 / §2.1）。
- **導出を調整する裁量パラメータを持たない。** 重み・留保率・ready 閾値・スキルは廃止。c だけは裁量ノブではなく外的事実入力（雇用契約・組織アサイン・暦）であり例外（`MODEL.md §1` 末尾の注 / A4）。
- **決定的マージ。** イベントは `(ts,id)` で大域ソート可能・決定的に畳む（I3 / R-D5）。同一ログ・同一実装なら再導出は再現する。
- **再導出契機はイベント追記と c 変更の双方。** c はイベントではないため、c の変更も P7 スケジュール再導出をトリガーする（R-S2）。
- **スケジュールは凍結スロット + 生きた予測の分離。** ベースライン（予算は合意時・スロットは初回スケジュールで凍結）と、増分再計算する生きた予測は別物。平準化は発見的・非最適（P7 / P8 / §3）。

## 具体スタックの確定状況

- **実在する参照実装（製品スタックとしては未確定）:** S4 の最小バックエンドが `moira/backend/` にある——**純 TypeScript（ESM）+ Vitest** の導出エンジン、追記専用ログ（メモリ／JSON）、薄い read 口（snapshot CLI が主、Hono の read-only HTTP は任意）。DB・UI を持たず、上記アーキ不変条件を最小コストで実証する。コードは `moira/MODEL.md` を仕様として直接参照する。
- **未決（TBD; ADR で確定する）:** 永続ストレージ、UI フレームワーク、デプロイ形態、S4 を超える全体アプリ構成。現状の縦スライスをそのまま製品スタックへ昇格させるかも含めて未決。決定が満たすべき基準: ①追記専用ログと決定的 `(ts,id)` 順序（I3）が自然に書けること、②ログからの再導出が再現可能であること、③ c の第二層を独立 tier として扱えること。決定は `moira/` ではなく ADR（[adr.md](adr.md)）に記録し、本ファイルは確定後にそれを指す。

## 言語・記述規約・ゲート

- 生成ドキュメント（requirements／design／tasks 等）の言語は **ja**。EARS は英文 + 和訳併記（[requirements-style.md](requirements-style.md)）。トレーサビリティ表記は [trace-notation.md](trace-notation.md)、テスト戦略は [testing-conventions.md](testing-conventions.md)。
- **正典・確定文書の更新はゲート経由**（どのゲートを使うかは [moira-model.md](moira-model.md)「変更時のゲート対応」が所有。`moira/MODEL.md` は `moira-model-update`、steering 等の確定文書は `doc-refine`）。
