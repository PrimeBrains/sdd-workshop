# Moira 検証・品質ゲート方針

本ファイルは Moira（`moira/MODEL.md` を正典とするシステム）を、**どの手法で・誰が・何をレビューして**開発・確定していくかの方針。制約の順守チェックリストは [moira-model.md](moira-model.md)、指標定義は [moira-naming.md](moira-naming.md)、テスト規律の learning は [testing-conventions.md](testing-conventions.md) が所有する。本ファイルはそれらを束ねる**検証アーキテクチャの方針**であり、矛盾時は MODEL が勝つ（現に `moira-naming.md` の見積カバレッジ P2 は旧ノード基底のまま陳腐化しており、MODEL v18 の葉基底が正——`moira-naming.md` の同期は `moira-model-update` で要フォローアップ）。

## なぜ（課題）

`requirements.md` は MODEL を忠実に参照するため抽象度が高く、非専門家による直接レビューが難しい。既存の敵対ループ（`moira-model-update` / `doc-refine`）が見るのは**散文の内的整合性**、受け入れシナリオ（`.kiro/scenarios/`）が見るのは**外的妥当性**。だが**システムの不変条件＝オラクルそのもの**は、これまで機械的検証を欠き AI 信頼に委ねられていた——参照実装 `moira/backend` のテストは長らく example ベースのユニット／golden が中心で、不変条件の**空間**を機械的に試すものが乏しかった。ここが最大の無防備面であり、本方針が埋める（現在は最小パイロットとして PBT が稼働し、PR-DONE-LOCK★ が実装乖離を RED で露出している＝後述「動機」）。

**承認ゲートは陳腐化したのではなく、レビューの焦点が移設された。** 人間の希少な注意を、最も低シグナルな成果物（MODEL から派生した散文 `requirements.md`/`design.md`/`tasks.md`）の逐語レビューから、より荷重を支える成果物——実行可能仕様（プロパティ・シナリオ）と設計判断（Decisions）——へ移す。Kiro 式 spec-driven development の哲学核（intent を build 前に人間が批准する・単一の真実源・トレーサビリティ）の**精神は保持**する——変えるのはその実現形（何をレビューするか）であって、人間が方向を所有する原則そのものではない。なおこれは正典 MODEL を先に持つ Moira 固有の特殊化であり、Kiro 式の一般的否定ではない——正典を持たないプロジェクトでは `requirements.md` こそが意図の初出＝真実源であり、そこでは散文ゲートが一次レビュー面である。本方針は「形式化可能なコアを持つ」という適用条件の下で成り立つ。

## 組織原理：プロダクトオーナーがレビューするのは3面

人間（プロダクトオーナー＝非専門家）が直接向き合うレビュー面は、次の**3面**とする。各面は別々の問いに答える（焦点が異なる）。

- **外的妥当性 → シナリオ**（作るべきか／ユーザーが気づけるか）。人間が所有（`.kiro/scenarios/README.md`）。
- **内的整合性 → プロパティ目録の「平易な一文」**。人間が批准する（`moira/PROPERTIES.md`）。
- **構造的健全性 → 設計判断（Decisions）の「平易な判定文」**。MODEL にもプロパティにもシナリオにも bind できない 0→1 の設計判断——CQRS 境界・所有権分配・`ready` 二語分離・永続化の所在・タスク分解の構造など（実例: `.kiro/specs/moira-core/requirements.md` の Boundary Context「core は構造のみ所有・値は下流」）。人間が批准する（`moira/DECISIONS-CATALOG.md`・D-1〜D-3 agreed＋D-4〜D-72 proposed。D-63 は v20 目録化・D-64〜D-72 は 2026-07-04 実装先行分の目録化）。**ただし第三面は仕分け弁を持つ**——後述のとおり (a) 責任・帰結・ドメイン意味の判断はプロダクトオーナーが見、(b) 純工学判断は技術ゲートへ分流する。よって「3面」はプロダクトオーナーが向き合う**入口**の数であり、第三面の (b) 成分は別主体（技術敵対ゲート）が担う。
- **残りは機械（自動検証・整備途上、下記「計器構成」節）と技術敵対ゲートが担う。** プロダクトオーナーがこの3面（の人間担当分）だけを見れば、意図どおりのシステムが立つ——を**設計目標**とする（達成済みの断定ではない）。
- 消えない**残余＝未発見（P2）**。いかなる手法の組合せも「ドメイン妥当性」を立証しない。これは**非ゴールとして明記**し、green が穴を塞いだという誤った安心を禁ずる。

第三の面（Decisions）はこれまで居場所がなく、`requirements.md` の散文に紛れて誰の批准も通らずに実装へ降りていた——本方針がその面を一級化して穴を塞ぐ。なおこれら3面が人間レビューの**すべて**だと主張するものではない（運用・性能・セキュリティ等、Moira のスコープ外や技術ゲート側に属する面は別途扱う）。

## 設計判断目録（Decisions）

- **場所**: `moira/DECISIONS-CATALOG.md`（人間レビュー用の判定目録。2026-06-26 実体化＝D-1〜D-3 agreed＋D-4〜D-62 は洗い出し proposed・要レビュー。その後 D-63〔v20 目録化〕・D-64〜D-72〔2026-07-04 実装先行分・issue #19 追いつき〕を proposed で追補）。`PROPERTIES.md` と並ぶ、人間がレビューする一次成果物。**既存の `moira/DECISIONS.md` は別物**——確定済み分岐を時系列に記録し `moira-model-update` が保守する**意思決定ジャーナル（来歴・専門家向け）**である。目録で `agreed` になった判断は、必要に応じてジャーナルへ来歴として記録される（目録＝人間レビューの入口、ジャーナル＝archive）。
- **列**: 名前／**どんな場面の話か**（専門用語ゼロの状況描写）／**決めたこと（案）**（採った選択肢と**誰の責任か**）／**他にありえた選び方**（却下した代替——これが無いと判子押しになる）／**外すとどうなるか**（観測できる被害＝賭け金）／**あなたが判定する一文**（`proposed → agreed` で批准する yes/no）／**仕分け (a)/(b)**（下記）／〔**裏面**・人間は読まなくてよい〕根拠 clause（**安定 A/I/P/R ID で参照・行番号禁止**）・関連実装 ref。
- **規律**:
  - シナリオ README・プロパティ目録と同等の**平易な言語規約**（専門用語・skill 実名を本文に出さない）。
  - 起票者（AI）が `proposed` で判定文と仕分け案を起こし、人間（プロダクトオーナー）が `agreed` で批准（Moira 自身の規律を設計判断にも自己適用）。批准前のハードニングは `doc-refine`——`moira/DECISIONS-CATALOG.md` は `moira/` 配下の確定文書ゆえ doc-refine の**汎用対象**であり、Decisions 専用モードを要しない（doc-refine が穴を出し、人間が `proposed → agreed` を下す）。
  - **仕分け規律（第三カテゴリの内訳）**: (a)**責任・帰結・ドメイン意味の判断**（「完了した出来高を誰が守るか」「`ready` は非実装ノードに当てはまるか」等、人間が平易な言葉で賛否を持てるもの）は**プロダクトオーナーがレビュー**する。(b)**純工学判断**（永続化方式・アルゴリズムの精密化・モジュール配置・タスク順序など、賛否に保存技術や実装知識を要するもの）は**プロダクトオーナーのレビュー対象にせず、技術敵対ゲート（`doc-refine`）へ分流**する。**(a)/(b) にまたがる判断は (a) 側（人間レビュー）を既定**とし、(b) 成分は裏面で技術ゲートへ回す——どちらにも割れないことを理由に判断を居場所なしに戻さない。
- **正典との関係**:
  - プロパティとシナリオが**既存正典（MODEL）の蒸留**であるのに対し、Decisions は **MODEL にまだ無い 0→1 判断を人間が批准できる形で起こす面**である（ただし 0→1 を正典化する唯一の経路ではない——シナリオ駆動 ESCALATE や `moira-model-update` の FORK 裁定も正典を起こす。Decisions の固有性は、それらを**人間レビュー可能な目録として一級化**する点にある）。ゆえに Decisions は本質的に疎（0→1・網羅不能）であり、プロパティ目録のような **clause→項目の被覆レポートは持たない**（全設計判断を機械的に列挙できないため。被覆ではなく「起きた判断を漏れなく目録化する」規律で担保する）。
  - `agreed` Decision が **MODEL の制約・語彙・既定に触れる**なら `moira-model-update` で MODEL／steering へ昇格させる（触れなければ目録層に留まる）。**MODEL が当該論点に沈黙する場合は「衝突」ではなく「拡張」であり、Decision は中間状態として有効**（昇格で正典化）。「矛盾時は MODEL が勝つ」（本書冒頭）が適用されるのは MODEL が**積極的に矛盾する**ときのみで、その場合の Decision は**未昇格＝未確定**であり、昇格手続きを経て初めて正典になる。いずれにせよ Decisions が MODEL を直接上書きすることはない。

## R/D/T は使い捨ての再生成物（2026-07-19 改訂・issue #40）

> 旧方針（「R/D/T は維持する派生 IR——①一致チェック＋②技術敵対ゲートの整備を条件に逐語レビューを
> 段階的に縮退」）は 2026-07-19 に人間批准のうえ**廃止**した。改訂理由と撤回条件は本節末尾。

- `requirements.md`／`design.md`／`tasks.md` は MODEL 正典の実装向け射影＝**中間表現（IR）**である——
  そして本物の IR がそうであるように、**コミットして手で同期し続ける対象ではなく、必要時にソース
  （正典）から再生成する使い捨てのビルド成果物**として扱う。コンパイラは IR をリポジトリに置いて
  保守しない。毎ビルド、ソースから作り直す。
- **永続する正典は人間が書いた・批准したものだけ**: `moira/MODEL.md`・シナリオ（When/Then は人間
  発案・`.kiro/scenarios/`）・プロパティ（`moira/PROPERTIES.md`）・Decisions
  （`moira/DECISIONS-CATALOG.md`）。プロダクトオーナーのレビュー面は上の3面のまま変わらない。
- **運用**:
  - 実装サイクルの開始時に、必要なら R/D/T を正典（MODEL・agreed シナリオ・agreed Decisions・
    プロパティ）から**その場で再生成**する。再生成は **R→D→T の整合した組**として行うのが原則——
    tasks.md 単独の再生成は、著者が既存 R/D の現行正典との整合を確認・記録した場合のみ許す（新しい
    T を古い R/D と組み合わせて、使い捨て化が消したはずのファイル間ドリフトを再生産しない）。
    再生成物のヘッダには生成日と基準（正典の版・base commit）を記す——アーカイブと現役を見分ける印。
  - サイクル内では従来どおり運用する（`/kiro-impl` の tasks.md チェックボックス・Implementation
    Notes 等）。「使い捨て」とは**サイクル終了後に正典として維持・同期しない**という意味であり、
    サイクル内の作業台帳としての持続を否定しない。
  - 既存の `.kiro/specs/moira-*` の R/D/T は**生成時点のアーカイブ**であり正典性を持たない。参照する
    場合は「当時の射影」として扱う（正典と食い違えば正典が正）。
  - **R/D/T を読む既存ツールの開示（「消えた」と言い切らないための正直枠）**: `moira` CLI の cc-sdd
    provider は tasks.md を進捗照合に読む——**現役サイクルの tasks.md** を読む分には本方針と両立し、
    アーカイブに向ければ stale を表示しうる（provider の対象は現役サイクルに限る運用）。`spec-model`／
    sdd-dashboard は R/D/T をダッシュボード表示に読む——アーカイブの表示は「当時の射影」であり現況を
    主張しない（表示側での明示は follow-up）。`kiro-validate-impl` 等の検証プロンプトも R/D/T を入力に
    取る——サイクル内文書に対して走る限り両立する。いずれも「ドリフト面が消えた」の例外ではなく、
    **アーカイブを現役と誤読させない運用**が付帯条件であることをここに明記する。
  - 適用条件は従前どおり: 本節は **MODEL を SSOT として宣言する spec（moira-* 系）のみ**に適用され、
    正典を持たない spec には `CLAUDE.md` の3フェーズ承認（散文ゲート）が引き続き適用される。
- **旧方針の何が消え、何が残るか**:
  - 消えるもの: ①正典↔R/D/T 一致チェック（同期対象が存在しなくなる）・R/D/T 逐語レビューの
    「段階的移行」と移行期人間レビュー・R/D/T を対象とする doc-refine 常設ゲート。ドリフト問題は
    解決されたのではなく**消滅**する——持続しない文書はドリフトしない（動機節の R-E3/I4 は「維持して
    いた IR が黙ってズレた」事故であり、本方針はその発生面自体を除去する）。
  - 残るもの: **計器群は全て残る**。第1〜4器・⑥AI整合性チェック（`decision-conformance`）・
    clause→property 被覆が検査するのは**実装↔正典**（および Decision↔実装）であって R/D/T では
    ない——これらの役割は本改訂で変わらない。変わるのは「R/D/T 縮退の前提条件」という追加の役割が
    外れることだけ。
- **正直枠**: 再生成物の忠実性は生成時点のモデル能力に依存し、それ自体を検査する専用ゲートは持たない
  （持てば維持コストが戻ってくる）。実装の正しさは既存計器（PBT・E2E・アーキ適合・codex レビュー・
  CI）が検査し、正典との整合はシナリオ／プロパティ／Decisions の被覆が支える——R/D/T はその**間の
  足場**にすぎない、というのが本方針の賭けである。
- **撤回条件（反証可能性）**: `.kiro/postmortem/defects.md` に「維持された spec 文書の不在」または
  「再生成物の不忠実」を根本要因とする欠陥が未レビューで 2 件積み上がったら（`kiro-postmortem-review`
  の同件閾値と同型）、本方針を再審査する（維持路線への復帰を含む）。**トリガの実行可能化**: 該当欠陥の
  記録時は「根本要因詳細」に literal タグ **`[rdt-disposal]`** を含める（`/kiro-postmortem-add` の
  タグ規律に登録済み）。トリガは同タグを持つ未レビュー（`Status: recorded`）entry 2 件——検知は grep で
  可能だが起動は運用規律であり自動ではない（タクソノミーのラベル一致に依存しない専用トリガ）。

## 計器構成（4器：不変条件と構造判断を SUT 別に割り当てる）

最初の3器は**不変条件**（プロパティ）を、第4器は**構造判断**（Decisions の境界・所有権・依存方向）を担う。

| 計器 | 対象（SUT） | 担う判断クラス | 実行契機 |
|---|---|---|---|
| **PBT・メタモルフィック** | 参照実装（`moira/backend` の純 fold + 導出） | 数値・関係・代数（境界・ロールアップ・スケーリング不変・順序不変・round-trip） | 毎 CI |
| **境界モデル検査** | **MODEL を仕様として**（実装に依らず） | 順序・到達可能性・状態機械（I2/I3・R-U12 同時刻・supersede×cancel 到着順・lifecycle 到達可能性） | `moira-model-update` ゲート内・MODEL の形式スライス変更時のみ |
| **E2E・シナリオ回帰** | `agreed` シナリオ unit から生成 | 妥当性の**preservation（回帰固定）**。establishment ではない | 毎 CI |
| **アーキ適合（fitness関数）** | 実装の依存構造・モジュール境界 | **構造判断**（境界・所有権・依存方向。「core は下流の値を import しない」「式は evm にしか無い」「surface は seam 経由でのみ読む」等） | 毎 CI（dependency-cruiser。**トポロジ2ルール導入済・稼働**＝`no-circular`＋`fold-no-downstream`＝monolith 段階の弱い境界。強い CQRS fitness は物理分解後） |

**整備状況（正直・2026-06-27 テスト基盤ステージで更新）**: **CI（`.github/workflows/ci.yml`）新設・稼働**（backend: `tsc` build＋型テスト＋`vitest run`＋カバレッジ＋dependency-cruiser／frontend: typecheck＋lint＋`vitest run`＋カバレッジ）。カバレッジ計測（@vitest/coverage-v8）配線済。第1器 PBT は**最小パイロット稼働**（green 5＋PR-DONE-LOCK★ は RED tripwire）、**第3器 E2E はブラウザ・シナリオ回帰 suite を新設・稼働**（`moira/frontend/e2e/`＝Playwright。`agreed` unit から `kiro-scenario-e2e` が生成し、シナリオの**全目標**をアサート。実装済み観測＝green 回帰固定、unit が「(スライス未描画)」/(新規) と明示する観測＝`test.fail()` 期待失敗トリップワイヤ〔`it.fails` PR-DONE-LOCK★ のブラウザ版・スライスが追いつくと予期せず pass→CI 赤→昇格〕。fixture は §2/§5 の忠実転記＋derive-golden で先に検証、各 green は非空虚を証明、`e2e-scenario-checker`〔著者≠照合者〕＋`doc-adversary`＋`doc-gate-judge` で確定。`e2e/coverage-check.test.ts` が agreed unit↔spec↔§6 EARS 被覆を毎 CI ゲート。**preservation（回帰固定）であって establishment ではない**——従来の golden/rederive＋frontend SSR golden パリティは in-process の補完として継続〔「e2e を整合性検証の代替にしない」〕）、**第4器 アーキ適合は基盤導入＝トポロジ2ルール稼働**（`no-circular`＋`fold-no-downstream`＝monolith 段階の弱い境界。強い CQRS fitness は分解後）。**第2器 境界モデル検査は依然未実装**。⑥AI整合性チェックは**サンプル試走済**（D-34/D-40＝ALIGNED）・網羅実走は後続。**整備は agreed（D-1〜D-3）分のみ**反映し proposed は据え置き（「割付≠検証済」）。表の「`moira-model-update` ゲート内」（第2器）は**到達目標**であり現状の稼働ではない。

**爆発制御の規律**：境界モデル検査は**小スコープ＋対称性簡約＋プロパティごとの最小フラグメント**で回し、**検証スコープを明示**する（例「ノード≤4・actor≤3・イベント≤8」）。**連続／大領域（日付・c(i,d)・予算）はモデル検査に入れない**——状態爆発を招くため PBT 側へ。離散・順序のみをモデル検査へ。反例の witness は実際小さい（R-E3/I4＝1ノード数イベント、R-U12＝1ノード2人2イベント等）。

## Decisions の検証割付（どの判断をどの計器で）

設計判断（Decisions）は不変条件でもふるまいでもないため、**判断の型に応じて計器へ割り付ける**。1判断が複数計器に跨ることを許す。

- **①アーキ適合（第4器）** — 境界・所有権・依存方向。
- **②型・スキーマテスト** — 表現の構造（「イベントは4種だけ」「担当は1」「isBuffer フィールド無し」等の正/負テスト）。
- **③PBT（第1器）** — 選択の帰結が不変条件になるもの。
- **④E2E・シナリオ回帰（第3器）** — 観測できるふるまい。
- **⑤境界モデル検査（第2器）** — 順序・状態機械（Decisions では該当せず——順序・状態機械はプロパティ側）。
- **⑥AI整合性チェック** — 自動テストに落とせないが実装に footprint がある判断を、AI が「Decision↔実装」で照合する（負の照合＝「**強制していない**」の確認を含む）。`decision-conformance` スキル＋`decision-conformance-checker` エージェント（独立 checker が設計判断はせず照合に徹し `ALIGNED/DRIFTED/UNVERIFIABLE` を返す。現に稼働する分離は「**著者≠照合者**」のみで、**採点ゲートの主体は未確定の設計点**——doc-gate-judge は doc-refine 専用基準ゆえ流用できず、専用採点者の新設か当面 checker＋著者集約かを実装時に決定。**定義済・実走は後続**）。
- **⑦人間レビューのみ** — P0 委譲・footprint 無し（テスト対象が定義上存在しない）。人間レビュー＋doc 敵対ゲートが見る（自動検査の対象外）。

**自動化できないものは ⑥（または ⑦）が受け持つ**——「テストに落ちないから無検証」にしない（本書の正直枠ゆえ『担保』とは言わない：⑥は定義済・実走は後続、⑦は人間レビュー＋doc 敵対ゲートが見るだけ）。**被覆規律**: 全 Decision に計器を1つ以上割り付け、割付は `moira/DECISIONS-CATALOG.md` の**被覆マップ**で管理する。未割付は可視ギャップとして示す（プロパティ目録の clause→property 被覆と同型）。なお被覆マップが言うのは「割り付けたか」であって「割り付けが妥当か」ではない——後者は人間レビューと doc 敵対ゲートが見る（§5/P2 の正直枠と同様、自動チェックは conformance であって validity ではない）。

## 枠組みの規律（過去の罠の封じ込め）

> PBT 実装の詳細規律・生成器設計の learning は `testing-conventions.md`／各 PBT spec が所有する（PBT 固有の learning は稼働に伴い順次追記。現状そちらに PBT 節は未整備で、本節が方針面の要約を持つ）。本節と次の2節（deny-list・生成器）は**方針面の要約**であり、詳細はそちらへ集約する（二重所有でなく階層所有：ここが方針、向こうが実装）。

- **PBT は AI 信頼を「除去」せず「移転」する**（プロパティ自体がオラクル）。ゆえに：
  1. **プロパティ著者を実装著者と別エージェントにする（マスト）**。相関誤りを断つ。
  2. プロパティも**散文と同じ敵対ゲート**に通す。
  3. 人間が **`proposed → agreed`** で一文を批准（Moira 自身の規律をプロパティに自己適用）。
  4. 非空虚性を**mutation testing**（注入バグを1つ以上殺せること）＋**到達 witness**（生成器が当該条件を実際に踏んだこと）で担保。
- **「担保／ガッチリしたガードレール」と言わない**。正直枠は「**宣言スコープ内の高被覆 falsification ＋ 有界 guarantee**」。§5/P2（網羅性は帰納で証明できない）に従う。

## wrong-by-design deny-list（書いてはいけないプロパティ）

> 正典は `moira/PROPERTIES.md` の DENY 節。ここはその方針要約であり、未収録の角は PROPERTIES.md へ昇格させる。

- **進捗の単調増加**——非単調は正直な信号（P5・R-E4・R-E3）。下がる／復帰で上がるのは正しい挙動。
- **forecast／D_pred／buffer の絶対値**——実装依存（P8）。※対照的に **PV／SPI／frozenSlot はログ決定的＝不変条件クラス**（frozenSlot はレベラ出力でなくイベント属性）。
- **意図的沈黙の検知**（§7 ドリフト・暦の穴・前倒し成熟度盲）——「黙る」ことを**否定プロパティ＋到達 witness**で確認する。
- **cost 重複の (node,amount) 基準**——dedup は **id 基準**（§2.8）。同 (node,amount) で別 id は二重計上が正。
- **無スコープのスケーリング**——時間軸は非線形ゆえ指標・予算次元のみ対象。

## プロパティ目録（catalog）

- **場所**: `moira/PROPERTIES.md`。人間がレビューする一次成果物（MODEL 散文への直接の答え）。
- **列**: ID／根拠 clause／**人間がレビューする平易な一文**／固定する量／意図的に FREE な量／計器／検証スコープ／現状。
- **規律**: **安定 A/I/P/R clause ID で参照（行番号禁止）**——リポジトリは既に行番号がズレている。シナリオ README と同等の**平易な言語規約**（専門用語・skill 実名を本文に出さない）。**clause → property 被覆レポート**を持ち、未 bound の clause は可視ギャップとして示す。`moira-model-update` で触れた clause は、その bound プロパティを**同一 run で再批准**する。

## 生成器（最大のリスク。プロパティ本体ではない）

- **2つ**：イベントログ生成器 ＋ 容量 c(i,d) 生成器。いずれも整形規則（I2 非循環・R-U4 人間合意・supersede 向き等）を満たす**敵対的構成の一級レビュー対象**。
- **自己被覆アサーション**：completion-without-agreement（R-U8）／agent 合意拒否（R-U4）／agreed→proposed 復帰（R-E3）／supersede→cancel 復帰（R-S5）／未見積誕生（I1）を各 ≥1 回生成したことを確認。生成されない隅は**未テストの不変条件**。
- 構造保存 shrinker ＋ 決定的 seed ログ（CI 失敗の再現）。

## 動機（手法の効き目の実証）

本方針の発端として、実装に**潜在乖離**を1件発見した：MODEL は R-E3/I4 で「`agreed→proposed` 復帰は未完了限定・完了ノードは I4 施錠で EV_abs 不動」とするが、参照実装の `fold.ts` は完了ガード無しで見積を `proposed` に戻し、`ev.ts` は `agreed` のみ算入する——**完了ノードを再見積に戻すと EV_abs が静かに脱落**しうる。「完了 agreed ノードを revert → EV_abs 不変」というプロパティは**現実装に FAIL する**。散文ゲートは MODEL を検証したが、その制約が実装へ降りる途中で漏れた——**実行可能なプロパティゲートが必要**であることの実証（この乖離は現在 `moira/backend/src/pbt/done-lock.pbt.test.ts` の PR-DONE-LOCK★ が RED tripwire として実走捕捉している）。なおこの乖離は「不変条件の実装漏れ」であると同時に、**enforce 手段の所有権**——R-E3 の incomplete 限定（MODEL v17 で確定済み。I4 完了施錠の enforce は core が担うと `moira-core` requirements 7.4 が規定）を、fold が構造的に拒否するのか、書込み層規律＋下流警告（P5 系）が担うのか——という設計判断でもある。これは Decisions が本来詰める第三カテゴリであり、実行プロパティと Decisions の両面が要ることの実証でもある。

## 変更時のゲート対応

- MODEL の制約・語彙・既定を変える → `moira-model-update`（不変条件の形式スライス・bound プロパティ・モデル検査を同一 run で再確定）。
- **設計判断（Decisions）を新規・改訂し批准する → `doc-refine`（`moira/DECISIONS-CATALOG.md` を確定ゲートで磨く）。MODEL の制約・語彙・既定に触れる判断は `moira-model-update` で正典へ昇格。**
- 確定文書（steering・エージェント定義等）を磨く → `doc-refine`。
- **`agreed` シナリオ unit から E2E シナリオ回帰（計器③）を生成・更新する → `kiro-scenario-e2e`**（fixture は §2/§5 忠実転記＋derive-golden、全 EARS 節を green/xfail/deferred で計上、`e2e-scenario-checker`〔著者≠照合者〕＋`doc-adversary`＋`doc-gate-judge` で確定、`e2e/coverage-check.test.ts` が被覆を毎 CI ゲート）。
